# 第十六部分：Prompt Engineering

## 一、Prompt Engineering 概述

### 1.1 什么是 Prompt Engineering？

**Prompt Engineering（提示工程）** 是指通过设计和优化输入文本（即 Prompt，提示词）来引导大型语言模型（Large Language Model, LLM）生成期望输出的技术和方法论。

通俗地说，Prompt Engineering 就是学习"如何跟 AI 对话才能得到你想要的结果"。这并不像表面看起来那么简单——同样的模型、同样的能力，不同的提问方式可能导致天壤之别的输出质量。

```python
# 同一个模型，同一个问题，不同的提问方式 → 截然不同的输出质量

# ❌ 模糊的 prompt
prompt_模糊 = "解释一下机器学习"
# 输出可能太宽泛，不知道从何说起，得到一段教科书式的定义

# ✅ 具体的 prompt  
prompt_具体 = "我是一名刚接触编程的高中生，请用生活中的类比解释什么是机器学习，
包括：1) 它是什么  2) 它和传统编程的区别  3) 一个日常生活中的例子。
控制在300字以内。"
# 输出会更有针对性、更容易理解
```

**为什么需要 Prompt Engineering？**

LLM 的本质是一个**概率预测器**：给定上文，预测下一个最可能的词。模型本身并没有"理解"你真正想要什么——它只是在统计上判断，在你给出的文字后面，最可能跟着哪些文字。

这意味着：
- 如果你问得模糊，模型会"猜"你最可能想要什么——猜中的概率不高
- 如果你给出清晰的上下文、约束和示例，模型的"猜测"空间大幅缩小，输出质量显著提升

> **核心洞察**：Prompt Engineering 的本质不是"欺骗"或"操纵"模型，而是**缩小模型的输出概率空间**，让它在你期望的区域内采样。

### 1.2 Prompt 的基本构成

一个高质量的 Prompt 通常包含以下几个要素：

| 要素 | 说明 | 示例 |
|------|------|------|
| **角色（Role）** | 告诉模型以什么身份回答问题 | "你是一名资深 Python 工程师" |
| **任务（Task）** | 明确告诉模型需要做什么 | "请审查下面这段代码的安全性问题" |
| **上下文（Context）** | 提供完成任务所需的背景信息 | "这段代码将部署到生产环境的公网服务器上" |
| **约束（Constraints）** | 限制输出的格式、风格、长度等 | "请用中文回答，不超过200字，使用列表格式" |
| **示例（Example）** | 给出输入-输出的样例 | 提供一个问答对，说明你期望的格式和风格 |
| **输出格式（Format）** | 明确输出的结构 | "输出JSON格式：{\"result\": ...}" |

这六个要素不是每次都需要全部使用，但理解它们能帮助你在写 Prompt 时更系统化。

### 1.3 Prompt Engineering 与 LLM 能力的关系

要理解为什么不同的 Prompt 策略有效，需要先了解 LLM 的几个关键特性：

**特性一：上下文学习（In-Context Learning）**

LLM 不需要针对每个新任务重新训练。你只需要在 Prompt 中提供几个例子，模型就能"学会"执行这个任务。这叫做**上下文学习**——模型从 Prompt 中的示例推断出任务规则，并在推理时应用这些规则。

这与传统的监督学习完全不同：
- 传统 ML：需要收集标注数据 → 训练模型 → 部署模型 → 新任务需要重新训练
- LLM + In-Context Learning：在 Prompt 中放几个例子 → 模型直接推理 → 新任务只需要换例子

**特性二：指令遵循（Instruction Following）**

经过指令微调（Instruction Tuning）的 LLM（如 GPT-4、Claude 等）被训练成"遵循人类指令"。这意味着：如果你在 Prompt 中清晰、直接地给出指令，模型会比没有指令时更有可能按你的要求做。

**特性三：涌现能力（Emergent Abilities）**

当模型规模足够大时，会出现小模型没有的能力，比如**链式推理（Chain-of-Thought）**。这些能力不是被明确训练出来的，而是随着模型规模和训练数据的增加"涌现"出来的。

Prompt Engineering 的核心任务就是**设计恰当的"触发器"来激活这些涌现能力**。

## 二、Zero-Shot Prompting（零样本提示）

### 2.1 什么是 Zero-Shot Prompting？

Zero-Shot Prompting 是指**不提供任何示例**，直接让模型执行任务的方式。这是最基础、最常用的 Prompt 形式。

```python
# Zero-Shot Prompting 示例
prompt = """将下面的英文句子翻译成中文：
'Artificial intelligence is transforming every industry.'
"""

# 模型直接输出中文翻译，不需要任何示例
# 输出：人工智能正在改变每一个行业。
```

"Zero-Shot"（零样本）这个名字来源于机器学习中的"零样本学习"（Zero-Shot Learning）——模型在训练时没有见过这个特定任务的标注数据，但仍然能完成任务。

### 2.2 Zero-Shot 为什么能工作？

Zero-Shot Prompting 之所以有效，是因为大型语言模型在预训练阶段已经接触了海量的、覆盖几乎所有领域的文本数据。当你提出一个任务时，模型实际上是在**回想**它在训练数据中见过的类似模式。

以翻译任务为例：
- 模型在训练数据中见过大量"中英对照"的文本（双语网站、翻译文档、字幕文件等）
- 当你说"翻译成中文"时，模型识别出这是一个"翻译模式"
- 模型从内部知识中调取它"学到"的翻译能力

**Zero-Shot 生效的底层逻辑**：

```
训练阶段：模型看到数万亿 tokens 的文本
          ├── 其中包含大量"指令→响应"的模式
          ├── 也包含大量"问题→答案"的模式
          └── 还包含大量"翻译、总结、改写"等任务描述

推理阶段：用户给出指令
          ├── 模型识别出与训练数据中某类模式匹配
          ├── 激活相关的"能力模块"
          └── 生成符合该模式预期的输出
```

### 2.3 Zero-Shot Prompting 的核心技巧

虽然 Zero-Shot 看起来很"零门槛"，但写得好和写得差之间差距巨大：

**技巧一：指令清晰明确**

```python
# ❌ 模糊 —— 模型需要猜你要什么
prompt_模糊 = "写一些关于Python的东西"

# ✅ 清晰 —— 模型知道你要什么
prompt_清晰 = """写一篇300字的短文，介绍Python语言的三个主要特点：
1. 简单易学（适合初学者）
2. 生态丰富（大量第三方库）
3. 应用广泛（Web开发、数据分析、AI等）
要求：面向有编程经验的开发者，语气专业但不学术化。
"""
```

**技巧二：使用分隔符明确区分内容**

当 Prompt 中包含用户输入的内容时，使用分隔符帮助模型区分"指令"和"被处理的内容"：

```python
prompt = """请对以下文本进行摘要，提取3个关键要点：

文本内容：
```
今天发布了2025年第三季度的财报。公司营收达到120亿美元，同比增长15%。
净利润为28亿美元，同比增长22%。增长主要得益于云计算业务的强劲表现，
该业务收入同比增长35%。公司预计第四季度营收将在125-130亿美元之间。
```

请用中文输出，每个要点不超过20字。"""
```

**技巧三：指定输出格式**

如果你需要特定格式的输出，必须明确告诉模型：

```python
# 指定 JSON 格式输出
prompt = """分析以下商品的评论情感，输出 JSON 格式：

评论："这个手机电池续航很差，但屏幕显示效果非常出色。"

输出格式：
{
  "整体情感": "正面/负面/中性",
  "评分": 1-5,
  "优点": [],
  "缺点": [],
  "详细分析": ""
}
"""
```

**技巧四：让模型分步思考**

即使是 Zero-Shot，也可以通过简单的引导让模型表现更好。后面会详细介绍 Chain-of-Thought，但这里有一个简单的版本：

```python
# 直接让模型回答复杂问题可能出错
prompt_直接 = "一个篮球和一个足球总共220元，篮球比足球贵40元，篮球多少钱？"
# 模型可能脱口而出一个错误答案

# 引导模型一步步思考，准确率大幅提升
prompt_分步 = "一个篮球和一个足球总共220元，篮球比足球贵40元。让我们一步步分析：
1. 设足球价格为x元
2. 篮球价格 = x + 40
3. 总价方程：x + (x + 40) = 220
请给出答案。"
```

### 2.4 Zero-Shot 的局限性

Zero-Shot 虽然方便，但有以下不足：

| 局限性 | 原因 | 表现 |
|--------|------|------|
| **格式不稳定** | 没有示例约束输出格式 | 同样的 prompt 每次输出结构可能不同 |
| **复杂任务效果差** | 模型难以仅凭指令理解复杂任务 | 分类粒度不够细、推理步骤容易跳 |
| **易受措辞影响** | 对关键词敏感 | 换一种说法效果可能大幅波动 |
| **缺乏一致性** | 没有"锚定"示例 | 同一任务多次输出风格差异大 |

这些局限性正是 Few-Shot Prompting 和 Chain-of-Thought 要解决的问题。

## 三、Few-Shot Prompting（少样本提示）

### 3.1 什么是 Few-Shot Prompting？

Few-Shot Prompting 是在 Prompt 中**提供若干个输入-输出示例**（通常2-5个），让模型通过这些示例"理解"任务模式，然后处理新的输入。

```python
# Few-Shot Prompting 示例：情感分类

prompt = """请判断以下评论的情感是"正面"还是"负面"：

评论：这家餐厅的菜非常好吃，服务也很周到。
情感：正面

评论：等了一个小时还没上菜，体验很差。
情感：负面

评论：价格合理，环境不错，但菜品一般。
情感：正面

评论：这个产品质量太差了，用了三天就坏了。
情感："""
# 模型会根据前面的模式，输出：负面
```

### 3.2 Few-Shot 为什么有效？

Few-Shot 有效的原因在于 LLM 的**模式匹配能力**。当你提供多个例子时，模型在做以下几件事：

**第一，排除歧义**。有些任务描述本身有歧义。例如，"把这个句子分类"——但分成几类？按什么标准？给出示例后，所有歧义都消失了。

**第二，建立格式模版**。示例定义了输入和输出的格式，模型只需遵循这个模版。

**第三，学习任务边界**。通过正例和反例，模型理解了你想要的"边界"在哪里。

```python
# 没有示例 → 模型可能按自己的理解分类
prompt_无示例 = "判断情感：这个产品还行。"
# 模型可能输出："中性"——但你可能只需要"正面/负面"二分类

# 有示例 → 模型严格按照你的标准分类
prompt_有示例 = """判断以下评论的情感是"正面"还是"负面"（不需要中性）：

评论：太棒了！
情感：正面

评论：很差劲。
情感：负面

评论：还行吧。
情感："""
# 模型输出："正面"——因为"还行"在你的示例中更接近正面描述
```

### 3.3 如何选择合适的 Few-Shot 示例？

示例选择直接影响 Few-Shot 的效果，以下几个原则至关重要：

**原则一：示例的代表性**

选择的示例应该覆盖目标任务的"典型"情况。如果你在做新闻分类，示例应该覆盖最常见的几个类别。

**原则二：示例的多样性**

示例不应该太相似。如果3个例子都是"正面评价"，模型可能认为所有输出都应该是"正面"。

**原则三：标签和格式的一致性**

所有示例必须使用完全一致的标签和输出格式。不一致的示例会让模型困惑。

**原则四：示例的顺序**

研究表明，示例的顺序会影响效果。一般来说：
- 把更相似于目标输入的示例放在最后（靠近查询的位置）
- 把代表性最广的示例放在最前面

```python
# 好的 Few-Shot 示例配置
prompt_好 = """将以下英文句子翻译成法语：

英语：Hello
法语：Bonjour

英语：Good morning
法语：Bonjour（注意：法语中"早上好"也用Bonjour）

英语：Thank you
法语：Merci

英语：How are you?
法语："""

# 分析：
# ✓ 示例覆盖了不同的表达类型（问候、感谢）
# ✓ 所有示例的格式完全一致
# ✓ 最后一个示例与目标查询类型相近（都是问句）
```

### 3.4 Few-Shot 的局限性与改进

**Few-Shot 的局限性**：

1. **Token 消耗大**：每个示例消耗上下文窗口的 tokens，如果示例太长，留给实际输入的空间就变小了
2. **受示例质量影响大**：不好的示例会"误导"模型，不如不用
3. **对顺序敏感**：同样的示例换个顺序，结果可能不同
4. **不适用于所有任务**：某些任务（如数学推理）仅靠示例不足以让模型学会

**改进方案**：

当 Few-Shot 效果不佳时，可以尝试以下改进：

| 问题 | 改进方法 |
|------|----------|
| 示例太多超出上下文窗口 | 减少示例数量（2-3个），或者用更短的示例 |
| 示例质量参差不齐 | 精心挑选"错误边界"附近的示例（即模型容易搞混的情况） |
| 任务太复杂 | 用 Few-Shot + Chain-of-Thought（提供推理过程示例） |
| 格式不稳定 | 在示例中明确标注格式，并在最后一条指令中再强调一次 |

### 3.5 Zero-Shot vs Few-Shot：何时使用哪个？

| 对比维度 | Zero-Shot | Few-Shot |
|----------|-----------|----------|
| **适用场景** | 简单、常见的任务；任务边界清晰 | 复杂、特殊格式的任务；需要精确控制输出 |
| **Token 消耗** | 最少 | 较多（取决于示例长度和数量） |
| **灵活性** | 高（随时切换任务） | 低（切换任务需要换示例） |
| **输出一致性** | 低（容易波动） | 高（被示例"锚定"） |
| **实现成本** | 低（不需准备示例） | 较高（需要设计和筛选示例） |

> **经验法则**：先尝试 Zero-Shot。如果 Zero-Shot 效果不佳，再尝试 Few-Shot。不要在一开始就堆砌示例——有时候一个精心设计的 Zero-Shot Prompt 比草率的 Few-Shot 效果更好。

## 四、Chain-of-Thought Prompting（思维链提示）

### 4.1 什么是 Chain-of-Thought Prompting？

Chain-of-Thought Prompting（思维链提示，简称 CoT）是一种让模型**展示推理过程**的 Prompt 技术。它要求模型在给出最终答案之前，先输出一步步的推理步骤。

```python
# 标准 Prompt vs Chain-of-Thought Prompt

# 标准 Prompt（直接问答案）
prompt_标准 = "小明有12个苹果，送给小红三分之一，又给了小刚2个，还剩几个？"
# 模型可能直接输出一个数，可能对也可能错

# Chain-of-Thought Prompt（引导推理过程）
prompt_cot = """小明有12个苹果，送给小红三分之一，又给了小刚2个，还剩几个？让我们一步步来：

第一步：小明有12个苹果
第二步：送给小红三分之一，12 × 1/3 = 4，所以给了小红4个
第三步：又给了小刚2个
第四步：剩余 = 12 - 4 - 2 = 6

所以小明还剩6个苹果。"""
```

CoT 的核心理念是：**让模型"出声思考"**（Think Aloud）。对模型来说，推理过程不是黑盒——如果你让它一步一步推演，它在每步产生 tokens 时都会重新评估上下文，从而大幅降低在中间步骤出错的可能性。

### 4.2 为什么 Chain-of-Thought 有效？

这是一个深层的问题。CoT 之所以有效，与 Transformer 架构的**自注意力机制**有直接关系。

**原因一：分解复杂问题**

多步推理问题对模型的"单步"预测能力要求极高。如果不展示推理过程，模型需要在**一个预测步骤**中完成所有计算——这相当于要求一个人不经过演算直接写出复杂算式的答案。

CoT 将复杂问题分解为多个简单子问题，每个子问题只需要模型做一次相对简单的预测。

```
没有 CoT：         有 CoT：
问题                问题
  │                   │
  ▼                   ▼
[一步预测]         [第一步推理] ← 只需要一点局部计算
  │                   │
  ▼                   ▼
 答案              [第二步推理] ← 基于上一步结果
                      │
                      ▼
                   [第三步推理]
                      │
                      ▼
                    答案
```

**原因二：中间步骤提供"推理锚点"**

每产生一个中间推理步骤，模型就多了一个"锚点"。后续的推理可以基于这些锚点，而不是完全依赖模型内部的隐式表示。

这就像你在陌生的城市导航：如果只给你最终目的地（无 CoT），你可能走错路；但如果每到一个路口都有人告诉你"直走200米，左转，看到红色大楼后再右转"（有 CoT），你几乎不可能走错。

**原因三：激活模型的"推理能力"**

研究发现，CoT 能力是随着模型规模增长而**涌现**的——小模型（如 10B 参数以下）即使给 CoT 示例也做不好推理，而大模型（如 100B+ 参数）在 CoT 下的表现会显著提升。

这意味着 CoT 并不是"教"模型推理，而是**激活**模型已有的推理能力。

### 4.3 Zero-Shot Chain-of-Thought

Zero-Shot CoT 是目前最实用的 CoT 变体之一。它不需要人工编写的推理示例，只需要在 Prompt 末尾加上一句神奇的提示词：

```python
# Zero-Shot Chain-of-Thought
# 关键技术：在 prompt 末尾加上"让我们一步步思考"

prompt_zeroshot_cot = """一个水池有一个进水管和一个出水管。单独打开进水管需要4小时把空水池注满，单独打开出水管需要6小时把满水池排空。如果同时打开两个水管，需要多少小时能把空水池注满？

让我们一步步思考。"""

# 模型会自行输出推理过程：
# 进水管每小时注水：1/4
# 出水管每小时排水：1/6
# 同时打开，净注水速度：1/4 - 1/6 = 3/12 - 2/12 = 1/12
# 所以需要 1 / (1/12) = 12 小时
```

**常见的 Zero-Shot CoT 触发短语**：

| 短语 | 适用场景 |
|------|----------|
| "让我们一步步思考" | 通用，多数场景有效 |
| "Let's think step by step" | 英文环境下最稳定 |
| "请逐步推理" | 中文环境 |
| "首先，...然后，...最后..." | 当模型不自动分步时，给一个结构模板 |
| "我们来分析一下这个问题" | 适合分析类问题 |

> **为什么"让我们一步步思考"这一句话就够了？**
>
> 这句话实际上是一种"元指令"（meta-instruction）——它不直接告诉模型推理内容，而是告诉模型**推理的策略**。模型在训练数据中见过大量带有"让我们一步步思考"的推理文本，这句话触发了模型内部的"推理模式"。

### 4.4 Few-Shot Chain-of-Thought

Few-Shot CoT 是在示例中同时提供**问题 + 推理过程 + 答案**，让模型从完整的推理示例中学习。

```python
# Few-Shot Chain-of-Thought
prompt_fewshot_cot = """解决以下数学问题，每一步都要展示推理过程。

问题：小明买了3支笔，每支2元，又买了5个本子，每个4元，一共花了多少钱？
推理：买笔花了 3 × 2 = 6 元。买本子花了 5 × 4 = 20 元。总共花了 6 + 20 = 26 元。
答案：26元

问题：一个长方形的长是8米，宽是5米，它的周长是多少？
推理：周长 = 2 × (长 + 宽) = 2 × (8 + 5) = 2 × 13 = 26 米。
答案：26米

问题：商店有120个苹果，上午卖出了45个，下午又运来了30个，现在商店有多少个苹果？
推理："""
```

**Few-Shot CoT 的关键要点**：

1. **推理步骤要完整**：不能跳跃推理步骤，模型会模仿你的"粒度"。你跳步了，模型也会跳步
2. **推理步骤要清晰**：每步只做一件事，不要在一个步骤中做多个计算
3. **推理和答案要分离**：用"推理："和"答案："明确区分过程和结果

### 4.5 Chain-of-Thought 的改进变体

**变体一：自洽性思维链（Self-Consistency CoT）**

同一个问题，让模型推理多次（如5次），然后对多次推理的结果进行投票，选择出现频率最高的答案。

```python
import random

def self_consistency(prompt, model, n=5):
    """
    自洽性思维链
    原理：同一个问题，多次推理取多数答案
    为什么有效？推理过程有随机性，但正确答案通常更稳定
    """
    answers = []
    for i in range(n):
        # 每次用不同的随机种子生成推理路径
        response = model.generate(prompt + "\n让我们一步步思考：", temperature=0.7)
        answer = extract_answer(response)
        answers.append(answer)
    
    # 投票选择最常见的答案
    final_answer = max(set(answers), key=answers.count)
    return final_answer
```

**为什么自洽性有效？** 模型的推理路径有一定的随机性（受 temperature 参数影响），但通往正确答案的路径通常比通往错误答案的路径更多样化。通过多次采样并投票，相当于"集成了"多个推理路径的智慧。

**变体二：思维树（Tree-of-Thought, ToT）**

相比 CoT 的一条线性推理链，思维树让模型在每一步探索多个可能的推理方向，并评估每个方向的可行性，然后选择最有希望的路径继续探索。

```
CoT:  问题 → 步骤1 → 步骤2 → 步骤3 → 答案（单一路径）

ToT:  问题 → 步骤1a ──→ 步骤2a ──→ 步骤3a
            ↘ 步骤1b ──→ 步骤2b ──→ 步骤3b (被剪枝)
            ↘ 步骤1c ──→ 步骤2c (被剪枝)
```

ToT 适用于需要探索和回溯的复杂问题（如数学证明、规划问题），但计算成本远高于普通 CoT。

**变体三：思维链 + 自我修正**

在 CoT 的基础上，增加"检查"步骤：让模型在得出答案后，检查自己的推理过程是否有误。

```python
# CoT + 自我修正
prompt_cot_selfcheck = """问题：如果一列火车以每小时120公里的速度行驶，需要3小时到达目的地，那么这列火车行驶了多少公里？

让我们一步步思考：
1. 速度 = 120 公里/小时
2. 时间 = 3 小时
3. 距离 = 速度 × 时间 = 120 × 3 = 360 公里

现在让我们检查一下答案是否正确：
- 如果火车每小时走120公里，3小时走 120 × 3 = 360 公里 ✓
- 单位一致：公里/小时 × 小时 = 公里 ✓
- 答案合理：360公里符合高速铁路的行驶距离 ✓

答案：360公里"""
```

### 4.6 CoT 的适用场景与局限

**CoT 最有效的情况**：
- 数学推理、逻辑推理
- 复杂问题分解
- 需要多步决策的任务
- 代码调试和算法设计

**CoT 不太适用的情况**：
- 简单、直觉性任务（如情感分类、主题识别）
- 需要创造性、发散性回答的任务（CoT 可能让模型过于"死板"）
- 对延迟敏感的场景（CoT 增加了输出 tokens 数量）

| 场景 | 推荐策略 |
|------|----------|
| 简单分类任务 | Zero-Shot 或 Few-Shot（不需要 CoT） |
| 数学/逻辑问题 | Zero-Shot CoT 或 Few-Shot CoT |
| 需要高准确率 | Self-Consistency CoT |
| 复杂规划/探索 | Tree-of-Thought（如果有计算资源） |
| 编程问题 | CoT + 代码分步执行 |

## 五、RAG Prompt 模板设计

### 5.1 什么是 RAG？

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将信息检索与文本生成相结合的技术框架。在 RAG 系统中，LLM 不是直接凭"记忆"回答，而是先从外部知识库中检索相关文档，然后将这些文档作为上下文提供给模型，由模型基于检索到的信息生成答案。

RAG 的架构可以简化为三个步骤：

```
用户问题
    │
    ▼
[步骤1：检索] ──→ 从知识库中找到与问题最相关的文档片段
    │
    ▼
[步骤2：合成] ──→ 将检索到的文档+原始问题拼成 Prompt
    │
    ▼
[步骤3：生成] ──→ LLM 基于 Prompt 生成答案
```

### 5.2 RAG Prompt 与普通 Prompt 的区别

RAG Prompt 与普通 Prompt 在结构上有本质区别：

```python
# 普通 Prompt（凭模型内部知识回答）
prompt_普通 = "什么是量子计算？请用简单易懂的语言解释。"

# RAG Prompt（基于检索到的文档回答）
prompt_rag = """请基于以下提供的文档内容回答用户的问题。
如果你在文档中找不到相关信息，请明确说明"文档中没有相关信息"，
不要自行编造。

【文档内容】
{检索到的文档片段}

【用户问题】
什么是量子计算？

【回答】
"""
```

RAG Prompt 必须额外考虑以下几个关键点：

| RAG Prompt 要素 | 为什么重要 | 如果不加会怎样 |
|----------------|-----------|---------------|
| **限定回答来源** | 告诉模型"只基于检索内容回答" | 模型可能混合自己的知识和检索内容 |
| **处理无信息** | 告诉模型检索内容不足时怎么做 | 模型可能编造（幻觉） |
| **文档引用** | 要求模型在回答中标明信息来源 | 用户无法验证回答的真实性 |
| **上下文窗口管理** | 控制检索文档的长度和数量 | 可能超出模型的最大 token 限制 |

### 5.3 标准的 RAG Prompt 模板

以下是一个经过实践验证的 RAG Prompt 模板，以及每个部分的详细说明：

```python
RAG_PROMPT_TEMPLATE = """你是一个基于知识库的问答助手。请严格遵循以下规则：

## 规则
1. 回答必须完全基于【文档内容】，不要使用你自己的知识
2. 如果【文档内容】中没有足够的信息回答问题，请说"根据提供的文档，无法回答这个问题"
3. 如果【文档内容】中包含相关信息，请用自己的话重新组织，不要直接复制粘贴
4. 在回答末尾标注信息来源的文档序号，如 [来源：文档1]

## 文档内容
{context}
（说明：{context} 是检索系统从知识库中取回的相关文档片段，
每个片段前标注了文档序号，如 【文档1】、【文档2】等）

## 用户问题
{question}

## 回答
"""

# 使用示例
def build_rag_prompt(question, retrieved_docs):
    """
    构建 RAG Prompt
    retrieved_docs: 检索到的文档列表，每个元素包含内容和序号
    """
    # 组装文档上下文
    context_parts = []
    for i, doc in enumerate(retrieved_docs, 1):
        context_parts.append(f"【文档{i}】\n{doc['content']}")
    context = "\n\n".join(context_parts)
    
    # 填入模板
    prompt = RAG_PROMPT_TEMPLATE.format(
        context=context,
        question=question
    )
    return prompt
```

**模板各部分的深度解析**：

**"回答必须完全基于【文档内容】"**：
这是 RAG Prompt 中最关键的一句话。没有这句话，LLM 倾向于混合自己的内部知识和检索到的信息。你可能检索到正确的信息，但模型仍然会"补充"一些它自己"知道"的内容——这些补充内容可能是错的。

**"请用自己的话重新组织"**：
防止模型直接复制粘贴检索文档。直接复制会导致：1）回答可能不连贯（检索的文档片段可能来自不同位置） 2）版权问题 3）无法展示模型的理解能力。

**"标注信息来源"**：
这是 RAG 系统与用户建立信任的关键机制。用户可以查看来源，验证回答的准确性。如果没有来源标注，用户无法区分哪些是检索内容、哪些是模型自己的输出。

**"文档中没有足够信息"的处理**：
RAG 的一个核心优势是减少幻觉。但如果你不给模型一个"说不知道"的出口，当检索不到相关信息时，模型会倾向于编造一个看起来合理的答案。明确告诉模型可以说"不知道"，是减少幻觉的重要策略。

### 5.4 处理检索内容过长的策略

当检索到的文档太长，超出模型的上下文窗口时，需要对 Prompt 进行裁剪：

```python
def truncate_context(docs, max_tokens=3000, tokenizer=None):
    """
    截断检索文档以适应上下文窗口
    策略：优先保留与问题最相关的部分
    """
    truncated = []
    total_tokens = 0
    
    for doc in sorted(docs, key=lambda x: x['relevance_score'], reverse=True):
        doc_tokens = estimate_tokens(doc['content'])  # 估算 tokens 数量
        
        if total_tokens + doc_tokens <= max_tokens:
            truncated.append(doc)
            total_tokens += doc_tokens
        else:
            # 如果放不下整个文档，尝试截取文档的开头部分
            remaining = max_tokens - total_tokens
            if remaining > 100:  # 至少保留100个token才有意义
                doc['content'] = truncate_to_tokens(doc['content'], remaining)
                truncated.append(doc)
            break
    
    return truncated
```

**常用的文档排序策略**：

| 策略 | 方法 | 适用场景 |
|------|------|----------|
| **相关性排序** | 按检索得分（如余弦相似度）降序排列 | 通用，最简单的基线 |
| **先相关后完整** | 优先放高相关性的文档，同一文档尽量放完整片段 | 需要完整上下文理解 |
| **多样性排序** | 选择覆盖不同方面的文档，避免内容重复 | 需要全面回答时 |
| **时间排序** | 按文档时间戳，最新的优先 | 新闻、时效性强的场景 |

### 5.5 RAG Prompt 的进阶技巧

**技巧一：分块边界说明**

检索到的文档往往是从长文档中截取的片段，片段开头和结尾可能突然断掉。在 Prompt 中说明这一点，可以帮助模型理解：

```python
prompt = """以下是检索到的文档片段。注意：这些片段可能只包含文档的一部分，
前后内容可能不完整，请基于已有信息回答。

{chunk2_content}

问题：决策树的剪枝和随机森林有什么关系？
"""
```

**技巧二：多轮对话中的 RAG**

在对话场景中，用户可能会追问。每次都重新检索和构建 Prompt 是最直接的方式，但也可以利用对话历史：

def build_conversational_rag(chat_history, new_question, retrieved_docs):

**技巧三：多源信息融合**

当检索到多个来源的文档，且这些文档之间可能存在不一致时：

prompt = """你从多个文档中找到了相关信息。请综合这些信息回答用户问题。

{content1}

【文档2（技术博客）】
{content2}

【文档3（学术论文）】
{content3}

问题：什么是 Transformer 注意力机制？

要求：
1. 综合各文档的共同点

### 5.6 RAG 与 CoT 的结合

RAG 和 CoT 并不互斥——实际上它们可以很好地结合。当一个问题需要基于检索到的信息进行推理时，可以要求模型先分析文档内容，再逐步推理出答案：

```python
# RAG + CoT 组合 Prompt
prompt_rag_cot = """基于以下文档内容回答用户问题。在回答前，请一步步分析：

## 文档内容
【文档1】关于2024年GDP增长的报告：
2024年第一季度GDP增长5.3%，第二季度增长4.7%，
第三季度增长5.0%，第四季度增长5.2%。

【文档2】关于2025年GDP预测：
经济学家预测2025年GDP增长率约为4.8%到5.2%之间，
主要受到出口和消费的双重驱动。

## 问题
2024年全年GDP平均增长率是多少？与2025年的预测相比如何？

## 分析过程
（请你一步步计算和比较，最后给出答案）
"""
```

这种结合方式特别适合"需要从文档中提取数据 → 进行计算 → 得出结论"的复杂查询场景。

## 六、幻觉（Hallucination）与 Prompt 层面的缓解策略

### 6.1 什么是 LLM 的幻觉？

**幻觉（Hallucination）** 是指 LLM 生成了**看似合理但实际错误或无根据**的内容。模型"信誓旦旦"地陈述一件不存在的事、引用一篇不存在的论文、或者把错误的信息包装得滴水不漏。

```python
# 幻觉示例：模型编造了不存在的论文
prompt = "有哪些关于使用深度学习预测股票价格的重要论文？"

# 模型可能输出类似这样的回答：
"""
1. "Stock Prediction using LSTM" by Zhang et al. (2019) 
   —— 这篇论文可能真实存在
2. "Deep Portfolio Optimization" by Wang et al. (2020)
   —— 这篇论文可能真实存在
3. "Transformer-based Stock Forecasting with Attention Mechanism" by Liu et al. (2021)
   —— 这篇论文很可能是模型编造的！模型只是"觉得"这个标题看起来合理
"""
# 问题：模型无法区分"合理的标题"和"真实存在的论文"
```

**幻觉为什么特别危险？** 因为 LLM 的幻觉内容通常看起来**非常可信**——语法正确、逻辑自洽、引用格式规范。不熟悉该领域的读者很难辨别真伪。

### 6.2 为什么会产生幻觉？

要理解如何减少幻觉，必须先理解幻觉的**根本原因**：

**原因一：模型本质上是"下一个词预测器"**

LLM 的核心训练目标是预测"下一个最可能的词"，而不是判断"下一个正确的词"。这两个目标在大多数情况下一致，但在需要精确事实的场景中存在根本冲突。

**原因二：知识存储是"隐式的"**

LLM 将知识存储在神经网络权重中，而不是像数据库一样以结构化方式存储。这意味着：
- 模型无法"查询"自己的知识——它只能根据概率"生成"下一个词
- 模型无法区分"它确定知道的事"和"它不太确定的事"
- 模型没有内置的"我不知道"机制

**原因三：训练数据的偏差**

训练数据中本身就包含错误信息、矛盾的表述和过时的内容。模型"学到了"这些不一致。

**原因四：模型倾向于"讨好"用户**

经过 RLHF（基于人类反馈的强化学习）训练的模型被优化为生成"用户喜欢"的回答。如果模型认为一个编造的答案比"我不知道"更讨用户喜欢，它会选择编造。

```
模型内部可能做出的"权衡"：

选项A："我不知道这个问题的答案。"
  → 后果：用户不满意，模型被认为"无能"
  → 优化目标得分：低

选项B："这个问题的答案是XXX（编造但是听起来合理的内容）。"
  → 后果：用户得到答案（虽然可能是错的），模型被认为"有帮助"
  → 优化目标得分：高

→ 模型倾向于选择B
```

### 6.3 Prompt 层面的幻觉缓解策略

虽然无法在 Prompt 层面完全消除幻觉（幻觉的根源在模型训练阶段），但良好的 Prompt 设计可以**显著减少幻觉的发生概率**。

**策略一：给出"不知道"的出口（最重要的策略）**

```python
# ❌ 没有给"不知道"的出口
prompt_不好 = "请回答：2025年诺贝尔物理学奖得主是谁？"
# 如果模型不知道，它会编造一个

# ✅ 明确告诉模型可以说"不知道"
prompt_好 = """请回答以下问题。如果你不确定答案，或者问题涉及的事实不在你的知识范围内，
请明确说"我不知道"或"我无法确认"——不要编造信息。

问题：2025年诺贝尔物理学奖得主是谁？
"""
```

**为什么这个策略有效？** 它改变了模型的"权衡"——在优化目标中，"说不知道"不再是一个低分选项。你明确允许了"不知道"的存在。

**策略二：要求提供引用和来源**

```python
prompt = """请回答以下问题，并为你的每一个关键陈述提供引用来源。
如果你找不到可靠的引用来源，请明确说明这是你的推测。

问题：Transformer 架构中的注意力机制有什么优点？

回答格式：每个要点后标注来源，例如 [来源：Vaswani et al. 2017]
"""
```

但要注意：模型可能会编造来源（假装引用了一篇不存在的论文）！所以这个策略只能作为一个辅助手段，不能完全信任模型给出的引用。

**策略三：限定回答范围**

```python
# ❌ 范围太宽，模型容易跑偏
prompt_宽 = "介绍一下微服务架构的优缺点"

# ✅ 限定范围，减少模型"即兴发挥"的空间
prompt_窄 = """请从以下三个方面介绍微服务架构：
1. 可扩展性
2. 部署独立性
3. 团队组织影响

对于每个方面，请分别说明优点和缺点。
注意：仅讨论以上三个方面，不要延伸到其他话题。
"""
```

**策略四：分段验证**

对于复杂问题，将回答分为多个部分，让模型逐个回答：

```python
prompt = """请分步骤回答以下问题。我将在每一步确认后，再问下一步。

第一步：请仅告诉我中国的四大发明是什么，不需要任何解释。
"""
# 先确认基本信息
# 用户："正确"
# 再追问细节
prompt_2 = """第二步：请针对"造纸术"，详细说明它的发明时间、发明者和历史意义。
如果对任何细节不确定，请明确说明。
"""
```

**策略五：温度参数控制**

在调用模型 API 时，降低 temperature 参数可以减少输出的"创造性"从而减少幻觉：

```python
import anthropic

# 高 temperature → 更多样化但也更容易幻觉
client = anthropic.Anthropic()
response = client.messages.create(
    model="claude-3-opus-20240229",
    max_tokens=1000,
    temperature=0.9,  # 高温度：创造性回答，容易幻觉
    messages=[{"role": "user", "content": prompt}]
)

# 低 temperature → 更保守但也更准确
response = client.messages.create(
    model="claude-3-opus-20240229",
    max_tokens=1000,
    temperature=0.2,  # 低温度：保守回答，减少幻觉
    messages=[{"role": "user", "content": prompt}]
)
```

### 6.4 幻觉缓解策略总结

| 策略 | 适用场景 | 效果 | 代价 |
|------|----------|------|------|
| 给出"不知道"的出口 | 所有需要事实准确性的场景 | ★★★★★ 最有效 | 可能有更多"不知道"的回答 |
| 要求提供引用 | 需要验证的场景 | ★★★☆☆ 部分有效 | 引用可能也是编造的 |
| 限定回答范围 | 开放式问题 | ★★★★☆ 有效 | 回答可能不够全面 |
| 分段验证 | 复杂、关键的问答 | ★★★★★ 非常有效 | 需要多轮交互 |
| 低 temperature | API 调用场景 | ★★★☆☆ 部分缓解 | 创造性降低 |
| RAG（外部知识） | 有知识库的场景 | ★★★★★ 最可靠 | 需要搭建检索系统 |

> **重要提醒**：没有任何 Prompt 技术能完全消除幻觉。最可靠的减少幻觉的方法是 RAG（将外部知识库中的真实文档作为上下文）。Prompt 技术是"第二道防线"，真正的"第一道防线"是确保模型有正确的事实信息可以使用。

## 七、Prompt 设计原则与最佳实践

### 7.1 五大核心设计原则

**原则一：清晰（Clarity）**

你的指令应该让一个**第一次接触这个任务的人**也能理解。

```python
# ❌ 不够清晰
prompt_不清晰 = "分析一下这段代码"

# ✅ 清晰的定义了任务、对象和输出要求
prompt_清晰 = """分析以下 Python 代码的性能瓶颈：

```python
{code}
```

请从以下角度分析：
1. 时间复杂度最差的部分在哪里
2. 是否存在不必要的循环或重复计算
3. 是否有更高效的替代方案

输出格式：每个问题一行，用 bullet point 列出。
"""
```

**为什么清晰如此重要？** LLM 没有"追问"的能力（在单轮对话中）。如果 prompt 表述不清，模型不会说"请澄清"，而是会**猜测你的意图**——猜对了还好，猜错了你就得到了一个莫名其妙的回答。

**原则二：具体（Specificity）**

相比于宽泛的要求，越具体的要求越容易得到满意的结果。

```python
# ❌ 宽泛
"写一封邮件"

# ✅ 具体
"""写一封商务邮件：
收件人：客户张经理
主题：关于下季度合作的提议
语气：正式而友好
要点：
1. 感谢过去一年的合作
2. 提出下季度合作方案（续约+新增两个服务项目）
3. 邀请对方参加下周的线上会议讨论细节
请控制在300字以内。"""
```

**原则三：分而治之（Decomposition）**

复杂任务拆解为多个子任务，每个子任务单独处理。

```python
# ❌ 一次要求太多
prompt_一次 = "分析这篇论文的研究方法、实验结果、创新点和局限性，并与该领域的其他相关工作进行比较，然后给出改进建议。"

# ✅ 分解为子任务
prompt_分解 = """我将分几步请你分析这篇论文。

第一步：请总结这篇论文的研究方法和实验设计。
"""
# → 得到回答后
prompt_第二步 = """第二步：基于你刚才总结的研究方法，请分析论文的主要创新点和局限性。
"""
```

**为什么分解有效？** 模型的注意力在长上下文中会分散。你把任务拆开后，每个子任务中模型的注意力更集中，输出质量更高。

**原则四：角色赋予（Role Assignment）**

给模型一个"角色"可以激活特定的知识域和表达风格。

```python
# ❌ 无角色
"解释什么是 SQL 索引"

# ✅ 有角色
"你是一名有10年经验的 DBA（数据库管理员），正在向刚入职的初级开发人员解释什么是 SQL 索引。请用通俗的语言，但保持技术准确性。"
```

**为什么角色赋予有效？** 角色提示本质上是一个**上下文开关**——它告诉模型从哪个"知识区域"和"表达模式"中采样。"DBA"角色激活了模型在训练数据中学到的"DBA 说话方式"和"DBA 优先考虑的方面"。

**原则五：迭代优化（Iterative Refinement）**

好的 Prompt 不是一次性写出来的——你需要**测试 → 分析不足 → 修改**的循环。

```python
# 迭代过程示例：第一版 → 发现问题 → 第二版

# 第一版
prompt_v1 = "写一篇关于人工智能的博客"
# 问题：输出太宽泛，没有聚焦点

# 第二版：增加具体聚焦
prompt_v2 = "写一篇关于人工智能在医疗诊断领域应用的博客，面向非技术读者"
# 问题：篇幅太长，结构不清晰

# 第三版：增加结构约束
prompt_v3 = """写一篇关于人工智能在医疗诊断领域应用的博客，面向非技术读者。
结构如下：
1. 引言（AI如何改变医疗？）
2. AI在影像诊断中的应用（1-2个具体例子）
3. AI在药物研发中的应用
4. 挑战和局限
5. 总结
每部分控制在100字以内。"""
```

### 7.2 常见的 Prompt 反模式

了解"不该怎么做"和了解"该怎么做"同样重要：

| 反模式 | 示例 | 问题 | 改进 |
|--------|------|------|------|
| **过于模糊** | "帮我写点东西" | 模型不知道你要什么 | 指定主题、格式、长度、受众 |
| **指令冲突** | "简洁地回答，但要详细全面" | 模型无法同时满足矛盾要求 | 只说一个明确要求 |
| **过度引导** | "这个产品是不是很棒？请列出它的优点" | 诱导模型给出biased回答 | "请客观列出优缺点" |
| **假设前提** | "为什么 Python 是最好的编程语言？" | 前提本身有争议 | "Python 和其他语言相比有什么优势和劣势？" |
| **过于复杂的单轮指令** | 一个prompt里包含5个复杂任务 | 模型会忽略部分要求 | 分解为多轮对话 |

### 7.3 Prompt 设计的"元原则"

以上所有技巧背后，可以总结为几条更底层的**元原则**：

**元原则一：模型是"超级模仿者"**

模型会模仿你提供的所有模式——包括好的和坏的。如果你给的示例是高质量的，模型的输出也会高质量。如果你给的指令是模糊的，模型的输出也模糊。

所以：**你的 Prompt 本身就是给模型的"标杆"**。如果希望模型输出结构化的答案，你的 Prompt 本身就应该结构化。

**元原则二：模型没有"常识"——只有"统计模式"**

对人类来说显而易见的常识（如"不要编造信息"），对模型来说并不是内建的。模型不"知道"它应该诚实——它只知道在它的训练数据中，大多数情况下"诚实的回答"和"编造的回答"在统计上看起来差不多。

所以：**不要假设模型会做"正确的事"——你必须明确要求它**。

**元原则三：Prompt 工程是"实验科学"**

几乎没有"放之四海而皆准"的 Prompt 写法。同一个 Prompt 在不同的模型上（GPT-4 vs Claude vs Llama）表现可能完全不同。同一个 Prompt 在同一个模型的不同版本上也可能表现不同。

所以：**测试、测试、再测试**。没有测试的 Prompt 只是"猜测"。

### 7.4 Prompt 开发的工作流程

一个系统化的 Prompt 开发流程应该包含以下步骤：

```
步骤1：明确需求
  └── 我要模型完成什么任务？输入是什么？输出应该是什么格式？
  
步骤2：编写初版 Prompt
  └── 使用本文提到的原则：清晰、具体、角色、约束

步骤3：测试
  └── 用3-5个测试用例评估输出质量

步骤4：分析失败模式
  └── 模型在哪些输入上表现不好？为什么？

步骤5：针对性改进
  └── 修改 Prompt，解决识别出的问题

步骤6：回归测试
  └── 确保修改没有破坏之前表现好的用例

步骤7：版本管理
  └── 保存不同版本的 Prompt，记录改动和效果
```

### 7.5 总结：Prompt Engineering 进阶路线图

| 阶段 | 掌握的技术 | 能解决的场景 |
|------|-----------|-------------|
| **入门** | Zero-Shot Prompting 技巧 | 简单的问答、翻译、总结 |
| **进阶级** | Few-Shot Prompting | 需要格式控制、分类、提取 |
| **高手级** | Chain-of-Thought、Self-Consistency | 数学推理、逻辑分析、复杂问题 |
| **系统级** | RAG Prompt 设计、幻觉缓解策略 | 知识库问答、事实性要求高的场景 |
| **专家级** | 多轮优化、复杂工作流编排 | 自动化 Agent、多步推理系统 |

**与前面各部分的联系**：理解 Transformer 的注意力机制（第十四部分）是理解为什么 Chain-of-Thought 有效的理论基础——注意力机制让模型在生成长文本时能"回头看"之前的内容，多步推理正是利用了这一点。RAG 和向量数据库（第七、八部分）构成了 RAG Prompt 的基础设施层。

---

*Prompt Engineering 不是一个"学一次就会"的技能——随着模型能力的提升和新的 Prompt 技术的出现，你需要在实践中持续迭代和优化。最好的学习方式是：写一个 Prompt → 观察模型的输出 → 思考为什么这里不够好 → 修改 → 再测试。*
