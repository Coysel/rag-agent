# 第十五部分：RAG / Agent / LangGraph

---

## 一、RAG（Retrieval-Augmented Generation）综述

### 1.1 什么是 RAG？

RAG（检索增强生成）是一种将**信息检索**与**文本生成**相结合的技术架构。它的核心思路非常直观：当 LLM 需要回答一个问题时，不是仅靠模型自身的"记忆"来生成答案，而是先从外部知识库中检索出相关的文档片段，然后将这些检索结果作为上下文提供给 LLM，让 LLM 在这个上下文的基础上生成答案。

为什么要这样做？要理解 RAG 的价值，我们需要先看清 LLM 本身的三个固有缺陷：

**第一，知识截止日期（Knowledge Cutoff）。** 任何 LLM 的训练数据都有截止日期。例如 GPT-4 的知识截止于 2023 年，这意味着如果询问"2024 年的诺贝尔奖得主是谁"，模型要么编造一个答案，要么回答"我不知道"。而 RAG 可以检索到最新的新闻文章，让模型基于实时信息回答。

**第二，幻觉问题（Hallucination）。** LLM 本质上是一个"下一个词预测器"，它没有"事实"和"虚构"的概念。当它不确定答案时，它会"顺着话头往下说"，产生看似合理但实际错误的内容。RAG 通过提供检索到的真实文档作为依据，大幅降低幻觉的概率——因为模型不再需要"编造"事实，它只需要"总结"给出的材料。

**第三，领域知识不足。** 通用 LLM 对特定领域（如医疗、法律、企业内部文档）的知识覆盖往往不够深入。传统做法是对模型进行微调（Fine-tuning），但微调成本高、周期长、且每次更新知识都需要重新微调。RAG 则只需要更换知识库中的文档即可，成本极低。

### 1.2 RAG 的三阶段完整流程

RAG 的完整工作流程可以分为三个阶段：**索引（Indexing）**、**检索（Retrieval）** 和 **生成（Generation）**。

#### 第一阶段：索引（Indexing）

索引阶段是"准备工作"，在用户提问之前就已经完成。它的目的是将原始文档库转换为计算机可以快速搜索的结构化格式。

**步骤一：文档加载与解析。** 原始文档可能是 PDF、Word、HTML、Markdown 等各种格式。首先需要将这些文档加载到内存中，并解析为纯文本。例如，PDF 需要提取每一页的文字内容，HTML 需要去除标签只保留正文。

**步骤二：文档分块（Chunking）。** 解析后的文档通常非常长（一本书可能有数十万字），不能直接存入向量数据库。需要将其切分成较小的"块"（Chunks）。分块策略直接决定了检索质量的上限，我们将在后文详细讨论。

**步骤三：向量化（Embedding）。** 每个文本块通过 Embedding 模型（如 text-embedding-3-small、bge-large-zh 等）转换为一个固定维度的向量（通常是 768 维或 1536 维）。这个向量编码了文本的语义信息——语义相近的文本在向量空间中距离相近。

**步骤四：存入向量数据库。** 将文本块及其对应的向量存入专门的向量数据库（如 Pinecone、Weaviate、ChromaDB、FAISS）。向量数据库支持基于向量距离的快速相似度搜索。

#### 第二阶段：检索（Retrieval）

当用户提出一个问题时，系统开始执行检索。

**步骤一：问题向量化。** 使用与索引阶段**相同**的 Embedding 模型，将用户问题转换为向量。这里必须使用同一模型，否则向量空间不统一，会导致检索结果完全偏离。

**步骤二：向量相似度搜索。** 在向量数据库中，计算问题向量与所有文档块向量的距离。最常用的度量方式是余弦相似度（Cosine Similarity）或点积（Dot Product）。余弦相似度的值域为 [-1, 1]，越接近 1 表示语义越相似。

**步骤三：返回 Top-K 结果。** 选择距离最近的 K 个文档块返回（K 通常取 3-10）。这 K 个文档块就是在语义上最有可能回答用户问题的内容。

#### 第三阶段：生成（Generation）

**步骤一：拼装 Prompt。** 将检索到的文档块和用户的问题组合成一个结构化的 Prompt。典型的模板如下：

```
你是一个问答助手。请基于以下提供的文档内容回答问题。
如果文档内容不足以回答问题，请如实告知。

文档内容：
{检索到的文档块}

问题：{用户问题}
答案：
```

**步骤二：LLM 生成。** 将拼装好的 Prompt 发送给 LLM，模型基于文档上下文生成最终答案。由于文档中已经包含了相关信息，LLM 不需要"回忆"或"编造"，只需要进行信息提取和语言组织。

**步骤三：返回答案并附带引用。** 优秀的 RAG 系统还会在答案中标注信息来源（如"根据文档第 3 章的描述……"），让用户可以追溯验证。

### 1.3 文档分块（Chunking）的深度分析

分块策略是 RAG 系统中影响检索质量最关键的因素之一。分块过小或过大都会带来严重问题。

#### 为什么分块大小如此重要？

考虑一个场景：你有一本 300 页的技术手册，用户问"如何配置 TLS 证书？"。

- **如果每个块太小（如 50 个 token）**：一个块可能只包含"TLS 证书需要配置以下参数"，而不包含具体的配置步骤。检索到这个块后，LLM 拿到的信息是不完整的，无法生成正确的答案。
- **如果每个块太大（如 5000 个 token）**：一个块可能包含整章内容。虽然信息完整了，但噪声也大幅增加——块中大部分内容与用户问题无关（比如同时包含了 TCP 配置、HTTP 路由等内容），LLM 需要从大量噪声中提取相关信息，容易出错。同时，大块意味着更少的块数，细粒度检索的能力下降。

#### 常见的分块策略

**固定大小分块（Fixed-size Chunking）。** 这是最简单直接的方法：设定一个固定的 token 数（如 256、512、1024），按照这个长度切分文档，相邻块之间可以设置重叠（Overlap）。重叠的作用是避免关键信息恰好落在两个块的分界处被切断。例如，如果块大小为 512，重叠为 64，则第一个块包含 token 1-512，第二个块包含 449-960，依此类推。

**语义分块（Semantic Chunking）。** 固定大小分块的问题是它完全不了解内容的语义结构，可能在一个段落中间截断。语义分块的目标是让每个块包含一个"完整的语义单元"——例如一个段落、一个小节。实现方式：计算相邻句子之间的语义相似度，当相似度突然下降时（表示话题切换），在此处分块。这种方式生成的块更加自然，但计算成本更高。

**递归分块（Recursive Chunking）**。LangChain 中常用的一种策略：先用大分隔符（如段落分隔符 `\n\n`）分割，如果得到的块仍然过大，则使用更细的分隔符（如句号、逗号）继续分割。这结合了固定大小和语义分块的优点。

#### 如何选择块大小？

块大小没有"万能最佳值"，取决于以下因素：

- **文档类型**：代码文档可能需要较小的块（因为每个函数是独立的），而叙事性文章可能需要较大的块（因为上下文连贯）。
- **检索任务类型**：问答任务需要信息完整，块适当大一些（512-1024 token）；相似度匹配任务可以小一些（128-256 token）。
- **LLM 的上下文窗口**：如果使用 128K 上下文的模型，可以承受更大的块；如果是 4K 上下文的模型，则必须严格控制块大小。
- **Embedding 模型**：有些模型对短文本效果好，有些对长文本效果好，需要参考模型文档。

经验值：对于大多数中英文混合的问答场景，**512 token 的块大小 + 128 token 的重叠**是一个合理的起点，然后通过实验调优。

### 1.4 为什么需要混合检索？

#### 稠密检索（Dense Retrieval）的局限性

基于向量的语义检索虽然强大，但它并非万能。典型的问题是：

- **专有名词和术语**：如果用户搜索"React Native 的 FlatList 组件"，语义检索可能会找到"移动端列表渲染"的内容，但因为 FlatList 是一个精确的术语名，语义检索并不能保证匹配到包含"FlatList"这个词的文档。
- **缩写和代码**：搜索"OOM"（Out of Memory）时，语义检索可能不知道 OOM 和"内存溢出"是同一概念，除非 Embedding 模型在训练时见过这种对应关系。
- **短查询**：当查询只有一两个词时，语义检索的效果往往不如简单的关键词匹配。

#### 稀疏检索（Sparse Retrieval）的回归

BM25（Best Matching 25）是一种经典的"稀疏检索"算法，它不基于语义，而是基于**词频统计**。核心思想：

- 一个词在文档中出现次数越多（Term Frequency / TF），该文档与查询越相关
- 但同时，一个词在越多的文档中出现（Inverse Document Frequency / IDF），它的区分度就越低，权重应该降低
- BM25 对 TF 做了非线性归一化（通过参数 k1 控制），避免某个词重复出现过多导致分数失控

BM25 的优势恰恰弥补了语义检索的短板：它对精确词匹配非常敏感，搜索"FlatList"就一定能找到包含"FlatList"的文档。但它也有明显的劣势：它无法理解"苹果"和"iPhone"之间的语义关系。

#### 混合检索的工作方式

混合检索（Hybrid Retrieval）的思路是：**同时使用稠密检索和稀疏检索，然后融合两者的结果。**

具体流程：

1. 用户提出查询
2. 将查询同时发送给稠密检索器（Embedding + 向量搜索）和稀疏检索器（BM25）
3. 两个检索器各自返回排序的结果列表
4. 使用融合算法合并两个结果列表，生成最终的排序

#### RRF（Reciprocal Rank Fusion）融合算法——为什么用它？

融合算法的核心挑战在于：**两个检索器输出的分数值域完全不同**。

- BM25 的分数理论上可以是任意正数，无上界
- 余弦相似度在 [-1, 1] 范围内

如果简单地将两者分数相加或加权求和，必须先做归一化。但归一化本身就有问题——min-max 归一化受极端值影响过大；z-score 归一化假设数据服从正态分布，但 BM25 的分数分布往往不是。

RRF 巧妙地绕过了这个问题：**它完全抛弃分数，使用排名（Rank）代替。**

RRF 公式：

```
RRF_score(d) = Σ 1 / (k + rank_i(d))
```

对于每个检索器 i，文档 d 的排名为 rank_i(d)，k 是平滑常数（通常取 60）。

为什么用排名代替分数就能解决问题？因为**排名是绝对值无关的**——无论 BM25 的分数分布如何，只要文档 A 排在文档 B 前面，A 的排名就小于 B。排名提供了一个跨检索器统一的比较尺度。

RRF 中 k 的作用是什么？k 控制排名差异的权重。当 k=60 时：

- 排名 1 的文档贡献：1/61 ≈ 0.0164
- 排名 10 的文档贡献：1/70 ≈ 0.0143
- 排名 100 的文档贡献：1/160 ≈ 0.0063

可以看到，排名靠前的文档贡献更大，但差距并不悬殊——排名 1 和排名 10 的贡献只差约 13%。这保证了排名靠后的优秀文档仍然有机会进入最终结果列表。

k 值越小，高排名文档的权重越大（差距越悬殊）；k 值越大，所有排名的贡献越平均。

### 1.5 RAG 面临的主要挑战

虽然 RAG 架构设计优雅，但在实际应用中仍然面临多个核心挑战：

**检索质量瓶颈。** 这是 RAG 系统最根本的约束——**检索质量决定了回答质量的上限。** 如果检索到的文档完全不相关，无论 LLM 多强大，都不可能凭空生成正确答案。这个原则可以表述为：Garbage In, Garbage Out（垃圾进，垃圾出）。正因如此，RAG 系统的调优重点往往不在 LLM 本身，而在检索环节。

**多步推理困难。** 对于简单的事实性问题（"巴黎的首都是什么"），RAG 效果极好。但对于需要多步推理的问题（"2024 年巴黎奥运会新增了哪些项目？这些项目在上一届奥运会上有没有出现？"），模型可能需要执行多次检索才能获得完整信息。这就引出了"Agentic RAG"的概念——赋予 RAG 系统多步推理和自主决策的能力，我们将在后文展开。

**检索过度。** 对于简单问题（"现在是几点"），RAG 仍然会执行向量检索，增加了不必要的延迟和成本。一个成熟的 RAG 系统应该能够判断何时需要检索、何时不需要。

## 二、Agent（智能代理）基础

### 2.1 什么是 LLM Agent？

在 AI 领域，"Agent"（代理）指的是一个能够**自主感知环境、做出决策、执行行动**的系统。具体到 LLM Agent，它是指以大型语言模型为核心"大脑"，可以自主调用工具、执行任务、并根据反馈调整行为的程序。

Agent 与普通 LLM 调用的根本区别在于：**普通 LLM 调用是"一问一答"的被动响应，而 Agent 是"目标驱动"的主动执行。** 你给 Agent 一个目标（"帮我订一张明天从北京到上海的机票"），它会自己规划步骤（搜索航班、比较价格、填写信息、提交订单），而不是等你一步步指挥。

### 2.2 Agent 的四个核心组件

一个完整的 Agent 系统包含四个核心组件：

#### 组件一：大脑（Brain / LLM）

这是 Agent 的"决策中枢"，通常是一个 LLM。大脑负责：
- 理解用户的原始目标
- 分析当前状态和已经获取的信息
- 决定下一步应该采取什么行动
- 判断任务是否已经完成

大脑的质量直接决定了 Agent 的智能水平。更强的 LLM（如 GPT-4、Claude 3.5）在 Agent 任务中表现远优于小模型，因为它们能够理解更复杂的指令、做出更合理的规划。

#### 组件二：感知（Perception）

Agent 需要能够感知外部环境。感知的渠道包括：
- 用户输入的文本（最基础）
- 工具调用的返回结果（如数据库查询结果、API 响应）
- 系统信息（当前时间、已执行的步骤数）
- 多模态输入（图像、音频等）

感知机制决定了 Agent 能获取多丰富的信息。例如，一个只能感知文本的 Agent 无法处理"请根据这张图片写描述"的请求。

#### 组件三：行动（Action / Tools）

这是 Agent 作用于外部世界的方式。行动通常表现为"工具调用"：
- 搜索引擎搜索
- 数据库查询
- API 调用
- 文件读写
- 代码执行
- 发送邮件

工具集合越丰富、接口越清晰，Agent 能完成的任务就越复杂。每个工具需要提供清晰的描述（Description），让 LLM 知道"这个工具是做什么的、什么时候该用它"。

#### 组件四：记忆（Memory）

Agent 需要记忆来维持任务的连贯性。记忆分为两层：

**短期记忆（Short-term Memory）**：指当前任务内的对话历史和中间状态。Agent 需要记住自己已经执行了哪些步骤、获得了哪些信息。这通常通过将完整的历史记录放入上下文中实现（In-Context Memory）。

**长期记忆（Long-term Memory）**：指跨任务的知识积累。例如 Agent 在执行任务过程中学到的信息，可以存入外部存储，在未来的任务中使用。长期记忆通常通过 RAG 或向量数据库实现。

### 2.3 Chain（链） vs Agent（代理）——如何选择？

这是一个重要的设计决策：什么时候应该用 Chain，什么时候应该用 Agent？

**Chain（链）** 是一种预定义好的、固定的执行流程。例如：

```
接收问题 → 翻译成英文 → 搜索英文资料 → 翻译回中文 → 总结
```

这个流程是硬编码的、确定性的。每个步骤做什么、下一步去哪，都在代码中写死了。

**Agent（代理）** 则将流程控制权交给了 LLM。步骤和顺序由模型动态决定。

对比表格：

| 维度 | Chain | Agent |
|------|-------|-------|
| 执行流程 | 固定，代码硬编码 | 动态，LLM 自主决策 |
| 可控性 | 高，行为可预测 | 低，可能有意外行为 |
| 适用场景 | 流程明确的常规任务 | 需要灵活决策的复杂任务 |
| 调试难度 | 低，每步结果是确定的 | 高，结果由 LLM 决定 |
| 安全性 | 高 | 需要护栏（Guardrails） |
| Token 消耗 | 低，按预定路径执行 | 高，可能反复调用 |
| 示例 | 客服对话分类器 | 自动化研究助手 |

**何时用 Chain：** 当任务的步骤是固定的、确定的。例如：一个"内容审核"流程——先检查是否包含敏感词，如果有则标记，如果没有则发布。这个过程不需要 AI 来"决定"下一步做什么。

**何时用 Agent：** 当任务的路径不确定，需要根据中间结果动态调整。例如："研究某个公司"——Agent 需要先搜索公司基本信息，根据返回的内容决定下一步搜索什么（财务数据？产品信息？竞争对手？）。

一个常见的误解是："Agent 比 Chain 高级，所以应该总是用 Agent"。事实恰恰相反：**能用 Chain 解决的问题，就不要用 Agent。** Chain 更便宜、更可控、更易调试。Agent 应该只用在 Chain 无法解决的场景。

### 2.4 ReAct 模式：思维+行动的循环

ReAct 是 "Reasoning + Acting"（推理+行动）的缩写，由 Yao 等人于 2022 年提出。它是最基础的 Agent 工作模式，几乎所有现代 Agent 系统都基于 ReAct 或其变体。

#### ReAct 的核心思想

ReAct 的核心洞察是：**推理和行动不是分离的两个阶段，而是交替进行的循环过程。** 人类解决问题的方式就是如此——我们想一步、做一步、观察结果、再想下一步。ReAct 将这个循环形式化：

```
Thought（思考）→ Action（行动）→ Observation（观察）→ Thought（思考）→ ...
```

具体来说，每次循环分为三个步骤：

1. **Thought（思考）**：LLM 分析当前状态，决定下一步需要做什么。这是"推理"环节。例如："用户问的是今年诺贝尔奖的得主，我目前还没有任何信息，需要先搜索一下。"

2. **Action（行动）**：LLM 调用一个工具。这是"行动"环节。例如：`search("2024 Nobel Prize winners")`。

3. **Observation（观察）**：将工具返回的结果作为新的观察输入。例如：搜索返回了诺贝尔奖的新闻。

然后模型再次进入 Thought 阶段，分析观察到的信息，决定：
- 如果信息足够 → 生成最终答案（Action: Answer）
- 如果信息不够 → 进行下一轮搜索或采取其他行动（Action: 再搜索）

#### ReAct 的完整示

假设用户问："特斯拉 2024 年第一季度的营收是多少？"

Agent 的运行轨迹：

```
Thought 1: 我需要查找特斯拉 2024 年 Q1 的财务数据，先搜索一下。
Action 1: search("Tesla 2024 Q1 revenue")
Observation 1: 返回了特斯拉 Q1 财报新闻，包含营收数据。

Thought 2: 搜索结果中有营收数据，让我提取具体数字。
Action 2: 阅读搜索结果中的相关段落
Observation 2: 特斯拉 2024 年 Q1 营收为 213.0 亿美元

Thought 3: 我已经获得了足够的信息，可以回答用户了。
Action 3: answer("特斯拉 2024 年第一季度的营收为 213.0 亿美元。")
```

#### 为什么 ReAct 比单纯的思维链（Chain-of-Thought）效果好？

思维链（CoT）让模型"一步一步思考"，但思考的结果并不能影响外部世界——模型只是在内部推理。而 ReAct 通过工具调用让推理结果作用于外部世界，并获取新的信息来修正推理。好比 CoT 是闭着眼睛解题，ReAct 是睁着眼睛解题——你不仅可以想，还可以查资料、做实验来验证你的想法。

#### ReAct 的 Prompt 模板

ReAct 模式通常通过特定的 Prompt 格式来触发：

```
你是一个智能助手，可以通过思考、行动、观察来完成任务。

可用工具：
- search(query): 搜索互联网
- calculator(expression): 执行数学计算

请按照以下格式回复：

思考：你的推理过程
行动：调用哪个工具，参数是什么
观察：工具返回的结果
...（可以重复思考-行动-观察）
思考：我已经获得足够信息
最终答案：你的最终回答
```

## 三、LangGraph：有状态的 Agent 框架

### 3.1 LangGraph 的核心理念

LangGraph 是一个基于 LangChain 生态系统的框架，专门用于构建有状态、多步骤的 Agent 应用。它的核心思想是：**将 Agent 的工作流建模为一个有向图（Directed Graph）。**

为什么用图？因为 Agent 的执行路径不是线性的。一个 Agent 可能需要根据中间结果决定下一步要做什么，可能出现循环（重复执行某个步骤）、分支（根据条件走不同路径）、以及并行（同时执行多个操作）。用图来描述这些复杂的控制流，是最自然的方式。

在 LangGraph 之前，构建 Agent 通常有两种方式：

1. **while 循环 + if-else 条件判断**：手动管理状态和流程。代码逻辑分散在各处，难以调试和维护。
2. **LangChain 的 AgentExecutor**：封装了 ReAct 循环，但定制能力有限。

LangGraph 提供了一种介于两者之间的方案：**你定义图的结构（节点和边），LangGraph 负责状态的维护和流转。**

### 3.2 三个核心概念

LangGraph 引入三个基本原语：**State（状态）**、**Node（节点）**、**Edge（边）**。

#### State（状态）——图中流转的数据

State 是贯穿整个图的生命周期的数据结构。你可以将 State 理解为 Agent 的"工作记忆"——它包含了 Agent 在当前任务中积累的所有信息。

在 LangGraph 中，State 使用 Python 的 TypedDict（或 Pydantic 的 BaseModel）定义：

```python
from typing import TypedDict, Annotated
from langgraph.graph.message import add_messages

class AgentState(TypedDict):
    # 消息历史，使用 add_messages reducer 来合并
    messages: Annotated[list, add_messages]
    # 用户的原始查询
    query: str
    # 最终答案
    answer: str
    # 当前已执行的步数
    step_count: int
```

这里的 `Annotated[list, add_messages]` 是 LangGraph 的关键设计之一。`Annotated` 的第一个参数是字段的类型，第二个参数是 **Reducer**（规约函数）。为什么需要 Reducer？

在图中，多个节点可能同时更新同一个 State 字段。如果没有 Reducer，后更新的值会覆盖先更新的值。Reducer 决定了如何合并这些更新。`add_messages` 是 LangGraph 内置的 Reducer，它会将新消息添加到列表中而不是替换——这符合对话场景的需求：每次新的 LLM 响应应该追加到消息历史中。

对于普通的字段（如 `query`、`answer`），多个节点同时更新就意味着"最后一个写入的生效"。如果你需要叠加效果（如计数器累加），也可以指定自定义的 Reducer。

#### Node（节点）——实际的执行单元

Node 是图中的"处理单元"。每个 Node 是一个函数，接收当前的 State 并返回 State 的更新：

```python
def search_node(state: AgentState) -> dict:
    """搜索节点：从 State 中取出 query，执行搜索，返回结果。"""
    query = state["query"]
    results = call_search_api(query)
    return {
        "search_results": results,
        "step_count": state["step_count"] + 1
    }

def answer_node(state: AgentState) -> dict:
    """回答节点：基于搜索结果生成最终答案。"""
    context = state["search_results"]
    answer = llm.generate(f"基于以下内容回答问题：\n{context}")
    return {"answer": answer}
```

Node 的输入是当前的完整 State，输出是一个字典——只包含需要更新的字段。没有在字典中出现的字段保持不变。这种设计模式（"partial update"）使得每个 Node 职责明确，只关注自己需要修改的部分。

#### Edge（边）——状态的流转路径

Edge 定义 Node 之间的连接方式。LangGraph 有两种类型的边：

**普通边（Normal Edge）**：确定性的连接，Node A 执行完成后总是进入 Node B。

```python
workflow.add_edge("search_node", "answer_node")
# 表示 search_node 执行完后，无条件进入 answer_node
```

**条件边（Conditional Edge）**：根据当前 State 动态决定下一步去哪里。

```python
workflow.add_conditional_edges(
    "search_node",           # 从这个节点出发
    router_function,         # 路由函数，接收 State，返回路径名称
    {                        # 路径名称到目标节点的映射
        "to_answer": "answer_node",
        "to_search": "search_node",
        "to_human": "human_review_node"
    }
)
```

条件边是 Agent 灵活性的核心来源。通过条件边，Agent 可以实现循环（信息不够就继续搜索）、分支（根据结果类型走不同处理路径）、以及人机协作（将不确定的情况交给人工处理）。

路由函数本身只是一个普通的 Python 函数：

```python
def router_function(state: AgentState) -> str:
    """判断信息是否足够，决定下一步。"""
    if state["step_count"] >= 5:
        return "to_human"       # 超过5步还没结果，交给人工
    if information_sufficient(state):
        return "to_answer"      # 信息足够，生成答案
    else:
        return "to_search"      # 信息不够，继续搜索
```

### 3.3 完整的 ReAct Agent 图结构

将上述概念组合起来，一个典型的 ReAct Agent 在 LangGraph 中的表达如下：

```
START ──→ should_search ──→ search_tool
                ↑                │
                │                ↓
                ├── should_answer
                │         │
                │         ↓
                │       answer_node ──→ END
                │
                └── should_ask_human ──→ human_review ──→ END
```

- `should_search`：判断是否需要搜索（条件边）
- `search_tool`：执行搜索（节点）
- `should_answer`：判断是否已经获得足够信息（条件边）
- `answer_node`：生成最终回答（节点）
- `human_review`：人工介入（节点）

### 3.4 Checkpoint 机制——让 Agent 拥有持久记忆

Checkpoint 是 LangGraph 中最具特色的机制之一。它在图的**每个节点执行后**自动保存 State 的快照。这意味着整个 Agent 的执行轨迹都被完整记录了下来。

#### Checkpoint 的核心用途

**1. 中断恢复（Fault Tolerance）**

假设 Agent 已经执行了 10 步，进行了 5 次搜索、调用了 3 次 API，突然程序崩溃了。没有 Checkpoint 的情况下，一切都要从头开始。有了 Checkpoint，只需要从最近的保存点恢复，State 中的消息历史、变量值都完好无损。

```python
# 使用 Thread 来持久化 Checkpoint
thread = {"configurable": {"thread_id": "task_001"}}
config = {"configurable": {"thread_id": "task_001"}}

# 第一次执行到中断点
result = graph.invoke({"query": "研究一下RISC-V架构"}, config)

# 即使程序重启，也可以通过 thread_id 恢复
result = graph.invoke(None, config)  # 从上次中断继续
```

**2. 调试可观测性（Observability）**

Checkpoint 记录了每一步的 State，这意味着你可以回溯查看 Agent 在执行过程中每一步的状态：

```python
# 查看所有历史状态
for state in graph.get_state_history(config):
    print(f"Step {state.metadata['step']}: {state.values['messages'][-1]}")
```

这对于调试 Agent 的异常行为极其有用——你可以精确地看到 Agent 是在哪一步做出了错误决策，以及当时的输入是什么。

**3. 人类审批（Human-in-the-Loop）**

这是 Checkpoint 最有实际价值的应用场景。在某些关键节点（如"执行数据库写操作"、"发送邮件"），我们希望先暂停执行，等待人类确认后再继续：

```python
workflow.add_node("confirm_action")
workflow.add_edge("execute_sql", "confirm_action")

# 在 confirm_action 节点设置中断
graph = workflow.compile(checkpointer=checkpointer)
graph.interrupt_before(["confirm_action"])
```

实际执行流程：

```python
# 第一步：Agent 执行到 confirm_action 前暂停
result = graph.invoke({"task": "更新用户资料"}, config)
# result 中包含了"等待审批"的信息

# 人工审查后：继续执行
result = graph.invoke(
    None,  # 不需要再次输入，Continue 即可
    config
)
```

**4. 分支回溯（Branching）**

人类审批时，不仅可以"批准"或"拒绝"，还可以修改 State 中的某些值，然后继续。这相当于创建了一个新的分支——从历史状态分叉出一条新的执行路径。

#### Checkpoint 的持久化存储

Checkpoint 需要写入持久化存储。LangGraph 支持多种存储后端：

- **内存（MemorySaver）**：临时存储，重启后丢失，适合开发和测试
- **SQLite（SqliteSaver）**：文件级存储，重启后保留，适合小规模生产
- **PostgreSQL（PostgresSaver）**：生产级存储，支持并发访问

```python
from langgraph.checkpoint import MemorySaver

checkpointer = MemorySaver()
graph = workflow.compile(checkpointer=checkpointer)
```

### 3.5 Human-in-the-Loop（人机协作）

Human-in-the-Loop（HITL，人在回路中）是 Agent 系统安全性的重要保障。其核心理念：**让 AI 自主决策，但在关键环节引入人工审核。**

#### 为什么需要 HITL？

即使最强的 LLM 也会犯错。在以下场景中，HITL 是不可或缺的：

- **高风险的决策**：如医疗诊断、金融交易、法律建议
- **不可逆的操作**：如删除数据库记录、发送合同文件
- **权限限制**：Agent 没有权限执行的操作（如支付），必须由人工完成

#### HITL 的实现方式

在 LangGraph 中，HITL 通过 **interrupt（中断）** 机制实现：

```python
# 在编译图时指定中断点
graph = workflow.compile(
    checkpointer=checkpointer,
    interrupt_before=["send_email_node"]  # 在发送邮件前中断
)

# 执行时，到达 send_email_node 前自动暂停
result = graph.invoke({"task": "给客户发报价单"}, config)
print("等待人工确认发送邮件...")

# 等待用户输入确认
user_confirm = input("确认发送？(y/n)")
if user_confirm == "y":
    # 继续执行
    result = graph.invoke(None, config)
else:
    print("已取消")
```

#### 动态中断 vs 固定中断

LangGraph 支持两种中断方式：

- **固定中断**：在编译时指定节点，每次执行到该节点都会中断。适用于已知的安全关键节点。
- **动态中断**：在节点执行过程中，根据当前状态决定是否中断。Node 函数可以返回一个特殊值来触发中断，适用于条件性的审批场景。

### 3.6 为什么选 LangGraph 而不是手写 while 循环？

这个问题常被问及。两者各有适用场景，理解其区别有助于做出正确的技术选型：

| 维度 | while + if-else | LangGraph |
|------|----------------|-----------|
| 状态管理 | 自己定义变量，容易遗漏或冲突 | TypedDict + Reducer 自动合并 |
| 可观测性 | 需要手动写日志 | Checkpoint 自动记录每一步 |
| 可视化 | 代码即文档，难以直观理解 | 图结构可导出 Mermaid/PNG |
| 中断恢复 | 需要自己实现序列化 | 内置 Checkpoint 机制 |
| 流式输出 | 需要手动实现异步 | 原生支持 astream |
| 组合复用 | 循环逻辑分散，难以复用 | 多个 Graph 可以组合嵌套 |
| 并发控制 | 自己处理竞态条件 | 单线程保证 State 一致性 |

简单来说：**如果你只是在做一个几步就完成的任务（Chain 就够用了），用 while 循环也可以。但如果你要构建一个复杂、多步、可能中断恢复的 Agent 系统，LangGraph 提供的结构化支持会大幅降低维护成本。**

## 四、Agentic RAG：让 RAG 学会思考

### 4.1 什么是 Agentic RAG？

Agentic RAG 是 RAG 和 Agent 两种思想的融合。普通的 RAG 是"一次检索、一次生成"的静态流程，而 Agentic RAG 赋予系统自主规划和多步检索的能力。

#### 为什么需要 Agentic RAG？

传统 RAG 的核心局限在于：**它无法处理需要多步推理的复杂问题。** 例如：

用户问："2024 年巴黎奥运会新增了哪些比赛项目？这些项目在 2021 年东京奥运会中有没有出现过？"

要回答这个问题，模型需要：
1. 检索"2024 巴黎奥运会 新增项目" → 得到列表 [A, B, C, D]
2. 对于项目 A，检索"东京奥运会 A" → 确认是否存在
3. 对于项目 B，检索"东京奥运会 B" → 确认是否存在
4. 重复直到所有项目检查完毕
5. 汇总结果

传统 RAG 无法执行这种"迭代检索"——它只会检索一次就生成答案。而 Agentic RAG 可以自主规划上述步骤，按需执行多次检索。

### 4.2 Agentic RAG 的架构

Agentic RAG 的典型架构包含以下角色：

```
用户问题
    │
    ▼
┌─────────────────────┐
│    Supervisor Agent  │  ← 整体规划、决策、协调
│    (Master Agent)    │
└────────┬────────────┘
         │
    ┌────┼────┬────┬────┐
    ▼    ▼    ▼    ▼    ▼
 ┌────┐┌────┐┌────┐┌────┐┌──────┐
 │搜索 ││问答 ││摘要 ││代码 ││其他工具│
 └────┘└────┘└────┘└────┘└──────┘
```

**Supervisor Agent（主管代理）**：这是大脑，负责：
- 将大问题分解为子问题
- 决定使用哪个工具或调用哪个子 Agent
- 评估子任务的结果是否足够
- 合成最终答案

**Tool Agents（工具代理）**：执行具体任务的"手"，每个 Tool Agent 封装了特定的能力：
- 搜索 Agent：封装了查询改写、多源搜索、结果重排序
- QA Agent：封装了基于特定文档的问答
- 摘要 Agent：封装了长文档摘要生成
- 代码 Agent：封装了代码执行和结果分析

### 4.3 Agentic RAG 的典型工作流

以回答"对比 Transformer 和 Mamba 架构的优劣势"为例：

```
Step 1: Supervisor 分析问题
   → "我需要先分别了解 Transformer 和 Mamba 的原理"
   → 调用 Search Agent 同时搜索两者

Step 2: Search Agent 返回结果
   → Transformer: 架构概述、优缺点
   → Mamba (SSM): 架构概述、优缺点

Step 3: Supervisor 评估
   → "已有基本信息，但需要更深入的对比"
   → 调用 QA Agent 针对具体文档提问
   → "Transformer 的注意力机制的缺点是什么？"
   → "Mamba 的状态空间模型如何解决长序列问题？"

Step 4: QA Agent 返回更详细的信息

Step 5: Supervisor 判断信息足够
   → 调用 Answer Agent 合成最终对比分析

Step 6: 返回完整答案给用户
```

### 4.4 从 RAG 到 Agentic RAG 的演进路径

不是所有场景都需要完整的 Agentic RAG。实际项目中可以渐进式引入：

**Level 0：基础 RAG**
一次检索 + 一次生成。适合简单的事实性问题。

**Level 1：查询改写 RAG**
在检索前加入"查询改写"步骤：LLM 将用户问题重写为更易检索的形式。例如用户问"那个自行车比赛什么时候开始"，改写为"2024 年环法自行车赛开始日期"。这已经引入了简单的 Agent 思维。

**Level 2：多步检索 RAG**
Agent 可以执行多次检索，每次检索基于上一次的结果调整搜索策略。适合研究类问题。

**Level 3：工具增强 RAG**
除了检索文档，Agent 还可以调用其他工具——如计算器、翻译器、代码执行器等。RAG 从"检索+问答"进化为"检索+工具+推理"的全功能系统。

**Level 4：多 Agent 协作 RAG**
多个专用 Agent 协同工作，由 Supervisor 统一调度。适合高度复杂的任务。

## 五、MCP（Model Context Protocol）

### 5.1 MCP 是什么？为什么需要它？

MCP（Model Context Protocol）是 Anthropic 提出的一种**开放协议**，旨在标准化 AI 模型与外部工具和数据源之间的交互方式。

理解 MCP 的最好方式是用一个类比：**MCP 之于 AI 应用，就像 USB-C 之于硬件设备。**

在 USB-C 之前，手机、电脑、打印机各有各的充电接口和数据传输协议——你需要不同的线缆连接不同的设备。USB-C 通过一个统一的接口标准，让所有设备都可以用同一根线缆连接。

在 MCP 之前，每个 LLM 应用与外部工具的交互方式各不相同：
- OpenAI 使用自己的 Function Calling 格式
- LangChain 使用自己的 Tool 抽象
- 每个框架定义工具的方式都不一样
- 工具开发者需要为每个框架适配

MCP 的目标是：**工具开发者只需实现一个 MCP Server，所有支持 MCP 的 LLM 应用都能直接使用。**

### 5.2 MCP 的架构

MCP 采用经典的客户端-服务器架构：

```
┌─────────────────┐     JSON-RPC      ┌─────────────────┐
│   MCP Client    │ ◄──────────────►  │   MCP Server    │
│  (LLM 应用端)    │    (协议通信)      │  (工具/数据源端)  │
└─────────────────┘                   └─────────────────┘
     │                                      │
     │ 嵌入在 LLM 应用中                    │ 独立运行的服务
     │ 如：Claude Desktop                  │ 如：文件系统、数据库、
     │     VS Code AI                      │      搜索引擎、API
     │     自定义 Agent                    │
```

**MCP Client（客户端）**：嵌入在 LLM 应用中，负责与 MCP Server 建立连接、发现可用工具、调用工具。一个 Client 可以同时连接多个 Server。

**MCP Server（服务器）**：提供具体工具和数据资源的后端服务。每个 Server 独立运行，可以有自己独立的生命周期。

**传输层（Transport）**：支持两种通信方式：
- **stdio**：通过标准输入/输出通信，适用于本地运行（Server 作为子进程启动）
- **SSE（Server-Sent Events）**：通过 HTTP 通信，适用于远程运行（Server 部署在另一台机器上）

### 5.3 MCP Server 提供的三种能力

MCP Server 可以暴露三类资源给 LLM 应用：

**1. Tools（工具）——可执行的函数**

Tool 是 MCP 中最核心的能力。它是一个可以被 LLM 调用的函数，有明确的输入和输出定义：

```json
{
  "name": "search_documents",
  "description": "搜索知识库中的文档",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "搜索关键词"
      },
      "limit": {
        "type": "integer",
        "description": "返回结果数量",
        "default": 10
      }
    }
  }
}
```

当 LLM 决定需要搜索文档时，它会构造一个 Tool Call，包含工具名称和参数，然后 MCP Client 将这个请求转发给 MCP Server 执行。

**2. Resources（资源）——可读取的数据**

Resource 是可以被 LLM 读取的静态或动态数据，类似于"文件"的概念。例如：
- 文件系统上的一个文件
- 数据库中的一条记录
- API 返回的 JSON 数据

与 Tool 不同，Resource 是"拉取"模式——LLM 主动读取资源中的数据，而不是调用一个函数。

**3. Prompts（提示模板）——预定义的 Prompt**

Server 可以提供预定义的 Prompt 模板，让 LLM 应用按照标准的格式与 Server 交互。这确保了交互的一致性和可预测性。

### 5.4 MCP vs Function Calling——本质区别

这是理解 MCP 的关键。两者不是竞争关系，而处于不同的抽象层次：

| 维度 | Function Calling | MCP |
|------|-----------------|-----|
| **层级** | API 参数格式 | 通信协议 |
| **耦合度** | 紧耦合：工具定义写在 API 请求中 | 松耦合：工具定义由 Server 独立提供 |
| **复用性** | 切换模型需重写工具定义 | 同一 Server 可被任意 MCP 客户端使用 |
| **动态发现** | 需预先知道所有工具 | 通过 `list_tools` 动态获取 |
| **生命周期** | 无状态，随请求结束 | Server 独立运行，可维护自己的状态 |
| **标准化** | 各厂商自定义格式 | 开放协议，社区标准 |
| **版本管理** | 随应用代码一起发布 | Server 可独立更新和版本管理 |

更直白的类比：

- **Function Calling** 相当于"每次点菜时手写菜单"——今天你告诉厨师要番茄炒蛋、明天你告诉厨师要宫保鸡丁，每次都要写一遍。
- **MCP** 相当于"餐厅有固定菜单本"——厨师把菜单写在本上，食客直接看菜单点菜，不需要每次都重新描述。

实际上，MCP 通常是构建在 Function Calling 之上的：MCP Client 从 Server 获取工具列表后，将这些工具以 Function Calling 的格式传递给 LLM，LLM 以 Function Calling 的方式调用工具，然后 Client 将调用转发给 Server 执行。

### 5.5 MCP vs A2A——纵向 vs 横向

MCP 和 Google 提出的 A2A（Agent-to-Agent）协议服务于不同的场景：

- **MCP（Model Context Protocol）**：Agent 与工具之间的通信协议。这是**纵向**关系——Agent 在上层，工具在下层。Agent 调用工具获取信息或执行操作。

- **A2A（Agent-to-Agent）**：Agent 与 Agent 之间的通信协议。这是**横向**关系——Agent 之间相互协作、委派任务。

两者不是竞争关系，而是互补关系。一个 Agent 系统可以同时使用 MCP 调用工具（获取信息、执行操作）和 A2A 与其他 Agent 通信（任务委派、结果汇总）。

### 5.6 MCP Server 的生命周期

一个完整的 MCP 交互过程包含四个阶段：

1. **初始化（Initialization）**：Client 连接 Server，交换协议版本和能力信息。双方确认使用相同的协议版本，Client 告知自己支持的特性，Server 告知自己的能力范围。

2. **发现（Discovery）**：Client 通过 `list_tools`、`list_resources`、`list_prompts` 等 RPC 调用，获取 Server 提供的完整能力列表。

3. **调用（Calling）**：Client 根据 LLM 的决策，通过 `call_tool`、`read_resource` 等方法使用 Server 的能力。每次调用是独立的，但 Server 可以维护跨调用的状态。

4. **关闭（Shutdown）**：连接断开，Server 清理资源（关闭文件句柄、释放内存等）。

### 5.7 MCP 的当前生态

截至 2026 年，MCP 的生态已经相当丰富：

- **官方 MCP Server**：文件系统、GitHub、Slack、Notion、数据库（PostgreSQL、SQLite）
- **社区 MCP Server**：搜索引擎（Google、Bing）、代码执行（Python、JavaScript）、云服务（AWS、GCP）
- **框架集成**：LangChain、Vercel AI SDK、LlamaIndex 均已支持 MCP

## 六、STAR 模板：Agent 面试中的结构化回答

### 6.1 什么是 STAR 模板？

STAR 是一种结构化的回答框架，最初用于行为面试（Behavioral Interview），后被广泛用于 Agent 和系统设计的技术面试中。STAR 是四个部分的缩写：

- **S（Situation）**：场景——项目/任务的背景是什么
- **T（Task）**：任务——你的具体目标和职责是什么
- **A（Action）**：行动——你做了什么、为什么这么做
- **R（Result）**：结果——最终取得了什么效果、学到了什么

### 6.2 为什么 STAR 在技术面试中有效？

对于 Agent 相关的技术问题，面试官不仅关心你是否了解某个概念，更关心你**在实际项目中是如何运用这些概念的**。STAR 模板提供了一个清晰的叙事结构，帮助面试官快速理解：

1. 你面对的实际问题是什么（S + T）
2. 你是如何分析问题、做出技术选型的（A）
3. 你的方案产生了什么实际效果（R）

### 6.3 使用 STAR 回答 Agent 问题的示例

**问题**："请描述一个你构建 RAG 系统的项目经历。"

**回答框架：**

**S（场景）：**
"在我们的客服系统中，用户提问的重复率高达 60% 以上，但每次都需要人工客服逐一回复。我们决定构建一个智能问答系统来自动回答这些常见问题。系统需要接入超过 5000 页的产品文档和 FAQ。"

**T（任务）：**
"我的任务是设计并实现一个 RAG 系统，要求：
- 响应时间在 3 秒以内（用户可接受范围）
- 答案准确率不低于 85%
- 能够处理中英文混合的查询
- 所有答案必须可溯源，附带文档引用"

**A（行动）：**
"我采取了以下技术方案：

首先，在文档处理阶段，我对比了固定分块和语义分块的效果。最终选择了 512 token、128 token overlap 的递归分块策略——因为我们的产品文档结构清晰（章节分明），递归分块能在章节边界自然切分，同时保持块大小的可控性。

在检索阶段，我采用了混合检索（BM25 + 向量检索 + RRF 融合）。原因是我们的文档包含大量专有名词（产品名、API 名），纯语义检索在这些精确匹配场景下效果不佳。BM25 解决精确匹配，向量检索解决语义匹配，RRF 通过排名融合避免了两套分数系统不统一的问题。

在生成阶段，我们设计了专门的 Prompt 模板，要求 LLM 在回答末尾标注引用的文档来源。这解决了答案可溯源的需求。

我选择了 LangGraph 来编排整个流程，而不是简单的 Chain——因为我们希望未来能够升级到 Agentic RAG，让系统能够自主判断是否需要多次检索。LangGraph 的图结构使得从单次检索升级到多步检索的改动成本很低。"

**R（结果）：**
"系统上线后：
- 平均响应时间 1.8 秒，满足 3 秒以内的要求
- 准确率达到 91%（在 2000 条测试集上评估）
- 人工客服的重复问题处理量减少了 55%
- 每次回答都附带引用来源，用户满意度提升 30%

这个项目的经验让我深刻理解到：RAG 系统的瓶颈不在 LLM，而在检索质量。检索策略的调优（分块大小、检索方式、融合算法）对最终效果的影响远大于 Prompt 工程的优化。"

### 6.4 在面试中使用 STAR 的要点

**选择有代表性的项目。** 不需要每个项目都讲——选一个最能体现你技术深度的项目。对于 RAG/Agent 相关面试，优先选择涉及多个技术点的项目（比如同时包含 RAG、Agent、LangGraph、MCP）。

**"A"的部分要详细展开。** STAR 中 Action 是最关键的部分。面试官想了解的是你解决问题时的思考过程——为什么选择了这个技术方案、有没有考虑过其他方案、权衡了什么因素。不要只说"我用了 RAG"，而要解释"为什么用 RAG 而不是微调"、"为什么选择这个分块策略"。

**"R"的部分要用数据支撑。** 尽量用量化的结果来证明你的方案有效。准确率提升、响应时间降低、成本节约——这些数据比"效果很好"有说服力得多。

**展示你踩过的坑。** 有经验的面试官更欣赏诚实的技术反思。"我们最初选择了 256 token 的分块大小，后来发现关键信息经常被切断，改为 512 后准确率提升了 8%"——这种具体的反思展示了你真正深入过项目。

## 七、概念关系总览

为了帮助理解本部分所有概念之间的关系，这里给出一个统一的关系图：

```
┌─────────────────────────────────────────────────────────┐
│                     RAG (Retrieval-Augmented Generation)│
│                                                         │
│  ┌──────────┐    ┌──────────┐    ┌──────────────────┐  │
│  │ 索引阶段  │───►│ 检索阶段  │───►│   生成阶段        │  │
│  │(分块+向量)│    │(混合检索) │    │(LLM + Prompt)    │  │
│  └──────────┘    └────┬─────┘    └────────┬─────────┘  │
│                       │                   │            │
│       需要多步检索时    │                   │            │
│                       ▼                   │            │
│              ┌─────────────────┐          │            │
│              │  Agentic RAG    │──────────┘            │
│              │  (多步推理+拆解) │                        │
│              └────────┬────────┘                        │
│                       │ 基于                              │
│                       ▼                                 │
│              ┌─────────────────┐                        │
│              │ Agent 框架       │                        │
│              │ ┌─ Brain (LLM)  │                        │
│              │ ├─ Perception   │                        │
│              │ ├─ Action(Tools)│─── 通过 MCP 协议 ──►   │
│              │ └─ Memory       │   统一调用外部工具       │
│              └────────┬────────┘                        │
│                       │ 通过                               │
│                       ▼                                 │
│              ┌─────────────────┐                        │
│              │  LangGraph       │                        │
│              │  (State/Node/    │                        │
│              │   Edge/Checkpoint)│                       │
│              └─────────────────┘                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

理解这张图的关键脉络：

1. **RAG 是基础架构**，解决了 LLM 的知识截止和幻觉问题
2. **混合检索** 提升了 RAG 的检索质量，通过 BM25 + 向量 + RRF 融合
3. **Agentic RAG** 解决了 RAG 的多步推理问题，让系统能够自主规划
4. **Agent** 提供了 AI 系统自主决策和执行的能力框架
5. **LangGraph** 是实现复杂 Agent 流程的工程工具，提供状态管理、条件路由、Checkpoint 等基础设施
6. **MCP** 是 Agent 调用外部工具的标准化协议，让工具生态互联互通

## 关键要诀速查

以下是对每个核心概念的"一句话理解"，帮助你在需要快速回顾时抓住重点：

| 概念 | 一句话理解 |
|------|-----------|
| RAG | 先查文档再回答，让 LLM 基于证据说话，减少幻觉 |
| Chunking | 文档切块的艺术——太小信息不完整，太大噪声过多 |
| 混合检索 | BM25 管精确匹配 + 向量检索管语义匹配，RRF 用排名融合两者 |
| Agent | 给 LLM 装上"大脑+眼睛+手+记忆"，让它能自主完成任务 |
| Chain vs Agent | Chain 是写死的流水线，Agent 是自己会思考的机器人 |
| ReAct | 思考→行动→观察的循环，让 Agent 边做边想边调整 |
| LangGraph State | 用 TypedDict 定义贯穿图的"万能背包"，Reducer 决定合并规则 |
| LangGraph 条件边 | 让 Agent 根据当前状态"自己决定下一步去哪" |
| Checkpoint | 每一步都自动存档，崩溃不怕、调试轻松、还能让人工介入 |
| Human-in-the-Loop | 关键步骤让人审批，AI 负责干活但不背锅 |
| Agentic RAG | RAG 学会拆解问题，从单次检索升级为多步自主研究 |
| MCP | 工具的 USB-C 接口，写一次 Server 处处可用 |
| STAR | 用故事结构回答技术问题：场景→任务→行动→结果 |
