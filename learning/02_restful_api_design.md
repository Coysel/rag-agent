# 第二部分：RESTful API 设计原则

## 一、REST 架构风格概述

### 1.1 什么是 REST

REST（Representational State Transfer，表述性状态转移）是 Roy Fielding 在 2000 年他的博士论文中提出的一种**软件架构风格**。它不是一套标准，也不是一个具体的协议，而是一组**架构约束条件**。如果一个架构符合这些约束，我们就称它为 RESTful 的。

理解 REST 的关键在于它的名字本身——"表述性状态转移"。我们将这个词拆开来看：

- **资源（Resource）**：一个可以被命名的信息实体，比如一篇博客文章、一个用户、一张订单。每个资源都有一个唯一的 URI（统一资源标识符）。
- **表述（Representation）**：资源在某个时刻的呈现形式。同一个资源可以有多种表述——JSON 格式、XML 格式、HTML 格式等。客户端看到的不是资源本身，而是资源的"表述"。
- **状态转移（State Transfer）**：客户端通过操作资源的表述来改变应用的状态。每一次 HTTP 请求，本质上都是从服务器获取或修改资源的表述，从而推动应用状态的变化。

**为什么需要 REST？** 在 REST 出现之前，Web 服务主要使用 SOAP（Simple Object Access Protocol）等协议。SOAP 是一个重量级的协议，它将自己的语义封装在 XML 消息体中，通过 HTTP POST 传输。这种方式有几个问题：第一，SOAP 操作通常只使用 POST 方法，把增删改查都藏在请求体里，导致 HTTP 本身的语义没有被充分利用；第二，SOAP 的 WSDL 定义复杂，调试和开发效率低；第三，SOAP 与 HTTP 的缓存机制不兼容。REST 正是为了解决这些问题而提出的——它充分利用 HTTP 协议本身的能力（方法、状态码、缓存头等），让 API 更简洁、更可预测、更易于扩展。

### 1.2 REST 的六个约束

REST 风格的核心是以下六个约束条件。注意，这些约束是一个整体——违反其中一个，理论上就不能称为真正的 RESTful API。但在实际工程中，我们常常看到"部分 RESTful"的设计，即只遵守了其中一部分约束。

#### 约束一：客户端-服务器（Client-Server）

**定义**：客户端和服务器分离，各司其职。客户端负责用户界面和用户体验，服务器负责数据存储和业务逻辑。

**为什么这样做？** 分离关注点，让两端可以独立演进。服务器不需要关心用户是在用 Web 浏览器、移动 App 还是命令行工具访问 API；客户端也不需要关心数据是存在 MySQL 还是 Redis 里。这种松耦合是现代分布式系统的基础。

#### 约束二：无状态（Stateless）

**定义**：从客户端到服务器的每个请求，都必须包含服务器理解该请求所需的全部信息。服务器不保存任何客户端上下文。

**为什么这样做？** 
- **可伸缩性**：无状态意味着任何服务器都可以处理任何请求，请求可以轻松地分发到任意服务器节点，水平扩展变得简单。
- **可靠性**：服务器崩溃后，请求可以简单地重发到另一台服务器，不需要恢复会话状态。
- **可见性**：监控系统只需看单个请求就能理解其意图，不需要追踪会话历史。

**但这是否意味着登录状态无法实现？** 不是。用户的身份凭证（如 JWT Token）由客户端保存，每次请求时通过 Authorization 头带过去。服务器验证 Token，但不保存"用户 123 已登录"这样的会话状态。这叫做"客户端管理状态"，不是"服务器无状态"的例外。

**为什么必须无状态？** 对于一个可能有成百上千台服务器的分布式系统来说，如果每台服务器都要维护会话状态，那么：
1. 需要使用分布式会话同步方案（如 Redis Session），增加系统复杂度。
2. 同一用户的请求必须由同一台服务器处理（粘性会话），负载均衡策略受限。
3. 某台服务器宕机，该服务器上的所有会话丢失。

#### 约束三：缓存（Cacheable）

**定义**：响应必须隐式或显式地标记为可缓存或不可缓存。

**为什么这样做？** 减少网络传输，降低延迟，减轻服务器负载。如果客户端或中间代理知道一个响应是可缓存的，就可以在过期之前重复使用它，避免重复请求。

**实现方式**：通过 HTTP 缓存头（Cache-Control、Expires、ETag 等）来控制。例如 `Cache-Control: max-age=3600` 表示这个响应在 3600 秒内有效，客户端和中间代理可以缓存它。

#### 约束四：统一接口（Uniform Interface）

**定义**：REST 的核心特征，要求使用统一的接口来操作资源。这是 REST 区别于其他架构风格最显著的特征。

统一接口包含四个子约束：
1. **资源的标识（Identification of Resources）**：每个资源有唯一的 URI。
2. **通过表述操作资源（Manipulation of Resources through Representations）**：客户端通过操作资源的表述（如 JSON）来修改资源状态。
3. **自描述消息（Self-descriptive Messages）**：每个消息包含足够的元数据让接收方理解如何处理它——Content-Type 告诉如何解析，Cache-Control 告诉是否可以缓存等。
4. **超媒体作为应用状态的引擎（HATEOAS - Hypermedia As The Engine Of Application State）**：这是最常被忽略的约束。它要求 API 响应中包含链接，客户端通过这些链接发现可执行的操作。

**为什么统一接口重要？** 它解耦了客户端和服务器的实现。只要接口约定不变，服务端的实现可以完全重构，客户端也可以自由切换技术栈。

#### 约束五：分层系统（Layered System）

**定义**：架构由多个层次组成，每个层次只知道与自己直接交互的相邻层，不知道更远的层。

**为什么这样做？** 引入中间层（负载均衡器、反向代理、API 网关、缓存层）而不影响客户端和服务器。例如，客户端访问 `api.example.com`，实际请求可能经过 CDN、负载均衡器、Nginx 反向代理，最后才到达应用服务器——但对客户端来说，它只知道自己在与 `api.example.com` 通信。

#### 约束六：按需代码（Code on Demand）（可选）

**定义**：服务器可以临时将可执行代码（如 JavaScript 脚本）传输到客户端执行。这是唯一可选的约束。

**为什么可选？** 因为它会降低可见性，而且不是所有客户端都能执行下载的代码。实际 API 设计中很少使用这个约束。

### 1.3 为什么 REST 适用于 Web API

REST 的约束组合起来产生了一个关键特性：**可预测性**。当一个开发者看到 `GET /users/123`，他不需要查阅文档就知道这是在做什么——获取用户 123 的信息。这种约定大于配置的思想大大降低了 API 的学习成本和集成成本。

相比之下，RPC 风格的 API（如 gRPC 或早期的 XML-RPC）通常是这样：`POST /api/getUserById`，方法名和请求结构完全由服务定义，缺乏统一的语义约定。

## 二、HTTP 方法与幂等性

### 2.1 核心方法详解

RESTful API 使用 HTTP 方法（也称为 HTTP 动词）来表达对资源的操作意图。这些方法不是 REST 发明的，而是 HTTP 协议本身定义的。REST 所做的是将这些方法映射到 CRUD（Create、Read、Update、Delete）操作上。

#### GET：获取资源

```http
GET /api/users/123 HTTP/1.1
Host: example.com
```

- **语义**：获取指定资源的表述。
- **幂等**：是。无论请求多少次，资源在服务器的状态不变，返回结果理论上相同。
- **安全性**：是。GET 是**安全方法**（Safe Method），表示它不应引起服务器端的状态变化。
- **请求体**：GET 请求不应该发送请求体。虽然 HTTP 规范不禁止，但实际上很多服务器和代理会忽略或拒绝 GET 请求体。
- **常见用途**：获取资源列表（`GET /users`）、获取单个资源（`GET /users/123`）、搜索（`GET /users?name=zhang`）。

**为什么 GET 必须是安全的和幂等的？** 这关系到网络基础设施的假设。浏览器、CDN、反向代理、爬虫都会自动重试或预取 GET 请求。如果 GET 不幂等，自动重试可能导致重复扣款、重复创建订单等灾难性后果。

#### POST：创建资源

```http
POST /api/users HTTP/1.1
Content-Type: application/json

{
  "name": "张三",
  "email": "zhangsan@example.com"
}
```

- **语义**：向指定资源集合提交一个新的资源，由服务器决定新资源的 URI。
- **幂等**：否。重复发送相同的 POST 请求，会创建多个资源（尽管资源内容可以完全相同，但它们的 URI 会不同）。
- **安全性**：否。POST 会修改服务器状态。
- **请求体**：包含要创建的资源的表述。
- **常见用途**：创建新资源（`POST /users`）。

**为什么 POST 不是幂等的？** POST 的本质是"请把这个实体作为我的资源集合的新成员处理"。每次请求都是一个新的"提交"动作，服务器应创建一个新资源。想象一个场景：用户提交订单，网络超时了——他不知道订单是否已创建，于是再次点击提交。如果 POST 是幂等的，第二次提交不会创建新订单；但 POST 不幂等，所以第二次提交会创建重复订单。这就是为什么在实际系统中，需要额外的**幂等性方案**（后面会详述）来防止重复提交。

#### PUT：整体更新/替换资源

```http
PUT /api/users/123 HTTP/1.1
Content-Type: application/json

{
  "name": "张三",
  "email": "zhangsan@new.com"
}
```

- **语义**：用请求体**完整替换** URI 指定的资源。如果资源不存在，有些实现会创建它（upsert 行为）。
- **幂等**：是。无论发送多少次相同的 PUT 请求，结果都一样——资源的状态是确定的。
- **安全性**：否。PUT 会修改服务器状态。
- **请求体**：包含资源的**完整**表述。客户端必须提供所有字段，缺失的字段应被重置为默认值或 null。
- **常见用途**：完整更新已有资源（`PUT /users/123`）。

**PUT 的幂等性意味着什么？** 假设客户端想要更新用户的邮箱，发送了 `PUT /users/123`，但网络包在中间延迟了。客户端不确定请求是否到达，于是重新发送相同请求。第二次请求到达时，服务器把资源替换成和第一次请求完全相同的内容——结果不变。所以重复发送是安全的。

**但这里有一个微妙之处**：如果另一个客户端在这两次请求之间修改了同一个资源，第二次 PUT 会覆盖掉那个修改。PUT 的幂等性是针对"相同请求"的，不是针对"并发安全"的。这就是为什么我们需要**乐观锁**（后面会讲）。

#### PATCH：部分更新资源

```http
PATCH /api/users/123 HTTP/1.1
Content-Type: application/json

{
  "email": "zhangsan@new.com"
}
```

- **语义**：对资源的某些字段进行**部分修改**。
- **幂等**：取决于实现。应用 JSON Merge Patch 标准（如上例）时通常是幂等的；但如果使用 JSON Patch 格式（如 `[{ "op": "add", ... }]`）,是否幂等取决于具体操作。
- **安全性**：否。PATCH 会修改服务器状态。
- **请求体**：只包含要修改的字段及其新值，不需要发送整个资源。
- **常见用途**：更新单个字段（`PATCH /users/123` 只改邮箱）。

**PUT 和 PATCH 的根本区别是什么？** 这是面试和实际工作中最常见的混淆点。

它们的核心区别在于**替换 vs. 修改**：

- **PUT 是幂等的替换**。你把整个资源放在请求体里，说"把这个 ID 的资源彻底换成这个"。缺失的字段会被重置。
  ```
  // 假设用户原有数据：{ name: "张三", email: "old@test.com", age: 25 }
  PUT /users/123  body: { name: "张三", email: "new@test.com" }
  // 结果：{ name: "张三", email: "new@test.com", age: null } ← age 被重置了！
  ```

- **PATCH 是部分更新**。你只告诉服务器想改哪些字段。
  ```
  // 假设用户原有数据：{ name: "张三", email: "old@test.com", age: 25 }
  PATCH /users/123  body: { email: "new@test.com" }
  // 结果：{ name: "张三", email: "new@test.com", age: 25 } ← 其他字段保留
  ```

**实际开发中应该用 PUT 还是 PATCH？** 对于大对象，每次都传完整数据会增加网络开销，所以很多 API 实际上用 PATCH 替代 PUT。但从规范角度：如果你接收完整资源替换，用 PUT；如果只接收部分更新字段，用 PATCH。一个常见的模式是：**创建用 POST，完整更新用 PUT，部分更新用 PATCH**。

#### DELETE：删除资源

```http
DELETE /api/users/123 HTTP/1.1
```

- **语义**：删除指定 URI 的资源。
- **幂等**：是。第一次删除后资源已不存在，第二次删除同样返回成功（通常是 204 No Content 或 404 Not Found）。从服务器的角度看，资源的状态已经"删除"或"不存在"，这个状态不会因为重复 DELETE 而改变。
- **安全性**：否。DELETE 会修改服务器状态。
- **请求体**：通常不需要请求体。
- **常见用途**：删除资源（`DELETE /users/123`）。

### 2.2 HTTP 方法速查表

| 方法 | CRUD | 幂等 | 安全 | 请求体 | 响应体 |
|------|------|------|------|--------|--------|
| GET | 读取 | 是 | 是 | 无 | 资源表述 |
| POST | 创建 | 否 | 否 | 新资源数据 | 新资源的 URI 和表述 |
| PUT | 整体更新 | 是 | 否 | 完整资源表述 | 更新后的资源表述 |
| PATCH | 部分更新 | 视情况 | 否 | 部分字段 | 更新后的资源表述 |
| DELETE | 删除 | 是 | 否 | 无 | 通常为空(204) |

### 2.3 核心概念深入：幂等性（Idempotency）

**定义**：一个操作是幂等的，当且仅当执行一次和执行多次的效果相同（从服务器的资源状态角度）。

**数学表达**：对于函数 f，如果 f(f(x)) = f(x)，则 f 是幂等的。在实际的 HTTP 上下文中，这意味著客户端可以安全地重试请求，而不用担心副作用。

**为什么幂等性如此重要？**

互联网是不可靠的。TCP 虽然保证了数据的可靠传输，但应用层的超时、网络断开、服务器崩溃等情况都可能导致客户端不确定请求是否被处理。这时候客户端需要重试。

- **如果操作是幂等的**：客户端可以简单地重试，没有任何顾虑。
- **如果操作不是幂等的**：客户端需要复杂的逻辑来判断是否已经执行过，或者需要自己实现幂等性方案。

**幂等与安全的区别**：

- **安全方法**：不会修改服务器状态。GET、HEAD、OPTIONS 是安全的。
- **幂等方法**：多次执行效果相同。GET 既安全又幂等；PUT 和 DELETE 不安全但幂等；POST 既不安全也不幂等。

安全方法的集合是幂等方法的一个子集。

### 2.4 四个幂等性方案

前面提到 POST（创建资源）不是幂等的，但在实际业务中，我们经常需要防止重复提交。比如用户提交订单、转账等操作，绝对不能因为网络重试而重复执行。以下是四种常见的幂等性实现方案：

#### 方案一：唯一约束（Database Unique Constraint）

**原理**：在数据库层面设置唯一索引，确保只能插入一次。

**实现**：
```sql
-- 创建订单表时，为业务唯一键设置唯一约束
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no VARCHAR(32) NOT NULL UNIQUE,  -- 业务订单号唯一
    user_id BIGINT NOT NULL,
    amount DECIMAL(10,2),
    status VARCHAR(16)
);
```

**为什么能保证幂等？** 如果客户端因为超时重试，第二次插入相同的 `order_no`，数据库会抛出唯一约束冲突异常。服务端捕获这个异常后，返回"订单已存在"而不是创建新订单。

**使用场景**：适用于业务本身有天然唯一标识的场景，如订单号、支付流水号、交易流水号。

**优缺点**：实现简单，完全不依赖外部系统。但需要在业务层先生成唯一 ID，不能用于任意请求的幂等。

#### 方案二：去重表（Deduplication Table）

**原理**：创建一个专门存储已处理请求 ID 的去重表，在业务操作前先检查。

**实现**：
```sql
-- 去重表
CREATE TABLE idempotent_records (
    idempotent_key VARCHAR(64) PRIMARY KEY,  -- 幂等键
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    expired_at TIMESTAMP
);

-- 业务方法伪代码
def create_order(request):
    # 1. 检查去重表
    if exists_in_idempotent_table(request.idempotent_key):
        return "已处理，请勿重复提交"
    
    # 2. 执行业务（创建订单）
    order = order_repo.create(request)
    
    # 3. 写入去重表
    idempotent_repo.save(request.idempotent_key)
    
    return order
```

**为什么能保证幂等？** 去重表的 `idempotent_key` 有唯一约束，多个相同的请求只有第一个能成功插入。但注意：第 2 步和第 3 步不是原子的。如果第 2 步成功，第 3 步失败（服务器宕机），客户端重试时会发现去重表中没有记录，导致重复创建。**解决方案**：将去重表的插入和业务操作放在同一个数据库事务中，确保原子性。

**使用场景**：通用的幂等性方案，适用于任何 POST 请求。

#### 方案三：Token 机制（Pre-generated Token）

**原理**：服务器预先生成一个唯一的 Token，客户端需要先申请 Token，然后在提交请求时携带这个 Token，服务器检查 Token 是否已被使用。

**实现**：
```
1. 客户端: GET /api/token  → 服务器返回唯一 Token
2. 客户端: POST /api/orders (token: "abc123", ...)  → 服务器检查 token 是否有效
3. 如果 token 有效，处理请求并使 token 失效
4. 如果 token 无效（已使用或已过期），拒绝请求
```

**为什么能保证幂等？** Token 是一次性的。用完即废，不可能用同一个 Token 提交两次有效请求。

**使用场景**：用在需要用户交互的场景，如表单提交、下单流程。"提交按钮防抖"的典型实现——用户每次打开表单，先获取一个 Token；提交时带上 Token，即使多次点击，只有第一个请求能成功。

**为什么比去重表更好？** Token 机制在服务器端不需要查询历史记录，只需要用 Redis 检查 key 是否存在（复杂度 O(1)）。而且 Token 可以设置过期时间，自动清理。

#### 方案四：分布式锁（Distributed Lock）

**原理**：使用 Redis 或 ZooKeeper 等分布式锁，以业务唯一 ID 作为锁的 key，确保同一个业务逻辑同一时间只有一个请求在处理。

**实现**（使用 Redis）：
```python
def create_order_with_lock(user_id, order_data):
    lock_key = f"lock:order:{user_id}"
    # 尝试获取锁，ttl 为 5 秒
    if redis.setnx(lock_key, "locked"):
        redis.expire(lock_key, 5)
        try:
            # 执行业务逻辑
            return create_order(order_data)
        finally:
            redis.delete(lock_key)
    else:
        return "请求正在处理中，请勿重复提交"
```

**为什么能保证幂等？** 锁确保同一时间只有一个请求能进入关键业务逻辑。如果请求重试，会因为锁未释放而被拒绝。

**为什么这还不够？** 锁解决的是"并发冲突"问题，不是"重复提交"问题。如果第一个请求执行完毕并释放了锁，但客户端还没收到响应就重试了——此时锁已释放，重试请求可以正常获取锁并再次执行。所以分布式锁通常需要配合其他方案（如去重表）一起使用。

#### 四种方案对比

| 方案 | 复杂度 | 性能 | 可靠性 | 适用场景 |
|------|--------|------|--------|----------|
| 唯一约束 | 低 | 高 | 高 | 有天然唯一键的业务 |
| 去重表 | 中 | 中 | 中（需事务保证） | 通用幂等 |
| Token 机制 | 中 | 高 | 高 | 用户交互场景 |
| 分布式锁 | 高 | 中 | 中 | 防止并发冲突 |

**工程实践中的推荐组合**：

```
Token 机制 + 唯一约束 = 双重保障
```

前端使用 Token 防止用户重复点击，后端数据库唯一约束作为最后防线。这样即使 Token 校验因 Bug 失效，数据库也会拦截重复数据。

## 三、HTTP 状态码

### 3.1 状态码分类

HTTP 状态码由三位数字组成，第一位数字表示类别：

| 类别 | 范围 | 含义 |
|------|------|------|
| 1xx | 100-199 | 信息性响应——请求正在处理 |
| 2xx | 200-299 | 成功——请求已成功处理 |
| 3xx | 300-399 | 重定向——需要进一步操作 |
| 4xx | 400-499 | 客户端错误——请求有问题 |
| 5xx | 500-599 | 服务器错误——服务器无法处理有效请求 |

### 3.2 RESTful API 常用状态码

#### 2xx 成功

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| 200 OK | 请求成功 | GET 成功返回资源；PUT 或 PATCH 成功更新资源 |
| 201 Created | 资源创建成功 | POST 成功创建新资源后返回。响应头 Location 应包含新资源的 URI |
| 204 No Content | 请求成功，但无返回内容 | DELETE 成功删除资源；PUT 成功后不需要返回资源内容时 |

**为什么需要区分 200 和 201？** 客户端可以根据状态码采取不同的后续操作。收到 201 时，客户端知道一个新资源被创建了，可以从 Location 头获取新资源的 URI，或者自动跳转到新资源页面。收到 200 时，客户端知道只是正常返回结果。

同样，**204 的一个重要应用场景**：DELETE 成功后，服务器不需要返回被删除的资源内容，返回 204 且无响应体，节省带宽。客户端收到 204 就知道"资源已删除"。

#### 3xx 重定向

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| 301 Moved Permanently | 资源已永久移动 | 旧域名跳转到新域名 |
| 302 Found | 资源临时移动 | 临时跳转到维护页面 |
| 304 Not Modified | 资源未修改 | 结合 If-Modified-Since 头用于缓存验证 |

**为什么 REST API 需要 304？** 如果客户端缓存了资源，用 `If-Modified-Since` 或 `If-None-Match` 头发起条件请求，服务器如果判断资源未修改，返回 304 和空响应体。客户端就知道缓存仍然有效，不需要重新下载。这对于节省带宽和提升响应速度非常重要。

#### 4xx 客户端错误

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| 400 Bad Request | 请求格式错误 | 请求体不是合法的 JSON；缺少必要字段 |
| 401 Unauthorized | 未认证 | 请求未提供认证凭据或凭据无效 |
| 403 Forbidden | 无权限 | 请求已认证，但无权访问该资源 |
| 404 Not Found | 资源不存在 | 请求的 URI 对应的资源不存在 |
| 405 Method Not Allowed | 方法不允许 | 对只读资源使用了 DELETE 方法 |
| 409 Conflict | 资源冲突 | 乐观锁版本号不匹配；资源已存在 |
| 422 Unprocessable Entity | 请求语义错误 | 请求体格式正确但业务验证失败（如邮箱格式不对） |
| 429 Too Many Requests | 请求过频 | 触发了限流 |

#### 5xx 服务器错误

| 状态码 | 含义 | 使用场景 |
|--------|------|----------|
| 500 Internal Server Error | 服务器内部错误 | 未捕获的异常，数据库连接失败 |
| 502 Bad Gateway | 网关错误 | Nginx 转发请求到后端，后端无响应 |
| 503 Service Unavailable | 服务不可用 | 服务器过载或正在维护 |
| 504 Gateway Timeout | 网关超时 | 后端处理超时，Nginx 返回 504 |

### 3.3 深度辨析：401 Unauthorized vs 403 Forbidden

这是面试和实际工作中最容易混淆的一对状态码。很多初学者甚至资深开发者都会搞混它们的确切含义。

**401 Unauthorized：未认证（Unauthenticated）**

意思是"我不知道你是谁"。请求中没有提供合法的身份凭证。

**典型场景**：
```
客户端请求: GET /api/users/123
请求头: （未携带 Authorization）
服务器响应: 401 Unauthorized
响应体: {"error": "请登录后再试"}
```

**如何修复**：客户端需要提供身份凭证——通常是在请求头中加入 `Authorization: Bearer <token>`，或者使用 Cookie 进行身份验证。

**值得注意的是**：这个状态码的命名其实有些误导。"Unauthorized"字面意思是"未授权"，但它的实际含义是"未认证"。这是 HTTP 规范的历史遗留问题。

**403 Forbidden：无权限（Unauthorized Access）**

意思是"我认识你，但你没有权限做这件事"。客户端已经通过了身份验证，服务器知道请求方的身份，但该身份没有访问所请求资源的权限。

**典型场景**：
```
客户端请求: DELETE /api/users/123
请求头: Authorization: Bearer <token_of_user_456>
服务器响应: 403 Forbidden
响应体: {"error": "你只能删除自己的账户"}
```

**如何修复**：让用户 456 登录用户 123 的账户（很少可行），或者联系管理员获取权限。

**一个更容易理解的类比**：

- **401** = 安检处，你没有出示身份证，保安拦住了你。
- **403** = 你已经刷了身份证进了大楼，但走到了写着"非授权人员禁止入内"的门前。

**为什么需要区分这两个状态？**

从客户端开发者的角度看，这两种错误需要的处理方式完全不同：

| 维度 | 401 | 403 |
|------|-----|-----|
| 根本问题 | 缺少身份 | 缺少权限 |
| 客户端的应对 | 提示用户登录 | 提示用户无权操作 |
| 是否可以重试 | 带上正确凭据后可重试 | 重试通常无意义 |
| 前端常见处理 | 跳转到登录页 | 显示"权限不足"提示 |

**实际案例**：

```python
from flask import request, jsonify, abort

@app.before_request
def authenticate():
    """所有请求先过认证"""
    token = request.headers.get('Authorization')
    if not token:
        abort(401, description="缺少认证令牌")
    
    current_user = verify_token(token)
    if not current_user:
        abort(401, description="令牌无效或已过期")
    
    request.current_user = current_user

@app.route('/api/admin/users', methods=['DELETE'])
def delete_user(user_id):
    """只有管理员才能删除用户"""
    if not request.current_user.is_admin:
        abort(403, description="需要管理员权限")
    
    delete_user_from_db(user_id)
    return '', 204
```

在这个例子中：
1. 如果请求不带 Token → 401（身份问题）
2. 如果 Token 无效 → 401（身份问题）
3. 如果普通用户试图删除他人 → 403（权限问题）

## 四、API 版本管理

### 4.1 为什么需要版本管理？

API 一旦发布，就会被多个客户端（Web、iOS、Android、第三方开发者）使用。不同的客户端升级节奏不同——App 用户可能几个月都不升级，而 Web 用户每次刷新都是最新版本。当你对 API 进行不兼容的修改（如更改字段名、删除字段、修改响应格式）时，旧的客户端会立刻崩溃。

**版本管理的目的**：让 API 可以持续演进，同时保证旧客户端不受影响。不同的版本可以共存，不同的客户端使用不同的版本。

### 4.2 常见的版本化方案

#### 方案一：URI 路径版本化（最常用）

```http
GET /api/v1/users/123
GET /api/v2/users/123
```

**特点**：版本号直接写在 URL 路径中，一目了然，简单直接。

**优点**：
- 非常显眼，浏览器和调试工具中一看就知道版本
- 缓存键天然包含版本号，不同版本的缓存不会互相干扰
- 实现简单，可以在路由层统一处理

**缺点**：
- URI 不再是纯粹的资源标识符——同一个资源（用户 123）有多个 URI（v1 版本和 v2 版本）
- 随着版本增多，URL 层级变深

**最佳实践**：在 Nginx 或 API 网关层做版本路由，将不同版本的请求转发到不同的服务实例或代码分支：
```nginx
# Nginx 配置示例
location /api/v1/ {
    proxy_pass http://backend-v1/;
}
location /api/v2/ {
    proxy_pass http://backend-v2/;
}
```

#### 方案二：请求头版本化

```http
GET /api/users/123
Accept: application/vnd.myapp.v2+json
```

**特点**：版本号放在 HTTP 请求头的 `Accept` 字段中，使用自定义的 media type。

**优点**：
- URI 保持干净，同一个 URI 对应同一个资源
- 符合 REST 的"自描述消息"原则——版本信息是消息元数据的一部分

**缺点**：
- 调试不方便，浏览器直接访问无法指定版本头
- 缓存键需要包含 Accept 头，配置更复杂
- 对客户端开发者的要求更高——他们需要知道要设置请求头

#### 方案三：查询参数版本化

```http
GET /api/users/123?version=2
```

**特点**：版本号作为查询参数。

**优点**：实现最简单，只需在代码中读取查询参数。
**缺点**：URL 变得杂乱；查询参数容易被缓存系统忽略，导致缓存混乱；不标准的做法，不推荐。

### 4.3 版本策略建议

| 方案 | 推荐程度 | 使用场景 |
|------|----------|----------|
| URI 路径（/v1/） | 最推荐 | 对外 API、公共 API |
| 请求头版本化 | 较推荐 | 内部微服务、对版本管理有严格要求的场景 |
| 查询参数 | 不推荐 | 几乎没有合适的场景 |

**工程实践建议**：

1. **不要轻易产生多个版本**。版本是成本——每个版本都意味着额外的测试和维护工作。优先考虑**兼容性扩展**（在响应中增加字段而不是修改或删除字段，使用可选字段而不是必选字段）。
2. **给版本设置生命周期**。定好时间表：v1 发布 → v2 发布 → v1 标记为废弃（Deprecated）→ 提前通知客户端迁移 → v1 下架。比如 v1 在 v2 发布后 6 个月下架。
3. **版本与 API 网关配合**。在 API 网关层做版本路由，后端可以运行多个版本的服务实例。

## 五、资源命名规范

### 5.1 核心原则

RESTful API 围绕"资源"展开，因此资源命名是 API 设计的基石。好的命名让 API 自解释——开发者不需要看文档就能猜出 URL 的含义。

#### 原则一：使用名词，不使用动词

```
✅ 正确: GET /users, POST /orders, DELETE /products/123
❌ 错误: GET /getUsers, POST /createOrder, DELETE /deleteProduct
```

**为什么不能用动词？** REST 使用 HTTP 方法来表达操作——GET 本身就表示"获取"，再用 `getUsers` 就是冗余的。用动词意味着你创造了无限新的 URL（每个动词一个），打破了统一接口的约束。

**但是有些操作无法用标准的 CRUD 表达怎么办？** 比如"发布文章"、"审核订单"、"重置密码"。这时候有三种常见处理方式：
1. 用 POST 表达非 CRUD 操作：`POST /articles/123/publish`（见下面的原则三）
2. 将操作视为资源的一个属性：`PATCH /articles/123 { "status": "published" }`
3. 将操作视为一个子资源：`POST /articles/123/publish-requests`（创建一个发布请求）

#### 原则二：使用复数形式

```
✅ 正确: GET /users, GET /users/123, POST /orders
❌ 错误: GET /user, GET /user/123, POST /order
```

**为什么用复数？** 一致性。集合通常用复数（得到一堆用户），单个资源用集合下的 ID（得到一个用户）。如果集合用单数，`GET /user` 看起来像是获取"用户"这个抽象概念，而不是用户列表。

#### 原则三：用子资源表达层级关系

```
GET /users/123/orders          # 用户 123 的所有订单
GET /users/123/orders/456      # 用户 123 的订单 456
GET /orders/456/items          # 订单 456 的所有商品项
```

**层级不要太深**。超过三层的嵌套会让 URL 变得臃肿且难以维护：
```
GET /users/123/orders/456/items/789/comments   ← 不推荐
```
对于过深的层级，可以考虑用查询参数简化：`GET /comments?item_id=789`。

**什么时候适合用子资源？** 当 B 的生命周期完全依赖于 A 时——订单属于用户，订单项属于订单。当 B 可以脱离 A 独立存在时（如评论可以独立查看文章，也可以从评论列表直接访问），用查询参数更合适。

#### 原则四：使用下划线或连字符连接多词

```
✅ 正确: GET /order-items, GET /blog-posts
❌ 错误: GET /orderItems, GET /blogPosts, GET /order_items（如果团队约定用连字符）
```

**选择标准**：团队统一。在实践中，大多数 API 选择**连字符（-）**，因为它是 URL 中各部分的分隔符，而下划线在某些字体中可能被误读为空格。但也有团队用下划线。**关键是统一**。

#### 原则五：URL 中不包含文件扩展名

```
✅ 正确: GET /users/123 (Accept: application/json)
❌ 错误: GET /users/123.json, GET /users/123.xml
```

**为什么不需要扩展名？** REST 通过 `Accept` 请求头（内容协商）来决定返回格式，而不是靠 URL 后缀。URL 只标识资源，资源的表述形式通过 HTTP 头协商。

### 5.2 操作映射速查表

| HTTP 方法 | URL 模式 | 描述 |
|-----------|----------|------|
| GET | /users | 获取用户列表 |
| GET | /users/123 | 获取单个用户 |
| POST | /users | 创建新用户 |
| PUT | /users/123 | 完整更新用户 |
| PATCH | /users/123 | 部分更新用户 |
| DELETE | /users/123 | 删除用户 |
| GET | /users/123/orders | 获取用户的订单列表 |
| POST | /users/123/orders | 为用户创建订单 |

### 5.3 非 CRUD 操作的命名

不是所有业务操作都能用标准 CRUD 表达。以下是常见非 CRUD 操作的命名策略：

| 操作 | 推荐方式 | 说明 |
|------|----------|------|
| 用户登录 | POST /auth/login | 认证是独立的概念，不属于用户资源 |
| 用户注册 | POST /auth/register | 同上 |
| 激活账户 | POST /users/123/activate | 视作对资源的操作 |
| 重置密码 | POST /password-reset | 单独的"密码重置"资源 |
| 搜索 | GET /search?q=xxx | 搜索是一个独立资源 |
| 批量操作 | POST /users/batch | 创建专门的批量端点 |

## 六、统一响应格式

### 6.1 为什么需要统一响应格式？

没有统一响应格式的 API 就像一本没有目录的书——每次都要猜测响应结构。

想象一个没有统一格式的 API：有的接口出错时返回 `{ "error": "xxx" }`，有的返回 `{ "message": "xxx" }`，有的返回 `{ "code": 400, "msg": "xxx" }`。客户端需要为每个接口编写不同的错误处理代码，混乱且容易出错。

**统一响应格式的目标**：让所有 API 响应遵循相同的数据结构，客户端可以用相同的代码解析所有响应。

### 6.2 推荐的统一响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 123,
    "name": "张三",
    "email": "zhangsan@example.com"
  },
  "timestamp": 1700000000123,
  "request_id": "req-abc-123-def"
}
```

| 字段 | 必须 | 类型 | 说明 |
|------|------|------|------|
| `code` | 是 | int | 自定义业务状态码，非 HTTP 状态码 |
| `message` | 是 | string | 提示信息，成功时为 "success"，失败时为错误描述 |
| `data` | 否 | object/array | 实际数据，成功时必须，失败时通常为 null |
| `timestamp` | 推荐 | long | 服务器处理完成的时间戳（毫秒） |
| `request_id` | 推荐 | string | 本次请求的唯一 ID，用于日志追踪和问题排查 |

**错误时的响应**：

```json
{
  "code": 40001,
  "message": "参数验证失败：邮箱格式不正确",
  "data": null,
  "timestamp": 1700000000123,
  "request_id": "req-abc-123-def"
}
```

**分页列表的响应**：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "items": [
      { "id": 1, "name": "用户1" },
      { "id": 2, "name": "用户2" }
    ],
    "pagination": {
      "page": 1,
      "page_size": 20,
      "total": 56,
      "total_pages": 3
    }
  },
  "timestamp": 1700000000123,
  "request_id": "req-abc-123-def"
}
```

**为什么不直接用 HTTP 状态码而要用自定义 code？**

HTTP 状态码粒度有限——一个 400 Bad Request 可能因为"参数缺失"、"参数格式错误"、"业务规则不满足"等不同原因。自定义 code 可以做到更精确的错误标识，让客户端能针对不同错误类型做出不同的处理。

| HTTP 状态码 | 自定义 code | 含义 |
|-------------|-------------|------|
| 400 | 40001 | 请求体不是合法 JSON |
| 400 | 40002 | 必填字段缺失 |
| 400 | 40003 | 字段格式错误 |
| 400 | 40004 | 业务验证失败 |
| 401 | 40101 | Token 缺失 |
| 401 | 40102 | Token 过期 |
| 401 | 40103 | Token 无效 |
| 403 | 40301 | 无资源访问权限 |
| 403 | 40302 | 无操作权限 |
| 404 | 40401 | 资源不存在 |
| 500 | 50001 | 数据库错误 |
| 500 | 50002 | 外部服务调用失败 |

**注意**：HTTP 状态码仍然需要正确设置——200 表示 HTTP 层面请求成功，400 表示 HTTP 层面客户端错误。自定义 code 在 HTTP 状态码的基础上提供更精细的语义。

### 6.3 服务端实现示例

```python
from flask import Flask, jsonify, request
from functools import wraps

def api_response(data=None, code=200, message="success"):
    """统一的 API 响应封装"""
    return jsonify({
        "code": code,
        "message": message,
        "data": data,
        "timestamp": int(time.time() * 1000),
        "request_id": request.headers.get("X-Request-Id", "")
    }), code // 1000  # 返回 HTTP 状态码（自定义 code 的千位以上）

# 使用示例
@app.route('/api/users', methods=['POST'])
def create_user():
    try:
        data = request.get_json()
        if not data.get('name'):
            return api_response(
                code=40002, 
                message="用户名不能为空"
            )
        user = user_service.create(data)
        return api_response(data=user.to_dict(), code=200)
    except Exception as e:
        return api_response(
            code=50001,
            message=f"创建用户失败：{str(e)}"
        )
```

**代码逐行解释**：

- `api_response` 函数封装了请求 ID 获取逻辑，让 API 响应保持一致的结构。
- `code // 1000` 这一行实现了一个约定：自定义 code 的高三位等于 HTTP 状态码。200 → 200（HTTP 状态码），40002 → 400。这简化了客户端的处理——客户端只需检查 HTTP 状态码决定大类（成功/客户端错误/服务端错误），再检查自定义 code 决定具体错误类型。
- `request.headers.get("X-Request-Id", "")` 尝试从请求头中获取透传的请求 ID。在实际分布式环境中，这个请求 ID 由 API 网关或 Nginx 在请求入口处生成，传递到整个调用链，用于链路追踪。

## 七、HATEOAS——最常被忽略的 REST 约束

### 7.1 什么是 HATEOAS

HATEOAS（Hypermedia As The Engine Of Application State，超媒体作为应用状态的引擎）是 REST 统一接口约束的四个子约束之一，也是实际工程中最常被省略的约束。

简单来说，HATEOAS 要求：**API 响应中应该包含链接，告诉客户端下一步可以做什么**。客户端通过解析这些链接来发现 API 的功能，而不是依赖于硬编码的 URL。

### 7.2 有 HATEOAS 和没有 HATEOAS 的对比

**没有 HATEOAS 的 API**：
```json
GET /api/orders/123
{
  "id": 123,
  "status": "paid",
  "amount": 99.00
}
```
客户端需要事先知道：要支付这个订单，应该 `POST /api/orders/123/pay`；要查看物流，应该 `GET /api/orders/123/tracking`。

**有 HATEOAS 的 API**：
```json
GET /api/orders/123
{
  "id": 123,
  "status": "paid",
  "amount": 99.00,
  "_links": {
    "self": { "href": "/api/orders/123" },
    "pay": { "href": "/api/orders/123/pay", "method": "POST" },
    "cancel": { "href": "/api/orders/123/cancel", "method": "POST" },
    "tracking": { "href": "/api/orders/123/tracking", "method": "GET" },
    "items": { "href": "/api/orders/123/items", "method": "GET" }
  }
}
```
客户端只需要跟随链接——"支付"的链接就在响应中，而不需要硬编码 URL 格式。

**HATEOAS 的最大价值**：服务器可以改变 URL 结构而不影响客户端。只要链接在响应中正确给出，客户端就不需要关心 URL 的格式是什么。

### 7.3 为什么实际项目中很少用 HATEOAS

1. **增加了响应大小**：每个资源都要附带链接信息。
2. **客户端仍然需要了解业务逻辑**：即使链接在响应中，客户端仍然需要知道"当我要支付时，去找 `_links.pay` 这个链接"。HATEOAS 没有消除业务语义的约定。
3. **调试不便**：URL 隐藏在响应体中，不如直接看 URL 直观。
4. **移动端开发模式不匹配**：原生移动 App 通常会先设计好所有页面的 UI，不会像网页一样通过链接动态跳转。

所以实际工程中，大多数 API 没有严格实现 HATEOAS。但"在响应中返回相关资源的链接"这个思想仍然值得借鉴。

## 八、总结与实践清单

### 设计原则快查

| 原则 | 要点 |
|------|------|
| 使用名词复数 | `/users` 而不是 `/getUserList` |
| 用 HTTP 方法表达 CRUD | GET=读, POST=创建, PUT=替换, PATCH=更新, DELETE=删除 |
| URI 层级不超过三层 | `/users/123/orders` 可以，再深用查询参数 |
| 统一响应格式 | 所有接口返回相同结构的 JSON |
| 正确使用状态码 | 401 是认证问题，403 是权限问题 |
| API 版本化 | 用 `/v1/` 路径前缀，不要用查询参数 |
| 幂等性保护 POST | Token 机制或唯一约束防止重复提交 |

### 常见反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| `GET /getUsers` | URL 中包含动词 | `GET /users` |
| `POST /updateUser` | 方法+URL 双重表示 | `PUT /users/123` |
| 所有操作都用 POST | 丢失 HTTP 语义 | 按 CRUD 选择对应方法 |
| `200 {"code":500}` | HTTP 层面说是成功但实际是错误 | 用正确的 HTTP 状态码 |
| `404 {"message": "服务器内部错误"}` | 状态码和消息矛盾 | 用 500 表示服务器错误 |
| 不设版本号 | 无法演进 | 在 URL 中加入 `/v1/` |
| 不对 POST 做幂等保护 | 重复提交导致数据异常 | 使用 Token 或唯一约束 |
| 把密码放在 URL 中 | URL 被日志记录、浏览器历史保存 | 密码放在请求体中 |
| 返回的字段名用驼峰 | 前后端约定不一致 | 统一风格（后端通常用蛇形） |

### 区分要点汇总

**REST 不是协议，是风格**。它不是像 HTTP 或 TCP 那样的协议规范，而是一组架构风格约束。所以"RESTful API"没有必须严格遵守的标准——不同的团队有不同的理解和实践。重要的是理解每个约束的目的和权衡，根据业务场景做出合理的设计决策。

**REST 的核心价值在于"统一接口"**。六个约束中，统一接口是最核心的，也是让 REST API 可预测、自解释的关键。而统一接口中最容易被忽略的是 HATEOAS，最难做好的就是合理的资源设计和 HTTP 方法的正确使用。
