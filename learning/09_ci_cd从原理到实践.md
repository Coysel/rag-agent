# 第九部分：CI/CD

> **学习目标**：这一部分的目标是让你真正理解 CI/CD —— 不再把它当作一个"自动部署脚本"或者"面试必背概念"，而是理解持续集成和持续部署背后的设计思想、为什么现代软件开发离不开它、以及它在实际项目中如何运作。学完之后，你应该能回答"CI 和 CD 到底是什么区别"、"为什么说‘制品不可变’很重要"、"蓝绿部署和滚动部署各自适合什么场景"、"流水线即代码解决了什么问题"这类深层问题。

---

## 一、CI/CD 概述：它解决了什么问题

### 1.1 在没有 CI/CD 的时代

想象一下 2010 年前后一个典型团队的工作方式：

开发者在本地写完代码，在本地测试通过后，把代码通过 FTP 或 SCP 手动上传到服务器。然后 SSH 登录服务器，执行 `git pull`，重启服务。如果出了问题，开发者在服务器上直接修改文件试试——"在服务器上修一下看看能不能跑通"。整个部署过程完全依赖人工操作，没有任何自动化保障。

这种工作方式存在几个严重问题：

**问题一：集成地狱（Integration Hell）**
团队中每个人在自己的分支上开发。当多个人的功能需要合并时，才发现 A 改的接口和 B 调用的方式不一致，或者 C 引入的新依赖和 D 的代码冲突。等问题暴露的时候，距离上次集成已经过去了数周甚至数月——需要花大量时间来解决冲突和修复兼容性问题。

**问题二：手动部署不可靠**
手动执行部署命令意味着每一步都可能出错：忘记上传某个文件、复制了错误的配置、漏掉了数据库迁移步骤、关错了服务器进程。越是复杂的环境，手动出错的概率就越大。更糟的是，如果 A 能部署但 B 不能部署（因为只有 A 的机器上有某些密钥和工具），整个团队的部署效率就被一个人的瓶颈卡死了。

**问题三：没有可追溯性**
当生产环境出了故障，没有人能确定当前线上运行的是哪个版本的代码。代码可能是昨天部署的，也可能是上周部署的，甚至可能是某个人直接在服务器上修改过但没有提交到仓库的版本。没有任何记录能告诉你"某次部署到底发布了哪些变更"。

**问题四：部署恐惧症**
因为上述所有原因，团队的发布流程会变得越来越沉重。发布之前要开"发布评审会"，要列出长长的"发布检查清单"，要安排专门的"发布值班"人员。发布当天所有人紧张地盯着屏幕，祈祷不要出问题。每次发布都是一次"高危操作"——这反过来又导致团队尽量避免发布，宁愿积攒大量变更一次性发布，结果变更量越大，出问题的风险就越高，形成一个恶性循环。

### 1.2 CI/CD 的核心思想

CI/CD 不是某一种工具，而是一套**方法论**——它通过自动化的方式来应对软件交付中的不确定性。它的核心思想可以总结为一句话：

**让软件从代码提交到生产交付的整个过程变成可预测的、自动化的、可重复的。**

这个概念由三个相互关联的实践组成：

| 实践 | 英文 | 核心目标 | 解决的问题 |
|------|------|---------|-----------|
| **持续集成** | Continuous Integration (CI) | 频繁合并代码并自动验证 | 集成地狱、代码质量问题延迟发现 |
| **持续交付** | Continuous Delivery (CD) | 让每个合格的构建都能一键部署到生产 | 部署过程复杂不可靠 |
| **持续部署** | Continuous Deployment | 每次通过自动化测试的代码都自动部署到生产 | 人工决策成为瓶颈，发布频率低 |

这三个概念层层递进：持续集成是基础，持续交付是在持续集成之上保证"可发布"，持续部署是把"发布"这一步也自动化了。

> **一个直观的类比**：想象你在做一道菜。
> - **持续集成**就像你每放一种调料就尝一口味道——随时确认到目前为止做的都对，而不是等整道菜做好了才发现味道不对。
> - **持续交付**就像你把做好的菜放在保温箱里，任何时候客人点了都可以立即上桌——代码随时处于"可发布"的状态。
> - **持续部署**就像你开了一家自动餐厅，菜做好后自动送到客人桌上——不需要任何人决策"要不要上菜"。

### 1.3 CI/CD 的起源

持续集成的概念最早由 Grady Booch 在 1991 年提出，但真正把它推广到主流开发实践的，是 Martin Fowler 和 ThoughtWorks 团队在 2000 年代的工作。Fowler 在 2006 年发表的那篇《Continuous Integration》文章，至今仍是这个领域的经典入门读物。

早期的 CI 工具（如 CruiseControl、Hudson，Hudson 后来分叉成为 Jenkins）主要关注"持续构建"——也就是定时从版本控制系统中拉取代码，编译，然后报告结果。

到了 2010 年代，随着云计算和容器技术的普及，CI/CD 发生了质的飞跃：
- **Docker（2013）**解决了环境一致性问题——"在我的机器上能跑"成为了历史
- **GitHub Actions（2018）**和 **GitLab CI（2011）**等平台将 CI/CD 直接集成到代码托管平台中
- **云原生 CI/CD**（如 ArgoCD、Tekton）使得持续部署和 Kubernetes 紧密结合

今天，CI/CD 已经成为现代软件开发的基础设施——就像版本控制系统一样，是一个项目"默认就该有"的东西。

## 二、持续集成（Continuous Integration）

### 2.1 什么是持续集成

持续集成的核心实践是：**每个开发人员每天至少一次将代码合并到主干分支，每次合并都触发一次自动化的构建和测试流程。**

这意味着两件事情：

1. **频繁合并**（而不是在功能开发完成后再合并）
2. **每次合并都自动验证**（而不是只提交却不验证）

```yaml
# 一个典型的 CI 流程（以 GitHub Actions 为例）
name: CI
on:
  push:            # 任何时候有人 push 代码到任意分支
    branches: [main, develop]
  pull_request:    # 任何时候有人创建或更新 PR
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3      # 1. 检出代码
      - uses: actions/setup-python@v4  # 2. 设置 Python 环境
        with:
          python-version: "3.11"
      - run: pip install -r requirements.txt  # 3. 安装依赖
      - run: pytest tests/              # 4. 运行测试
      - run: flake8 src/                # 5. 代码风格检查
```

这个配置文件做了什么：当开发者 push 代码或创建 Pull Request 到 main/develop 分支时，GitHub 会自动创建一个虚拟机，在这个虚拟机上执行定义好的步骤。每一步都自动化执行，结果会显示在 PR 页面上——任何人都可以看到这次变更是否通过了所有检查。

### 2.2 为什么需要频繁集成

你可能会有疑问："为什么一定要频繁合并？我专心在功能分支上开发两周，最后合并一次，不行吗？"

**答案是：行的通，但代价巨大。** 这里的关键问题不是"能不能合并"，而是"合并时你面临多大的风险"。

假设你在一个功能分支上开发了两周，主分支上其他同事已经合入了 10 次代码变更。当你最终合并时，你需要解决这 10 次变更与你的代码之间的所有冲突。冲突涉及的范围越大，解决难度越高——不仅仅是文本层面的合并冲突（"这行改了，那行也改了"），还有语义层面的冲突（"他们的代码依赖了一个方法，但你的重构删掉了这个方法"）。

频繁集成的核心洞察是：**冲突的解决成本与冲突发生的时间距离呈指数级增长**。今天合并昨天同事的代码，可能只需要花 5 分钟解决一个小冲突。但两周后合并同事的 10 次变更，可能要花两个小时才能把所有冲突理清楚——而且这个过程中还可能引入新的 bug。

```
冲突修复成本
     ↑
     │                    ● (两周后合并：高成本高风险)
     │                   /
     │                  /
     │                 /
     │                /
     │   ● (每天合并：低风险，快速修复)
     │  /
     │ /
     └──────────────────────────→ 时间
         每天合并             两周合并一次
```

### 2.3 CI 的反馈机制

CI 之所以有效，是因为它建立了一个**快速反馈循环**：

```
提交代码 → CI 启动构建 → 运行测试 → 报告结果 → 开发者收到通知 → 修复问题
   │                                                                   │
   └───────────────────────── 几分钟到十几分钟 ────────────────────────┘
```

这个循环越短越好。行业最佳实践是：**CI 流程应该在 10 分钟内完成**。如果 CI 跑一次需要一个小时，开发者提交代码后就去干别的事了，等回来时已经忘记了刚才写的代码的上下文——修复问题的效率会大大降低。

> **为什么 CI 要快？** 心理学研究发现，人们在任务切换时会产生"上下文切换成本"——从一个任务切换到另一个任务，需要一定时间重新进入状态。如果 CI 反馈时间太长，开发者会切换到其他工作，当 CI 报错时又要切换回来，每次切换都会消耗注意力。10 分钟以内的反馈意味着开发者可以在当前上下文没有被完全清除之前就处理问题。

### 2.4 CI 的核心支柱

一个有效的 CI 系统包含以下四个核心要素：

**1. 版本控制系统**

所有代码必须存放在版本控制系统中（Git、SVN 等）。这是 CI 的基础——没有版本控制就谈不上"自动触发构建"。更重要的是，版本控制让每一次变更都可追溯、可回滚。

**2. 自动化构建**

构建过程必须完全自动化，不依赖任何人工操作。对于编译型语言（Java、Go、Rust），构建包括编译代码、打包成可执行文件或容器镜像。对于解释型语言（Python、JavaScript），构建包括安装依赖、处理静态资源、生成必要的配置文件。

构建脚本和应用程序代码一起存放在版本控制中——这就是"构建即代码"的理念。

**3. 自动化测试**

CI 的核心价值在于验证代码的正确性。这需要自动化测试套件来保证：
- **单元测试**：验证单个函数或方法的正确性，运行速度最快
- **集成测试**：验证不同模块之间的交互是否正常
- **接口测试**：验证 API 的行为是否符合预期

**4. 反馈机制**

CI 的结果必须以最快速度通知到相关人员。常见的反馈方式包括：
- Pull Request 页面上的状态标记（绿色勾 / 红色叉）
- 即时通讯工具（Slack、飞书、钉钉）的通知
- 邮件通知

一个好的 CI 实践要求：**CI 失败被视为严重事件，团队必须优先修复**。如果允许"CI 红了也没关系，先合并代码"，CI 就失去了它的意义。

## 三、持续交付（Continuous Delivery）

### 3.1 从持续集成到持续交付

持续集成保证了你每次提交的代码都能通过编译和测试。但"代码能跑"和"代码可以发布到生产环境"之间还有很大距离。

持续交付（Continuous Delivery）在持续集成之上增加了一件事：**确保每个通过 CI 的构建版本都可以一键部署到生产环境**。注意，是"可以部署"而不是"已经部署"——是否实际部署到生产环境仍然由人工决策。

```
持续集成阶段：
提交代码 → 构建 → 单元测试 → 集成测试 → 代码检查 → 产出构建产物（artifact）

持续交付追加的阶段：
构建产物 → 部署到预发布环境 → 运行自动化验收测试 → 性能测试 → 安全扫描
                                                          ↓
                                                 等待人工审批 → 部署到生产环境
```

### 3.2 "随时可部署"意味着什么

持续交付的核心理念是：**让软件的每个版本都处于"可部署到生产环境"的状态**。

这听起来很抽象，但它的实际含义非常具体：

- **你的数据库迁移脚本必须自动化**，不能依赖于 DBA 手动执行 SQL
- **你的配置管理必须与环境解耦**，同一个构建产物可以在开发、测试、预发布、生产四个环境以不同的配置运行
- **你的发布步骤必须脚本化**，不能包含"手动修改服务器上的配置文件"这样的步骤
- **你的回滚流程也必须自动化**，如果发布出了问题，应该能一键回滚到上一个版本

### 3.3 持续交付的价值

持续交付解决的主要问题不是"加快发布速度"，而是**降低每次发布的风险**。

这个结论可能反直觉——你可能会想，"发布越多风险越大才对"。但实际数据表明恰恰相反：**发布频率越高的团队，单次发布的故障率越低**。

| 方面 | 低频发布（每月一次） | 高频发布（每天多次） |
|------|-------------------|-------------------|
| 单次发布的变更量 | 大（几十个功能、上百个 commit） | 小（1-2 个功能、几个 commit） |
| 出问题时的影响范围 | 大（无法确定哪个变更导致了问题） | 小（可以精确定位到特定变更） |
| 回滚复杂度 | 高（回滚意味着撤销大量变更） | 低（只需撤销特定变更） |
| 问题修复速度 | 慢（需要等到下次发布窗口） | 快（可以立即修复并发布） |

这个现象可以用一个简单的例子来理解：假设你要从 100 级台阶上走下来。如果你一次跨 50 级台阶，虽然总步数少（低频发布），但每次迈步的风险极高——一旦踩空，你摔得很重。如果你一次跨 1 级台阶，虽然步数多，但每一步都是安全的。即使某一步踩空了，你也就摔一个台阶的高度。

**持续交付的本质就是：用小步快跑代替大步跃进，让每次变更的风险可控。**

### 3.4 发布流水线（Deployment Pipeline）

持续交付的核心技术实现是**发布流水线**。发布流水线将"从代码提交到生产部署"的整个过程分解为多个阶段（stage），每个阶段有明确的输入、输出和验证标准。

```
代码提交
    │
    ▼
┌─────────────┐      ┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│   阶段 1     │      │   阶段 2     │      │   阶段 3     │      │   阶段 4     │
│   版本构建    │ ──→  │   单元测试    │ ──→  │   自动化验收   │ ──→  │   性能测试    │
│              │      │              │      │   测试       │      │              │
│ 输入：代码    │      │ 输入：构建    │      │ 输入：部署到   │      │ 输入：通过    │
│ 输出：二进制包 │      │ 产物         │      │ 预发布环境    │      │ 验收的构建    │
└─────────────┘      └─────────────┘      └─────────────┘      └─────────────┘
                                                                │
                                                                ▼
                                                          ┌─────────────┐
                                                          │   阶段 5     │
                                                          │   一键部署    │
                                                          │   (人工审批)  │
                                                          │ 输入：通过    │
                                                          │ 所有验证     │
                                                          └─────────────┘
```

每个阶段的输出是一个**经过验证的构建产物（verified artifact）**。只有通过了所有验证的构建才能进入生产环境。

> **为什么要把流水线分成多个阶段？** 核心原因是**尽早反馈**。想象一下，如果你只有一个"全流程"测试阶段——跑完全部测试需要 2 小时——那么提交代码 2 小时后你才知道是否通过。但如果把流水线拆成"快速单元测试（5 分钟）→ 集成测试（20 分钟）→ 端到端测试（1 小时）"三个阶段，开发者提交代码后 5 分钟就能收到最关键的反馈——即使后面的阶段还在运行。这和 CI 的"快速反馈"理念一脉相承。

## 四、持续部署（Continuous Deployment）

### 4.1 持续部署 vs 持续交付

持续部署和持续交付的唯一区别在于——**生产部署这一步是否由人工触发**。

```
持续交付：代码 → 构建 → 测试 → 预发布验证 → [人工审批] → 部署生产
持续部署：代码 → 构建 → 测试 → 预发布验证 → 自动部署生产
```

在持续交付中，团队可以选择在某一个特定的时间点发布（比如每周四下午三点）。而在持续部署中，每次 commit 通过所有自动化检查后都会自动部署到生产环境——没有等待、没有排期、没有人工审批。

### 4.2 持续部署需要哪些前提条件

持续部署是 CI/CD 中最"激进"的实践，它要求团队高度成熟。不是所有团队都适合持续部署。在实施持续部署之前，需要具备以下条件：

1. **完善的自动化测试覆盖**：代码的正确性完全由自动化测试来保障。如果测试覆盖不全，持续部署会把未经验证的代码直接推到生产环境，风险极高。

2. **快速的部署和回滚机制**：部署过程必须可以在几分钟内完成。如果发现问题，回滚也必须同样快速。部署不是"一次性事件"而应该是"常态操作"。

3. **功能开关（Feature Flag）**：有些功能虽然代码已经合入了，但还不适合对用户开放。通过功能开关，可以在运行时控制某个功能是否可见，而不需要重新部署代码。

4. **监控和可观测性**：必须有完善的监控系统（APM、日志、告警），在部署后能迅速发现异常。如果部署出了问题但没有人知道，持续部署就是一场灾难。

5. **小批量变更**：每次部署的变更范围要足够小。如果每次部署只包含 1-2 个 commit，定位问题的成本极低；如果积攒了 50 个 commit 才部署，出了问题很难排查。

### 4.3 什么团队适合持续部署

在实践中，不同的团队对持续部署的需求不同：

| 业务类型 | 部署策略 | 原因 |
|---------|---------|------|
| SaaS 产品（如 GitHub、Slack） | 持续部署或持续交付 | 用户可以接受在线服务频繁更新，且快速迭代是核心竞争力 |
| 金融系统（银行交易系统） | 持续交付（严格审批） | 合规要求高，必须有人工审核流程 |
| 移动应用（App Store/Google Play） | 持续交付 | 应用商店审核机制决定了不能自动发布 |
| 嵌入式系统（汽车、医疗设备） | 极低频部署 | 安全关键系统，每次发布都需严格的认证流程 |
| 内部的工具和服务 | 持续部署 | 用户容忍度高，快速获得反馈更重要 |

## 五、CI/CD 流水线的阶段详解

### 5.1 流水线的通用结构

虽然不同的技术栈和工具有不同的配置方式，但 CI/CD 流水线的基本结构是通用的。

```
┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
│ 代码  │─→│ 构建  │─→│ 测试  │─→│ 打包  │─→│ 部署  │─→│ 验证  │
│ 检出  │  │      │  │      │  │      │  │      │  │      │
└──────┘  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘
```

下面逐一详解每个阶段。

### 5.2 代码检出阶段

这是流水线的起点。CI/CD 系统从版本控制系统中拉取指定分支或 tag 的代码。

```yaml
# GitHub Actions 中的代码检出
steps:
  - name: Checkout code
    uses: actions/checkout@v3
    with:
      fetch-depth: 0  # 拉取完整历史（而非仅最新 commit）
```

**fetch-depth: 0** 的含义：默认情况下，为了节省时间，GitHub Actions 只拉取最新的一次 commit（`--depth=1`——浅克隆）。但某些场景需要完整的历史记录——比如需要根据 Git 历史来自动生成版本号、或者进行一些需要对比历史代码的静态分析——这时就需要设置 `fetch-depth: 0` 来拉取完整历史。

> **为什么需要完整历史？** 假设你的版本号策略是 `git describe` 生成的（如 `v1.2.3-5-gabc1234`——基于标签加 commit 数的格式）。如果只有浅克隆（`--depth=1`），`git describe` 找不到最近的 tag，就无法正确生成版本号。

### 5.3 构建阶段

构建阶段的目标是：**将源代码转换为可执行的产物**。不同语言和框架的"构建"含义不同：

| 技术栈 | 构建的含义 | 产出的产物 |
|--------|-----------|-----------|
| Java (Maven/Gradle) | 编译 Java 源代码为 .class 文件，打包为 JAR/WAR | `app.jar` |
| Go | 编译为独立的二进制可执行文件 | `app` (Linux 二进制) |
| Rust | 编译为二进制文件 | `app` |
| Node.js / Python | 安装依赖，可能还需要打包静态资源 | `node_modules/` + 代码 |
| React / Vue 前端 | 构建工具（Webpack/Vite）打包为静态文件 | `dist/` 目录 |
| Docker | 构建容器镜像 | `image:tag` |

一个典型的 Java 构建阶段：

```yaml
- name: Build with Maven
  run: |
    mvn clean package \
      -DskipTests=true \       # 跳过测试（测试在下一个阶段运行）
      -B                       # 批处理模式（减少日志冗长度）
```

注意这里 `-DskipTests=true` —— 这个参数的意思是"编译阶段不做测试"。"那测试什么时候做？"在流水线的下一个独立阶段专门做。把构建和测试分开的好处是：如果构建失败了（编译错误），可以立即失败并通知开发者，不需要等待测试环境就绪。这也体现了流水线"尽早反馈"的设计原则。

### 5.4 测试阶段

测试阶段是 CI/CD 流水线的质量核心。它通常包含多个层次的测试：

```
测试金字塔（Test Pyramid）——越往上，测试越慢、越贵、覆盖率越低
       
       /\
      /  \        端到端测试（E2E Tests）
     /    \       测试整个系统，模拟真实用户操作
    /──────\
   /        \     集成测试（Integration Tests）
  /          \    测试模块之间的交互
 /────────────\
/              \  单元测试（Unit Tests）
                 测试单个函数/方法，运行最快，覆盖面最广
```

**单元测试（Unit Tests）**

单元测试验证的是最小的可测试单元（函数、方法）。它的特点是：
- 运行速度快（毫秒级）
- 不依赖外部服务（数据库、文件系统、网络）
- 应该是每个 PR 的必过项

```python
# 单元测试示例（Python + pytest）
def test_calculate_discount():
    """测试折扣计算函数：会员满100减20"""
    result = calculate_discount(price=100, is_member=True)
    assert result == 80  # 100 - 20 = 80

def test_calculate_discount_non_member():
    """测试非会员不享受折扣"""
    result = calculate_discount(price=100, is_member=False)
    assert result == 100  # 不打折
```

**集成测试（Integration Tests）**

集成测试验证的是多个模块配合工作的正确性。它通常需要启动外部服务（数据库、消息队列）。运行速度比单元测试慢，但覆盖的边界条件更多。

```python
# 集成测试示例（需要数据库连接）
def test_user_registration_flow(db_session):
    """测试完整的用户注册流程"""
    # 1. 创建用户
    user = UserService.register(
        email="test@example.com",
        password="Password123!"
    )
    assert user.id is not None  # 用户被正确持久化

    # 2. 验证邮件被发送
    email = EmailService.get_latest(to=user.email)
    assert email.subject == "请验证您的邮箱"

    # 3. 使用验证码激活用户
    UserService.activate(email.verification_code)
    assert user.is_active == True
```

**端到端测试（E2E Tests）**

E2E 测试模拟真实用户从头到尾使用系统的过程。它通常使用浏览器自动化工具（如 Selenium、Playwright）来执行。

```python
# 端到端测试示例（Playwright）
def test_user_can_complete_purchase(page):
    """模拟用户在购物网站上完成一次完整购买"""
    page.goto("https://staging.example.com")

    # 1. 搜索商品
    page.fill('input[data-testid="search"]', '无线耳机')
    page.click('button[data-testid="search-btn"]')

    # 2. 加入购物车
    page.click('button[data-testid="add-to-cart"]')

    # 3. 结算支付
    page.click('button[data-testid="checkout"]')
    page.fill('input[name="card"]', '4111111111111111')
    page.click('button[data-testid="pay"]')

    # 4. 验证成功页面
    assert page.is_visible('text="订单已确认"')
```

> **为什么测试要分层？** 一个常见的误区是"测试越多越好"。实际上，不同的测试有不同的 ROI。单元测试运行快、维护成本低、发现问题精准——所以它应该占测试数量的最大比例（70-80%）。E2E 测试虽然最接近真实场景，但运行慢、维护成本高、失败后定位问题困难——所以它应该只占 5-10%。分层测试就是在质量保障和效率之间找到最佳平衡点。

### 5.5 代码质量检查

除了功能测试，流水线中还通常包含自动化代码质量检查：

```yaml
- name: Code style check
  run: flake8 src/ --max-line-length=100

- name: Type check
  run: mypy src/ --strict

- name: Security scan
  run: bandit -r src/ -ll    # 扫描常见安全漏洞

- name: Dependency vulnerability check
  run: safety check           # 检查依赖包是否有已知漏洞
```

**为什么静态检查也需要放在流水线中？** 因为静态代码检查保证的是代码库的"健康卫生"——代码风格统一、类型标注正确、没有明显安全漏洞。这些检查在本地也能跑，但它们放在流水线中的意义在于强制执行：任何不符合规范的代码都无法合并到主分支。

### 5.6 打包与制品管理

测试通过后，流水线进入打包阶段——生成最终的交付产物（artifact）。

对于容器化应用，制品通常是 Docker 镜像：

```yaml
- name: Build Docker image
  run: |
    docker build \
      -t myapp:${{ github.sha }} \       # 标签使用 Git commit SHA
      -t myapp:latest \                   # 同时也打 latest 标签
      .

- name: Push to registry
  run: |
    docker push myapp:${{ github.sha }}
    docker push myapp:latest
```

**为什么用 commit SHA 作镜像标签？** 因为 commit SHA 是唯一的、不可更改的。如果你使用 `latest` 标签，下次构建后 `latest` 指向的就是新版本——你无法通过 `latest` 知道线上跑的是哪次提交的代码。但如果你用 SHA：`myapp:abc123def`，那么任何人都能准确知道这个镜像对应的是哪个 commit。

### 5.7 部署阶段

部署阶段将经过验证的制品部署到目标环境。部署策略有很多种（详见第六节），但关键原则是——部署必须**幂等**（idempotent，即多次执行产生相同结果）、**可重复**、**可回滚**。

```yaml
- name: Deploy to staging
  run: |
    # 更新 Kubernetes 部署
    kubectl set image deployment/myapp \
      myapp=myapp:${{ github.sha }} \
      --namespace=staging

- name: Health check
  run: |
    # 等待部署完成并验证健康
    kubectl rollout status deployment/myapp \
      --namespace=staging --timeout=5m
```

### 5.8 部署后验证

部署完成后，流水线还要做最后一件事——验证部署确实成功了。这一步被称为**部署后验证**或**冒烟测试**：

```yaml
- name: Post-deploy smoke test
  run: |
    # 验证关键 API 是否正常响应
    curl -f http://staging.example.com/api/health

    # 验证首页是否正常返回 200
    curl -f http://staging.example.com/
```

这个阶段捕获的是那些在测试环境中没有发现、但在真实环境中才出现的问题——比如配置错误、依赖的服务没有正确连接等。

## 六、部署策略

当你需要将一个应用的新版本部署到生产环境时，面临的一个基本问题是：**如何以最小的风险完成这次变更？**

不同的部署策略给出了不同的答案。

### 6.1 蓝绿部署（Blue-Green Deployment）

**核心思想**：维护两套完全相同的生产环境——"蓝环境"（当前运行版本）和"绿环境"（新版本）。在新环境上完成部署和验证后，通过负载均衡器将流量直接切换到新环境。

```
部署前：
┌──────────────┐                     ┌──────────────┐
│  蓝环境 (v1) │ ←── 所有用户流量    │  绿环境 (v2) │
│  (当前生产)   │                     │  (空闲/待部署)│
└──────────────┘                     └──────────────┘

部署后（流量切换）：
┌──────────────┐                     ┌──────────────┐
│  蓝环境 (v1) │                     │  绿环境 (v2) │ ←── 所有用户流量
│  (备用/可回滚)│                     │  (当前生产)   │
└──────────────┘                     └──────────────┘
```

**优点**：
- 切换是瞬间完成的——用户几乎无感知
- 回滚极其简单——只需把流量切回旧环境
- 新环境可以充分验证后再切换

**缺点**：
- 需要两倍的资源（双倍的服务器成本）
- 如果新旧版本的数据格式不兼容（比如数据库 Schema 变更），切换后旧环境可能无法正常工作

**适用场景**：关键业务系统、对停机时间敏感的应用。

### 6.2 滚动部署（Rolling Deployment）

**核心思想**：逐步替换运行中的实例，一次替换一个或一批。在这个过程中，始终有实例在提供服务。

```
时间 → 

开始： [v1] [v1] [v1] [v1] [v1]   ← 5 个实例都运行旧版本
              ↓
步骤1： [v2] [v1] [v1] [v1] [v1]   ← 替换第一个实例
              ↓
步骤2： [v2] [v2] [v1] [v1] [v1]   ← 替换第二个实例
              ↓
步骤3： [v2] [v2] [v2] [v1] [v1]   ← 替换第三个实例
              ↓
步骤4： [v2] [v2] [v2] [v2] [v1]   ← 替换第四个实例
              ↓
完成： [v2] [v2] [v2] [v2] [v2]   ← 全部替换完成
```

**优点**：
- 不需要额外资源——用新版本逐步替换旧版本
- 部署过程中服务始终可用（只有减少了一部分容量）

**缺点**：
- 回滚复杂——需要反向逐步替换
- 新旧版本会共存一段时间，需要考虑兼容性问题
- 部署过程耗时较长（取决于实例数量）

**适用场景**：微服务架构、无状态应用、资源有限的环境。

### 6.3 灰度部署 / 金丝雀部署（Canary Deployment）

**核心思想**：先让一小部分用户（如 1%）使用新版本，观察一段时间后，如果一切正常，逐步扩大范围，直到 100%。

这个名称来源于煤矿中的"金丝雀"——矿工带金丝雀下井，如果金丝雀因为有毒气体死亡，矿工就知道环境不安全了。同样，新版本先暴露给一小部分用户，如果出现问题，也只影响这一小部分用户。

```
步骤1： [v1] [v1] [v1] [v1] [v1]   ← 100% v1
        1%流量→[v2]                ← 新版本仅服务 1% 用户

步骤2（验证通过后）：
[v1] [v1] [v1] [v1] [v1]          ← 旧版本
        10%流量→[v2]               ← 逐步扩大到 10%

步骤3（再次验证后）：
[v1] [v1] [v1]                    ← 旧版本逐步缩减
                  [v2] [v2]       ← 新版本逐步扩容
                  50%流量→[v2]

步骤4（最终）：
         [v2] [v2] [v2] [v2] [v2] ← 100% v2
```

**优点**：
- 风险最低——问题只会影响小部分用户
- 可以在生产环境上做真实的 A/B 测试
- 发现异常时可以立即将金丝雀流量切回旧版本

**缺点**：
- 部署逻辑复杂——需要流量路由控制能力
- 部署周期长——需要等待观测时间
- 需要完善的监控和告警支持

**适用场景**：对用户体验敏感的应用、大型分布式系统、需要逐步验证新版本稳定性。

### 6.4 红黑部署（Red-Black Deployment）

这是 Netflix 开发的部署策略，本质上是蓝绿部署在 AWS 环境下的实现变体。区别在于：红黑部署在部署完成后，会释放旧环境的资源（而不是像蓝绿部署那样保留备用）。

### 6.5 各种部署策略的对比

| 策略 | 资源需求 | 回滚速度 | 部署速度 | 风险控制 | 实现复杂度 |
|------|---------|---------|---------|---------|-----------|
| **蓝绿部署** | 高（双倍资源） | 极快（瞬间切换） | 快 | 中（切换后有全局影响） | 低 |
| **滚动部署** | 低（无需额外资源） | 慢（反向逐步替换） | 中 | 低（新旧版本共存有兼容风险） | 中 |
| **金丝雀部署** | 中（需要额外实例） | 快（切回流量即可） | 慢（需要观察期） | 极低（影响范围可控） | 高 |
| **重建部署（Recreate）** | 低 | 慢 | 快 | 高（有停机时间） | 最低 |

> **为什么会有这么多部署策略？** 部署策略的本质是在一组相互冲突的目标之间寻找平衡：**安全**（即使出问题影响也小）、**速度**（部署够快）、**成本**（不需要太多额外资源）、**复杂度**（容易理解和维护）。没有一种策略是"最好"的，只有"最适合当前场景"的。

### 6.6 部署策略选择指南

如何选择部署策略？考虑以下因素：

1. **你的应用是有状态还是无状态？**
   - 无状态应用（可以随时替换，不依赖本地存储的数据）适合所有部署策略
   - 有状态应用（数据库、缓存集群）的部署要复杂得多，通常需要蓝绿部署或专门的迁移方案

2. **你的用户能否接受短暂中断？**
   - 不能接受（电商网站、在线银行）：使用蓝绿、滚动或金丝雀部署
   - 能接受短暂中断（内部分析工具）：重建部署就足够了

3. **你的回滚机制是否成熟？**
   - 回滚机制完善：可以接受风险稍高的策略
   - 回滚机制不完善：选择风险最低的策略（金丝雀）

4. **你的资源预算是否充足？**
   - 充足：蓝绿部署最简单可靠
   - 有限：滚动部署在成本和安全性之间取得平衡

## 七、制品不可变性

### 7.1 什么是制品不可变性

**制品不可变性（Artifact Immutability）** 是一个非常简单的原则：**一旦制品（构建产物）被创建并发布到制品仓库，它就应该被标记为只读的，永远不允许被修改或覆盖。**

```
❌ 错误做法：
    - 构建了一个 JAR 包 v1.0，发现有个 bug，就修改了 JAR 包里的 class 文件
    - 把同一个 tag 的 Docker 镜像推送了两次，第二次覆盖了第一次的内容

✅ 正确做法：
    - 每次构建都生成一个新的、唯一标识的版本（如 v1.0.1）
    - 每个版本的制品一旦发布就不可修改
    - 如果需要修复 bug，从源码重新构建一个新版本
```

### 7.2 为什么制品不可变性如此重要

**原因一：可追溯性（Traceability）**

如果你能确定线上运行的版本就是某个特定构建的产物，那么当出现问题时，你可以回溯到这个构建的源 commit、当时的构建参数、使用的依赖版本。但如果制品是可变的（可以被覆盖），你就永远无法确认线上跑的到底是什么。

```
可变制品的问题：
代码 commit abc123 → 构建了 app.jar → 部署
          ↓（发现 bug）
有人修改了 app.jar（替换了某个 class 文件）
          ↓
线上跑的是修改后的 app.jar —— 但对应的源码 commit 还是 abc123
          ↓
调试时查看 abc123 的代码 —— 看到的和线上实际跑的代码不一致！
```

**原因二：可重现性（Reproducibility）**

不可变制品意味着：给定相同的源码版本、相同的构建条件，你应该能得到完全相同的制品。这听起来是理所当然的，但实际上并不简单——因为构建过程可能依赖外部因素：
- 今天从 PyPI 下载的依赖包版本和明天下载的可能不同
- 构建时间、随机数种子等因素可能影响构建结果

确保可重现性的常用方法是使用**依赖锁定**（如 `package-lock.json`、`poetry.lock`、`requirements.txt` 中锁定精确版本号）和**构建环境标准化**（如使用 Docker 镜像作为构建环境）。

**原因三：安全性**

如果制品可以被覆盖，攻击者可能利用这一点来替换制品——把"看起来是 v1.0 但实际上包含恶意代码"的制品推送到仓库。不可变性配合签名机制，可以确保制品的来源和完整性都是可信的。

### 7.3 版本号策略

实现制品不可变性的关键在于合理的版本号策略。最常见的方案是**语义化版本号（SemVer）**：

```
语义化版本号格式：主版本号.次版本号.修订号

主版本号（Major）：做了不兼容的 API 修改时增加
次版本号（Minor）：向下兼容的功能性新增时增加
修订号（Patch）：向下兼容的问题修正时增加

例子：
  v1.0.0 → v2.0.0  → API 发生了不兼容的变更
  v1.0.0 → v1.1.0  → 新增了一个功能，向后兼容
  v1.0.0 → v1.0.1  → 修复了一个 bug
```

在实践中，许多 CI/CD 系统使用**Git commit SHA** 或**构建号**作为唯一的制品版本标识：

```yaml
# 方法 1：使用 Git commit SHA（最常用）
docker build -t myapp:${GIT_COMMIT_SHA}

# 方法 2：使用构建号（在 CI 系统中自动递增）
docker build -t myapp:build-${BUILD_NUMBER}

# 方法 3：组合（Git tag + commit SHA）
docker build -t myapp:${GIT_TAG}-${GIT_COMMIT_SHORT_SHA}
```

> **为什么不用 `latest` 标签？** `latest` 是一个"浮动标签"——它随着每次构建被更新。虽然方便（你总是可以用 `:latest` 拿到最新版本），但也打破了不可变性原则。正确的做法是：使用唯一的、不可变的标签（如 SHA）来标识具体版本，同时也可以加 `latest` 作为"快捷方式"，但不要依赖 `latest` 来精确定位某个版本。

## 八、流水线即代码（Pipeline as Code）

### 8.1 传统流水线的痛点

在"流水线即代码"流行之前，CI/CD 流水线通常通过工具的 Web UI 来配置。这种方式有三个重大问题：

1. **配置不可追踪**：流水线配置保存在 CI 工具的数据库中，不在版本控制中。你无法通过 `git log` 查看"流水线什么时候被修改过"。

2. **不可重复创建**：如果你需要搭建一套新的 CI 环境（比如新项目启动），你必须手动在 Web UI 上重新配置一遍所有步骤。这个过程容易出错且不可自动化。

3. **环境依赖**：流水线配置和 CI 工具绑定在一起。如果从 Jenkins 迁移到 GitLab CI，所有配置需要重写。

### 8.2 流水线即代码的做法

**流水线即代码（Pipeline as Code）** 将流水线配置以代码文件的形式存放在项目的版本仓库中。这个文件和应用程序代码一起提交、一起审查、一起版本管理。

```yaml
# .gitlab-ci.yml —— 一个典型的流水线即代码配置
stages:
  - build
  - test
  - deploy

variables:
  DOCKER_IMAGE: registry.example.com/myapp
  DOCKER_TAG: $CI_COMMIT_SHORT_SHA

build:
  stage: build
  script:
    - docker build -t $DOCKER_IMAGE:$DOCKER_TAG .
    - docker push $DOCKER_IMAGE:$DOCKER_TAG

test:
  stage: test
  script:
    - docker run $DOCKER_IMAGE:$DOCKER_TAG pytest

deploy:
  stage: deploy
  script:
    - kubectl set image deployment/myapp myapp=$DOCKER_IMAGE:$DOCKER_TAG
  only:
    - main            # 只有 main 分支的提交才触发部署
```

> **为什么流水线配置本身也需要放在代码仓库中？** 这是"基础设施即代码"（Infrastructure as Code）理念在 CI/CD 领域的延伸。当你的流水线定义和代码放在一起时，修改流水线的过程与修改代码的过程完全相同：编写变更 → 提交 PR → 代码审查 → 合并 → 生效。没有人可以通过"偷偷修改 Jenkins 配置"来绕过代码审查。

### 8.3 流水线即代码的实践价值

**代码审查应用于流水线本身**

最直接的受益是：流水线的修改也经过 Pull Request 审查流程。团队成员可以 review "新的部署步骤是否正确"、"使用了正确的环境变量吗"等。

**分支环境隔离**

如果流水线配置文件在版本控制中，你可以为不同分支定义不同的行为：

```yaml
# main 分支部署到生产，feature 分支部署到测试环境
deploy:
  stage: deploy
  script:
    - ./deploy.sh $ENVIRONMENT
  environment:
    name: $CI_COMMIT_BRANCH
  except:
    - tags

# tag 触发的是发布到生产
deploy-production:
  stage: deploy
  script:
    - ./deploy.sh production
  only:
    - tags
```

**流水线的版本快照**

当你需要回滚到某个历史版本时，如果流水线配置也在版本控制中，你可以直接 checkout 到那个历史版本的代码和流水线配置，重新执行构建——得到的结果应该和当时一模一样。

### 8.4 主流流水线即代码工具对比

| 工具 | 配置文件位置 | 语法 | 运维方式 |
|------|------------|------|---------|
| **GitHub Actions** | 仓库中 `.github/workflows/*.yml` | YAML | 托管服务（GitHub 提供） |
| **GitLab CI** | 仓库中 `.gitlab-ci.yml` | YAML | 托管或自托管（GitLab Runner） |
| **Jenkins Pipeline** | Jenkinsfile（在仓库中） | Groovy DSL 或声明式 | 自托管 |
| **CircleCI** | 仓库中 `.circleci/config.yml` | YAML | 托管服务 |
| **Argo Workflows** | Kubernetes CRD YAML | YAML（Kubernetes 原生） | 自托管（Kubernetes 内） |
| **Tekton** | Kubernetes CRD YAML | YAML（Kubernetes 原生） | 自托管（Kubernetes 内） |

选择 CI/CD 工具时，需要考虑的因素：
- **托管 vs 自托管**：托管服务（GitHub Actions、CircleCI）免运维，但受限于平台能力；自托管（Jenkins、Tekton）提供最大灵活性，但需要人力维护
- **生态整合**：如果你的代码在 GitHub 上，GitHub Actions 提供了最无缝的整合
- **Kubernetes 原生**：如果你的基础设施是 Kubernetes，ArgoCD / Tekton 等 K8s 原生工具可以提供更一致的体验

## 九、质量门（Quality Gates）

### 9.1 什么是质量门

**质量门（Quality Gates）是流水线中设定的"检查点"——只有通过特定质量标准的构建才能进入下一个阶段。**

可以把质量门想象成流水线上的"安检"：每一道门检查特定的指标，只有通过了才能继续前进。任何一道门失败，流水线都会停止并标记失败。

```
代码提交 → [门1：编译通过] → [门2：单元测试通过] → [门3：代码覆盖率>80%]
    ↓                                                      ↓
  失败：通知开发者                                       成功 → 进入预发布环境
                                                              ↓
                                              [门4：API 测试通过] → [门5：性能测试通过]
                                                   ↓                      ↓
                                                失败                     成功 → 等待审批
```

### 9.2 常见的质量门

**编译/构建门**
最简单的质量门——代码能不能编译过？如果编译都失败了，后面的所有阶段都不需要运行。

**测试通过门**
所有单元测试必须绿色通过。这里的"通过"是指零失败——"95% 通过率"通常被认为是不够的。如果有测试本身不稳定（flaky test，即有时通过有时失败），应该修复或删除这些测试，而不是允许它们在质量门下"偶尔失败"。

**代码覆盖率门**
设定一个最低的代码覆盖率阈值，低于这个阈值就告警或阻止合并。

```bash
# 假设你使用 pytest-cov 来检查覆盖率
pytest --cov=src/ --cov-fail-under=80 tests/
# 如果覆盖率低于 80%，pytest 会返回非零退出码 → 流水线失败
```

> **为什么需要代码覆盖率门？** 代码覆盖率告诉你"有多少代码被测试覆盖到了"。80% 的覆盖率并不代表代码质量有多高——它只能告诉你"那 20% 没覆盖的代码是完全没有测试的"。覆盖率门的真正作用是：防止开发者写出新代码却不写对应的测试。

**代码风格门**
使用工具（如 ESLint、flake8、Pylint）检查代码是否符合团队约定的风格。风格门的作用不是"完美换行"而是维持代码库的可读性和一致性。

**安全扫描门**
- **静态应用安全测试（SAST）**：扫描源码中的安全漏洞模式（如 SQL 注入、XSS）
- **依赖漏洞扫描**：检查项目使用的第三方依赖中是否有已知的 CVE 漏洞
- **密钥泄露检测**：防止敏感信息（API Key、密码）被误提交到版本控制

**性能测试门**
如果新版本引入的性能退化超过一定阈值（如响应时间增加超过 5%），质量门阻止发布。这个门通常只在预发布环境或生产环境上运行。

### 9.3 质量门的设置原则

设置质量门时，有两条重要的原则：

**原则一：质量门需要通过"快速失败"来保护整个流水线**

把耗费时间最短的检查放在最前面。代码风格检查（几秒钟）应该放在编译（几分钟）之前，编译应该放在集成测试（几十分钟）之前。这样，如果某个检查失败，开发者在最短时间内收到反馈。

```
❌ 低效的质量门顺序：
端到端测试（1小时）→ 单元测试（5分钟）→ 代码风格检查（10秒）
—— 等了 1 小时才发现代码格式不对！

✅ 高效的质量门顺序：
代码风格检查（10秒）→ 单元测试（5分钟）→ 集成测试（20分钟）→ 端到端测试（1小时）
—— 10 秒内就能发现格式问题
```

**原则二：质量门应该是"硬性的"但不是"铁板一块的"**

硬性的意思是：自动化执行、不可绕过。每次 PR 合并都要经过质量门，没有例外。

但不是铁板一块的意思是：质量门的阈值可以根据项目阶段和实际情况动态调整。

- 在项目的早期（POC 阶段），代码覆盖率门可以设得低一些（30%）
- 在项目成熟期，代码覆盖率门可以设得高一些（80%）
- 如果一个门始终被打破且大家都知道它"不重要"（比如某个测试总是 flaky），那就应该修复或移除它，而不是保留一个"大家都知道可以忽略"的门——因为这会腐蚀整个质量体系的公信力

## 十、CI/CD 的常见坑和反模式

### 10.1 "CI 跑了好几个小时"

**问题**：流水线需要几个小时才能跑完。开发者提交代码后，等半天才能得到反馈。

**原因**：测试用例太多且没有得到合理分层；E2E 测试被当成了"唯一有效"的测试方式；硬件资源不足。

**解决**：
- 优化测试金字塔——把大量慢的 E2E 测试替换为快的集成测试和单元测试
- 并行执行测试（`pytest -x --parallel=4`）
- 只对受影响的模块运行相关测试（增量测试）
- 尝试"测试分级"——CI 阶段只跑关键的"烟雾测试"，完整的测试套件在后台异步执行

### 10.2 "总是在服务器上直接修复"

**问题**：线上出问题了，开发者在生产服务器上直接修改代码并重启服务测试，修好了后说"我回头补到代码仓库里"——然后从来不补。

**原因**：发布流程太慢、太复杂，走完整流程要花的时间远多于"直接上服务器改"。

**解决**：这是 CI/CD 流程需要改进的信号——如果"直接修"比"走流程修"快得多，说明流程太冗余了。优化方向包括减少不必要的审批环节、加快流水线速度、使部署足够简单到不需要"手动操作"。

### 10.3 "质量门变成了自动驾驶"

**问题**：所有人都不再看质量门的实际输出，只是机械地确认"门通过了"。测试覆盖率 80% 但覆盖的都是没有意义的测试（比如测试 getter/setter）。

**原因**：团队把"通过质量门"当作目标，而不是把"代码质量"当作目标。

**解决**：定期审计测试质量——"如果这些测试都删掉，我们会担心什么？"如果答案是什么也不担心，说明测试没有价值。鼓励开发者编写有意义的测试，而不是"为了凑覆盖率而写测试"。

### 10.4 "流水线只在 main 分支上跑"

**问题**：CI/CD 只对 main 分支生效。开发者在功能分支上从来不跑 CI，等到合并 PR 时才发现 CI 失败。

**原因**：流水线配置中没有对不同分支进行区分；或者因为 CI 资源有限，只能保障 main 分支。

**解决**：为所有分支都运行 CI（至少运行编译和单元测试）。这可以在配置中简单地实现——`on: [push]` 而不是 `on: push: branches: [main]`。对于资源受限的情况，可以为非 main 分支运行更简化的流水线。

### 10.5 "部署和配置修改混在一起"

**问题**：部署某个版本时，还需要手动去修改服务器上的配置文件（比如数据库连接串、API Key）。这次部署和上次部署之间的"差异"不仅仅是代码版本的变化，还包括配置的变化——导致定位问题时无法确定是代码问题还是配置问题。

**解决**：将配置与代码分离，通过环境变量、配置中心（如 Consul、Apollo）或 Kubernetes ConfigMap 来管理配置。部署时，代码和配置分开管理，各自有独立的版本管理。

## 十一、CI/CD 和 DevOps 的关系

### 11.1 DevOps 的起源

DevOps 不是一个工具，也不是一个职位（"DevOps 工程师"这个头衔实际上有些误导）。DevOps 是一种**文化和实践运动**，它的核心目标是通过打破"开发（Dev）"和"运维（Ops）"之间的壁垒来提高软件交付的效率。

传统上，开发团队负责写代码，运维团队负责部署和运行代码。这两个团队的目标天然存在冲突：
- 开发想要"越快发布越好"——希望尽可能多地交付功能
- 运维想要"越稳定越好"——希望生产环境尽可能不发生变化

CI/CD 是 DevOps 的核心技术实践之一。没有 CI/CD，DevOps 文化中对"快速交付"的追求就缺少了技术支撑。

### 11.2 CI/CD 在 DevOps 中的位置

DevOps 的实践体系常用的有 CALMS 框架（五个维度）：

| 维度 | 含义 | CI/CD 的贡献 |
|------|------|-------------|
| **文化（Culture）** | 协作、共享责任、信任 | CI/CD 自动化减少了手动交接的摩擦 |
| **自动化（Automation）** | 自动化重复性工作 | CI/CD 就是自动化的核心体现 |
| **精益（Lean）** | 减少浪费、持续改进 | CI/CD 缩短反馈循环，减少等待浪费 |
| **度量（Measurement）** | 用数据驱动决策 | CI/CD 提供了构建时间、部署频率、失败率等关键指标 |
| **共享（Sharing）** | 知识和责任共享 | 流水线即代码让团队共享部署知识 |

从技术角度看，CI/CD 是 DevOps 的"马达"——没有自动化的流水线，DevOps 文化中关于"频繁协作、快速反馈"的理念就难以落地。

## 十二、总结

### 核心要点回顾

1. **CI/CD 是方法论，不是工具**。它解决的核心问题是：让软件交付过程变得可预测、自动化、可重复。

2. **持续集成（CI）** 保证每次代码提交都被自动验证。它的关键实践是频繁合并 + 自动验证。

3. **持续交付（CD）** 保证每个通过 CI 的版本都处于"可部署"的状态。它通过发布流水线来分阶段验证构建产物。

4. **持续部署** 在持续交付的基础上，将生产部署这一步也自动化了。它需要更高水平的测试覆盖和监控能力。

5. **发布流水线** 将"代码到生产"的整个过程分解为多个阶段，每个阶段有独立的验证标准。流水线的设计原则是"尽早反馈"。

6. **部署策略** 有很多种（蓝绿、滚动、金丝雀），它们是在安全性、速度、成本之间寻找平衡。没有绝对"最好"的策略，只有最适合当前场景的策略。

7. **制品不可变性** 是一个核心原则：一旦发布，不可修改。它保证了可追溯性、可重现性和安全性。

8. **流水线即代码** 将流水线配置和代码放在一起管理，使得流水线本身也可以被审查、版本管理、代码审查。

9. **质量门** 是流水线中的检查点，确保只有达到质量标准的变化才能进入下一个阶段。质量门的设置需要在严格和务实之间找到平衡。

### 学习路径建议

如果这是你第一次系统学习 CI/CD，以下的学习路径可能对你有帮助：

1. **先理解为什么**：阅读第一节（CI/CD 解决的问题）——理解"为什么需要 CI/CD"比"怎么配置 CI/CD"更重要
2. **区分三个概念**：重点理解 CI / 持续交付 / 持续部署 的区别和联系（第二节到第四节）
3. **动手实践**：在 GitHub 上找一个简单项目，配置一个 GitHub Actions 流水线（第 5 节的内容亲自做一遍）
4. **理解部署策略**：当你需要把应用部署到生产环境时，再回来细读第六节
5. **建立质量意识**：理解质量门不是"为了卡人"，而是"为了保护系统稳定性"
