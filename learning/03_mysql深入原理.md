## 索引：从全表扫描到闪电定位

### 为什么需要索引？

假设你有一张 `users` 表，里面存了 1000 万条用户记录。现在要执行：

```sql
SELECT * FROM users WHERE id = 5000000;
```

如果没有索引，MySQL 只能从第一条记录开始，逐行对比 `id` 是否等于 `5000000`。这叫 **全表扫描（full table scan）** 。最坏情况下要对比 1000 万次才能找到目标行。

如果有了索引，情况完全不同——就像一本 1000 页的书有了目录，你不需要一页一页翻到第 500 页，而是通过目录直接定位到目标页码。

**索引的本质**：一种**额外的数据结构**，它将列值映射到物理存储位置。索引占额外的磁盘空间，但换来的是查询速度从 O(n) 降到 O(log n) 甚至 O(1)。

### B+Tree 数据结构详解

#### 什么是 B+Tree？

B+Tree 是一种**多路平衡查找树**，是 MySQL InnoDB 存储引擎默认的索引结构。

理解 B+Tree 的关键在于理解它的三层结构设计：

```
                     ┌──────────────────┐
                     │     [ 50  100 ]  │      ← 内部节点（只存键，不存数据）
                     └──────┬─────┬────┘
                  ┌─────────┘     └─────────┐
             ┌────┴─────┐              ┌───┴─────┐
             │ [20, 35]  │              │ [70, 85] │   ← 内部节点
             └─┬──┬──┬──┘              └─┬──┬──┬──┘
      ┌────────┘  │  └──────────┐  ┌─────┘  │  └──────┐
   ┌──┴──┐   ┌────┴───┐   ┌───┴──┐  ┌─┴──┐  ┌─┴───┐  ┌──┴───┐
   │1,5,8│   │21,25,30│   │36,40 │  │52,56│  │72,78│  │88,92│    ← 叶子节点（存完整数据行或行指针）
   └─────┘   └────────┘   └─────┘  └─────┘  └─────┘  └──────┘
      ↑                                           ↑
      └─────────────── 链表连接（范围查询的关键） ────┘
```

**三个关键特征**：

1. **多路（Multi-way）**：每个节点可以存储多个键值，而不是像二叉树那样只能存一个。MySQL 中每个节点通常对应一个**磁盘页（page，默认 16KB）**，一个节点可以容纳几百个键值。这意味着**树的高度非常低**——千万级数据也只要 3-4 层。

2. **平衡（Balanced）**：所有叶子节点在同一深度。这个特性保证了最差情况下的查询效率也是稳定的 O(log n)。任何插入、删除操作后，B+Tree 会自动通过**分裂（split）**和**合并（merge）**来保持平衡。

3. **叶子节点存真正数据，非叶子节点只存键值**：这是 B+Tree 和 B-Tree 最核心的区别。非叶子节点只做"指路牌"，不存数据，所以一个页面可以容纳更多键值，树更矮。

#### 为什么 MySQL 选择 B+Tree 而不是其他数据结构？

这是面试中极高频的问题，但更重要的是理解其中的**权衡**。

| 数据结构 | 查询速度 | 范围查询 | 写入速度 | 磁盘友好 |
|---------|---------|---------|---------|---------|
| **哈希表** | O(1) | ❌ 不支持 | O(1) | 一般 |
| **二叉搜索树** | O(log n) | O(log n + k) | O(log n) | ❌ 不友好 |
| **B-Tree** | O(log n) | O(log n + k) | O(log n) | 友好 |
| **B+Tree** | O(log n) | **O(log n + k) 最快** | O(log n) | **最友好** |
| **跳表** | O(log n) | O(log n + k) | O(log n) | 一般 |

**逐个分析为什么不选其他结构**：

**为什么不选哈希表？**
哈希表用 `hash(key) → bucket` 的方式定位，单条等值查询速度极快（O(1)）。但 MySQL 的查询远不止等值查询：
- 范围查询：`WHERE age > 20 AND age < 50`——哈希表无法做范围查询，因为哈希后的值失去了原始大小关系
- 排序：`ORDER BY age`——同样无法利用哈希索引
- 模糊匹配：`WHERE name LIKE '张%'`——前缀匹配也需要顺序特性
- 联合索引：哈希索引无法支持多列的部分匹配

**为什么不选二叉搜索树（BST / AVL / 红黑树）？**
二叉搜索树支持范围查询，理论上查询也是 O(log n)。但关键问题在于**磁盘访问**：
- 二叉树每个节点只存 1 个键，树的高度等于数据量取 log₂。1000 万条数据时，树高约 24 层。
- 每次访问一个节点，都需要一次磁盘 I/O（随机读取一个磁盘页）。
- 查找一条记录需要 24 次磁盘 I/O，每次约 10ms，总耗时约 240ms。这仅仅是查找一条记录的时间。
- 而 B+Tree 1000 万条数据只需要 3-4 层，3-4 次磁盘 I/O，耗时约 30-40ms。

**为什么不选 B-Tree？**
B-Tree 和 B+Tree 很像，唯一的区别是：B-Tree 的**每个节点都存数据**，而 B+Tree **只叶子节点存数据**。

- 这个区别导致 B-Tree 每个非叶子节点能容纳的键值更少（因为要留空间存数据指针），树更高，磁盘 I/O 更多。
- 更重要的是，B+Tree 的叶子节点通过**链表连接**，范围查询时找到起始位置后，沿着链表顺序读取即可，不需要反复回溯到父节点。这在范围查询上比 B-Tree 快得多。

**为什么选择 B+Tree 总结为一句话**：B+Tree 在**磁盘 I/O 次数**和**范围查询效率**两个维度上达到了最优平衡。

### 查询过程深度模拟

为了真正理解 B+Tree 的查询过程，我们模拟一个具体查询：

```sql
-- 假设有一张 users 表，id 是主键，有 2000 万条记录
-- InnoDB 默认以 id 建立聚簇索引（后面会讲）
-- 叶子节点每页 16KB，约存 200 条记录
-- 非叶子节点每页可以存约 1000 个键值对
SELECT * FROM users WHERE id = 999999;
```

**查询过程**：

```
第 1 步（根节点）：读根页 → 根页存了键值范围 [1, 20000000]
                  发现 999999 在某个范围内 → 根据指针进入下一层

第 2 步（中间节点）：读中间页 → 这个页存了 1000 个键值范围
                   999999 落在哪个范围 → 进入下一层

第 3 步（叶子节点）：读叶子页 → 这个页包含 200 条完整数据记录
                   在页内二分查找找到 id=999999 → 返回整行数据
```

整个过程只有 **3 次磁盘 I/O**。如果使用二叉树需要约 25 次 I/O，差了 8 倍以上。

## 聚簇索引与非聚簇索引

### 聚簇索引（Clustered Index）

**聚簇索引是什么意思？** 它的直接含义是：**索引的排列顺序和数据的物理存储顺序是相同的**。更具体地说，InnoDB 将整张表的数据行直接存储在索引的叶子节点中。

**InnoDB 聚簇索引的核心规则**：

1. 如果表定义了 PRIMARY KEY，InnoDB 使用主键列作为聚簇索引
2. 如果没有定义主键，InnoDB 选择第一个 NOT NULL UNIQUE 索引作为聚簇索引
3. 如果两者都没有，InnoDB 隐式创建一个 6 字节的 `ROW_ID` 列作为聚簇索引

```sql
CREATE TABLE users (
    id   BIGINT PRIMARY KEY,  -- 这列就是聚簇索引
    name VARCHAR(50),
    age  INT
);
```

**聚簇索引的存储示意图**：

```
聚簇索引（主键索引）的叶子节点：
┌─────────────────────────────────────────┐
│ 页 1 (叶子节点)                          │
│ ┌──────┬────────┬──────┬──────────────┐ │
│ │ id=1 │ name=张三│ age=25│ (其他行数据... )│ │
│ ├──────┼────────┼──────┼──────────────┤ │
│ │ id=2 │ name=李四│ age=30│ (其他行数据... )│ │
│ ├──────┼────────┼──────┼──────────────┤ │
│ │ ...                                  │ │
│ └──────┴────────┴──────┴──────────────┘ │
│  → 页内指针 → 下一叶节点页              │
└─────────────────────────────────────────┘
```

**注意**：聚簇索引的叶子节点存的是**完整的行数据**。这意味着如果通过主键查找，直接就能拿到所有列的值，不需要额外去别的地方找。

**聚簇索引的优缺点**：

| 优点 | 缺点 |
|------|------|
| 主键查询极快——直接从叶子节点拿到一整行 | 插入速度受主键顺序影响大 |
| 范围查询高效——叶子节点按主键顺序排列，且由链表连接 | 更新主键值代价极大（需要移动整行数据） |
| 覆盖索引能力：主键自身就是"覆盖"的 | 二级索引的叶子节点必须存主键值（占用空间） |

> **重要推论**：在 InnoDB 中，**主键最好是有序递增的整数**（如 AUTO_INCREMENT 的 BIGINT）。如果用 UUID 作为主键，每次插入要在 B+Tree 的随机位置插入新数据，会导致页分裂和碎片化，写入性能显著下降。这就是为什么生产环境几乎从来不用 UUID 做主键。

### 非聚簇索引（Secondary Index / 辅助索引）

非聚簇索引也叫**二级索引**。它的叶子节点**不存完整数据行**，而是存**主键值**。

```sql
-- 在 age 列上创建二级索引
CREATE INDEX idx_age ON users(age);
```

```
二级索引（idx_age）的存储：
非叶子节点：  [10, 20, 30, 40, ...]    ← 只存键值，用于导航
叶子节点：
┌───────────────────────────────────┐
│ (age=10, 主键id=5)               │
│ (age=10, 主键id=23)              │
│ (age=15, 主键id=8)               │
│ (age=20, 主键id=1)               │
│ (age=20, 主键id=12)              │
│ ...                              │
│ ↕ 链表连接，按 age 排序           │
└───────────────────────────────────┘
```

二级索引的叶子节点只包含两样东西：
1. **索引列的值**（这里是 age）
2. **对应行的主键值**（这里是 id）

它不存 name、不存 email、不存其他任何列。

### 回表查询（Back-Table Lookup / 回表）

**回表是什么意思？** 当你在二级索引上查找，但查询需要的列不全是索引列时，不得不拿着二级索引中找到的主键值，**回到聚簇索引中再查一次**来获取完整行数据。这个过程叫做回表。

**具体过程**：

```sql
-- 创建二级索引
CREATE INDEX idx_age ON users(age);

-- 执行查询
SELECT * FROM users WHERE age = 25;
```

查询过程：

```
第 1 步：在 idx_age 二级索引的 B+Tree 中查找 age=25
         → 找到叶子节点，得到 (age=25, 主键id=42)
         ↕ 一次索引查找

第 2 步：拿着 id=42 去聚簇索引的 B+Tree 中查找
         → 找到主键 id=42 对应的完整数据行
         ↕ 回表
         总耗时：2 次索引查找

如果是范围查询（如 WHERE age BETWEEN 20 AND 30），
可能找到 N 个主键值，然后回表 N 次。
```

**为什么回表慢？**
- 两次索引查找意味着两次 B+Tree 遍历（各 3-4 层，各 3-4 次 I/O）
- 如果二级索引命中了 1000 行，回表意味着 1000 次聚簇索引随机 I/O
- 这种"随机 I/O"非常慢，因为每次回表可能去不同的磁盘页面

### 覆盖索引（Covering Index）

**覆盖索引**是一种**优化手段**，而不是一种新的索引类型。当二级索引"覆盖"了查询所需的所有列时，就不需要回表了。

```sql
-- 有 idx_age(age) 索引时
SELECT name FROM users WHERE age = 25;
-- 需要回表：因为 name 不在 idx_age 中

-- 有 idx_age_name(age, name) 联合索引时
SELECT name FROM users WHERE age = 25;
-- 不需要回表：age 和 name 都在联合索引中
-- 直接从索引叶子节点拿到 name 值，结束
```

**覆盖索引的判断标准**：查询的 SELECT 列表和 WHERE 条件中使用的所有列，都在同一个索引中。

```sql
-- 索引：idx_a_b_c(a, b, c)

-- 被覆盖（不需要回表）：
SELECT a, b FROM t WHERE a = 1 AND b = 2;
SELECT a, b, c FROM t WHERE a = 1;

-- 不被覆盖（需要回表）：
SELECT d FROM t WHERE a = 1;     -- d 不在索引中
SELECT a, d FROM t WHERE a = 1;  -- d 不在索引中
```

**覆盖索引为什么快？**
- 索引文件通常比数据文件小得多（只存部分列）
- 不需要回表意味着减少了 B+Tree 的一次完整遍历
- 在极端情况下，覆盖索引可以减少 50% 以上的 I/O

> **实践建议**：对于频繁查询的热点查询场景，设计联合索引时可以将常用查询的 SELECT 列包含进来，避免回表。但不要为了覆盖索引而把太多列塞进联合索引——索引列越多，索引文件越大，写入越慢。这是典型的"空间换时间"权衡。

## 联合索引与最左前缀匹配

### 联合索引的存储结构

联合索引（也叫复合索引）是在**多个列**上建立的一个索引。

```sql
CREATE INDEX idx_age_city_name ON users(age, city, name);
```

联合索引在 B+Tree 中的排序规则是：**先按第一列排序，第一列相同再按第二列排序，以此类推**。

```
idx_age_city_name 的叶子节点存储：
(age=20, city='北京', name='张三') → 主键id=5
(age=20, city='北京', name='李四') → 主键id=8
(age=20, city='上海', name='王五') → 主键id=12
(age=25, city='北京', name='赵六') → 主键id=3
(age=25, city='广州', name='小明') → 主键id=7
(age=25, city='深圳', name='小红') → 主键id=9
(age=30, city='上海', name='小刚') → 主键id=1
```

可以看到：
- 数据先按 `age` 升序排列
- age 相同的，再按 `city` 排列
- age 和 city 都相同的，再按 `name` 排列

### 最左前缀匹配原则（Leftmost Prefix Matching）

**最左前缀匹配原则是什么意思？** MySQL 在联合索引中只能从最左边的列开始匹配，不能跳过中间的列，也不能单独使用最右边的列。

```sql
-- 索引：idx_age_city_name(age, city, name)

-- ✅ 能用上索引（匹配了第一列 age）
WHERE age = 25;

-- ✅ 能用上索引（匹配了 age 和 city，最左两列）
WHERE age = 25 AND city = '北京';

-- ✅ 能用上索引（匹配了所有三列）
WHERE age = 25 AND city = '北京' AND name = '张三';

-- ✅ 能用上索引（条件顺序不重要，优化器会重排）
WHERE city = '北京' AND age = 25 AND name = '张三';

-- ❌ 无法使用索引（没匹配第一列 age）
WHERE city = '北京';

-- ❌ 无法使用索引（跳过了 age）
WHERE city = '北京' AND name = '张三';

-- ⚠️ 部分使用索引（只用到了 age 列，city 的排序用不上）
WHERE age = 25 AND name = '张三';
```

**为什么会有最左前缀限制？**

想想 B+Tree 的排序方式——它是按 `(age, city, name)` 的顺序进行排序的。在 B+Tree 中，索引列的顺序就是**比较的优先级**。

- `WHERE age = 25`：MySQL 可以在 B+Tree 中定位到所有 age=25 的数据块，因为 B+Tree 顶层按 age 排序
- `WHERE city = '北京'`：B+Tree 顶层是按 age 排序的，age 列不同时，city 的顺序是"乱的"。无法在顶层直接定位到所有 city='北京' 的记录——因为不知道这些记录的 age 是什么
- `WHERE age = 25 AND name = '张三'`：MySQL 先利用 age=25 定位到数据范围，但这个范围内 city 是乱的，name 的排序依赖于 city。所以 name='张三' 的匹配需要在范围内逐行过滤，无法利用索引的排序特性

**最左前缀匹配的一个直观类比**：想象一本电话簿，按 `(姓氏, 名字)` 排序。你可以快速找到所有姓"张"的人，也可以快速找到"张"姓中叫"三"的人。但你无法直接找到所有名字叫"三"的人——因为电话簿没有按名字排序。

**实战中的核心推论**：

1. **联合索引的列顺序至关重要**。将区分度最高、查询最频繁的列放在最左边。
2. **不需要为最左列单独建索引**。`idx(a,b,c)` 已经覆盖了 `WHERE a=?` 的查询，额外建 `idx(a)` 是浪费。
3. **范围查询（>, <, BETWEEN, LIKE）会中断后续列的匹配**。

关于第三点：

```sql
-- 索引：idx_age_city(age, city)

-- ⚠️ age 的范围查询中断了 city 的索引匹配
WHERE age > 20 AND city = '北京';
-- MySQL 用 age>20 在索引中定位范围
-- 但在这个范围内，city 不是有序排列的
-- 所以 city='北京' 需要在每条记录上做过滤
-- 不是走索引快速定位

-- 相比之下，等值查询不中断：
WHERE age = 20 AND city = '北京';
-- 先按 age=20 定位到精确范围
-- 在这个范围内 city 是有序排列的
-- 可以用 city='北京' 快速定位
```

## 索引失效的 10 种场景

索引不是万能的——在某些情况下，即使 WHERE 条件列上有索引，MySQL 也不会使用它。理解这些场景，可以帮助你写 SQL 时避免"建了索引却没用"的尴尬。

### 场景 1：隐式类型转换

```sql
CREATE INDEX idx_age ON users(age);  -- age 是 INT 类型

-- ❌ 失效：把数字当字符串查
SELECT * FROM users WHERE age = '25';
```

**为什么失效？** MySQL 对 `age = '25'` 做了隐式类型转换，实际上变成了 `CAST(age AS CHAR) = '25'`。在索引列上应用了函数（CAST），索引自然失效了。

**同理**：如果 `phone` 是 VARCHAR 类型，`WHERE phone = 13800138000` 也会失效——MySQL 把字符串转数字比较，本质上也是函数操作。

### 场景 2：左模糊匹配（LIKE 以 % 开头）

```sql
CREATE INDEX idx_name ON users(name);

-- ✅ 能用：前缀匹配，可以走索引
SELECT * FROM users WHERE name LIKE '张%';

-- ❌ 失效：左模糊，无法利用 B+Tree 的有序性
SELECT * FROM users WHERE name LIKE '%三';

-- ❌ 失效：双侧模糊，同样无法走索引
SELECT * FROM users WHERE name LIKE '%小%';
```

**为什么失效？** B+Tree 按值排序存储，"张%"可以理解为"找所有以张开头的字符串"——在 B+Tree 中就是定位到"张"这个位置，然后往后扫描直到前缀不再是"张"。但"%三"无法在 B+Tree 中定位——你根本不知道该从哪个位置开始扫描。

### 场景 3：在索引列上使用函数或表达式运算

```sql
CREATE INDEX idx_created ON users(created_at);

-- ❌ 失效：在索引列上套了函数
SELECT * FROM users WHERE YEAR(created_at) = 2024;

-- ✅ 改为范围查询即可利用索引
SELECT * FROM users WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';

-- 其他常见错误：
WHERE id + 1 = 100;           -- ❌ 表达式在索引列上
WHERE salary * 1.1 > 50000;   -- ❌ 同上
```

**为什么失效？** MySQL 存储的是列本身的原始值，没有存储 `YEAR(created_at)` 或 `id+1` 的结果。所以无法在 B+Tree 中直接定位到 `YEAR(created_at)=2024` 的位置。

**核心原则**：把函数/运算放在等号的**右边**，让索引列保持"干净"。

### 场景 4：联合索引不满足最左前缀

已在最左前缀部分详细讲过，这里只重复要点：

```sql
-- 索引：idx_a_b(a, b)

-- ❌ 失效：没有用到第一列 a
WHERE b = 5;

-- ✅ 有效：用到第一列 a
WHERE a = 1 AND b = 5;
```

### 场景 5：OR 条件中有非索引列

```sql
CREATE INDEX idx_age ON users(age);

-- ❌ 失效：OR 的另一边（name）没有索引
SELECT * FROM users WHERE age = 25 OR name = '张三';

-- ✅ 改为 UNION 等效但能走索引
SELECT * FROM users WHERE age = 25
UNION
SELECT * FROM users WHERE name = '张三';
```

**为什么失效？** 如果 OR 有一边的列没有索引，MySQL 优化器的选择是：不如干脆全表扫算了——因为反正也要扫描全表去检查 name 的条件，不如一次性扫完，省得走索引再回表再和 name 条件合并，多了一步。

如果 OR 两边都有索引，MySQL 可能会走索引合并（index merge）。

### 场景 6：NOT IN / NOT EXISTS / != / <>

```sql
CREATE INDEX idx_age ON users(age);

-- ❌ 一般失效
SELECT * FROM users WHERE age != 25;
SELECT * FROM users WHERE age NOT IN (20, 25, 30);

-- ✅ 正查可以用
SELECT * FROM users WHERE age = 25;
SELECT * FROM users WHERE age IN (20, 25, 30);
```

**为什么失效？** B+Tree 擅长**定位到某个值**然后扫描相邻范围。但"不等于"意味着要扫**几乎全部数据**。这时候 MySQL 优化器判断：全表扫描比走索引再回表更快——因为走索引回表涉及大量随机 I/O，而全表扫描是顺序 I/O。

**例外**：如果 `!=` 的值只占极少量数据（比如只有 1% 的行满足），MySQL 可能还是会选择索引。

### 场景 7：IS NULL / IS NOT NULL（有争议，取决于数据分布）

```sql
CREATE INDEX idx_age ON users(age);

-- ⚠️ 可能走也可能不走，取决于 age IS NULL 的记录占比
SELECT * FROM users WHERE age IS NULL;
SELECT * FROM users WHERE age IS NOT NULL;
```

如果表中 99% 的记录的 age 都不为 NULL，`WHERE age IS NOT NULL` 几乎等于全表，优化器会直接全表扫。如果只有 1% 的记录的 age 为 NULL，`WHERE age IS NULL` 大概率走索引。

### 场景 8：数据量太小

当表的数据量非常少（比如只有几十行），MySQL 优化器认为**全表扫描比走索引更快**。因为 InnoDB 的数据和索引是分开存储的，走索引可能涉及"索引页"和"数据页"两次 I/O，而小表的全表扫一次 I/O 就搞定了。

这不是"失效"，而是优化器的合理选择。

### 场景 9：索引列的区分度太低

```sql
CREATE INDEX idx_gender ON users(gender);
-- gender 只有 '男' 和 '女' 两个值，区分度极低
```

如果索引列的取值很少（比如性别、状态码），通过索引找到的数据量占总表比例很高，MySQL 会认为回表代价太大，不如直接全表扫描顺序读取。

一般经验：**当通过索引筛选出的数据量超过全表的 20%-30% 时，MySQL 倾向于全表扫描**。

### 场景 10：查询优化器选择全表扫描（即使有可用的索引）

这实际上是前面多个场景的综合体现。MySQL 的查询优化器是一个**基于成本的优化器（CBO，Cost-Based Optimizer）**，它计算全表扫描的代价和走索引的代价，选择代价更小的执行计划。

```sql
-- 即使 name 有索引，优化器可能因为以下原因不走索引：
SELECT * FROM users WHERE name > 'a';  -- 几乎覆盖全部数据，不如全表扫
```

### 索引失效速查表

| 场景 | 原因 | 解决方案 |
|------|------|---------|
| 隐式类型转换 | 索引列被 CAST | 保证类型一致 |
| LIKE '%xxx' | 无法定位起始位置 | 改用前缀匹配或全文索引 |
| 索引列上的函数/运算 | 存储的是列值，不是计算结果 | 改写到等号右边 |
| 不满足最左前缀 | 联合索引排序优先级限制 | 调整查询或索引列顺序 |
| OR 含非索引列 | 优化器认为全表扫更优 | 用 UNION 替代 |
| NOT IN / != | 排除了大部分数据；B+Tree 不擅长"排除" | 改写为正查 |
| 数据量过小 | 索引额外 I/O 成本 > 全表扫 | 无需处理 |
| 区分度太低 | 回表成本太高 | 考虑前导索引、覆盖索引 |

## JOIN 查询详解：从集合论到实战

### SQL 中的 JOIN 到底是什么

JOIN（连接）是关系型数据库的核心操作。它做的事情是：**根据两个表之间的关联条件，将两个表的行组合成新的结果集**。

理解 JOIN 最直观的方式是**集合论（维恩图）**：

```
表 A（users）         表 B（orders）
┌────┬────────┐      ┌────┬──────────┐
│ id │ name   │      │ id │ user_id  │
├────┼────────┤      ├────┼──────────┤
│ 1  │ 张三   │      │ 1  │ 1        │
│ 2  │ 李四   │      │ 2  │ 1        │
│ 3  │ 王五   │      │ 3  │ 3        │
│ 4  │ 赵六   │      └────┴──────────┘
└────┴────────┘
```

### INNER JOIN（内连接）

**语义**：返回两个表中满足连接条件的**交集**。

```sql
SELECT u.name, o.id AS order_id
FROM users u
INNER JOIN orders o ON u.id = o.user_id;
```

```
结果：
┌──────┬──────────┐
│ name │ order_id │
├──────┼──────────┤
│ 张三 │ 1        │
│ 张三 │ 2        │
│ 王五 │ 3        │
└──────┴──────────┘
```

**执行过程**：对于 users 表中的每一行，去 orders 表中找 user_id 匹配的行。张三有 2 条订单 → 2 行结果；王五有 1 条 → 1 行；李四和赵六没有订单 → 不出现。

### LEFT JOIN（左外连接）

**语义**：保留左表的所有行，右表没有匹配就补 NULL。

```sql
SELECT u.name, o.id AS order_id
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;
```

```
结果：
┌──────┬──────────┐
│ name │ order_id │
├──────┼──────────┤
│ 张三 │ 1        │
│ 张三 │ 2        │
│ 李四 │ NULL     │
│ 王五 │ 3        │
│ 赵六 │ NULL     │
└──────┴──────────┘
```

### RIGHT JOIN（右外连接）

**语义**：保留右表的所有行，左表没有匹配就补 NULL。

实践中 RIGHT JOIN 用得很少，因为 LEFT JOIN 更容易理解和维护。通常用 LEFT JOIN 取代 RIGHT JOIN。

### CROSS JOIN（交叉连接 / 笛卡尔积）

**语义**：左表的每一行 × 右表的每一行，结果行数 = 左表行数 × 右表行数。

```sql
-- 两个表各有 100 行，CROSS JOIN 会生成 10000 行
SELECT * FROM users CROSS JOIN orders;  -- 没有 ON 条件
```

CROSS JOIN 在日常开发中很少直接使用，但理解它很重要——因为**任何 JOIN 如果不小心漏掉了 ON 条件（或 ON 条件写错了），就会变成 CROSS JOIN，产生巨大的中间结果**，这是 SQL 性能事故的常见原因。

### 不同类型 JOIN 的执行区别

| JOIN 类型 | 语义 | 左表无匹配 | 右表无匹配 | 典型场景 |
|-----------|------|-----------|-----------|---------|
| INNER JOIN | 只保留匹配的行 | 不出现 | 不出现 | 有订单的用户 |
| LEFT JOIN | 保留左表全部 | 出现（右表列=NULL） | 不出现 | 所有用户及其订单 |
| RIGHT JOIN | 保留右表全部 | 不出现 | 出现（左表列=NULL） | 所有订单及其用户 |
| FULL OUTER JOIN | 保留两表全部 | 出现 | 出现 | MySQL 不支持，用 UNION 模拟 |

### ON 和 WHERE 过滤条件的陷阱

这是 JOIN 查询中最容易被忽视的问题。先看一个例子：

```sql
-- 表结构和数据
-- users: 1(张三), 2(李四), 3(王五)
-- orders: 1(张三的), 2(李四的, 已取消), 3(王五的)

-- 需求：查所有用户，以及他们的未取消订单

-- 写法 A：过滤条件在 ON 中
SELECT u.name, o.id AS order_id
FROM users u
LEFT JOIN orders o ON u.id = o.user_id AND o.status = 'active';

-- 写法 B：过滤条件在 WHERE 中
SELECT u.name, o.id AS order_id
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.status = 'active';
```

**结果完全不同**：

```
写法 A 的结果（ON 中过滤）：
┌──────┬──────────┐
│ name │ order_id │
├──────┼──────────┤
│ 张三 │ 1        │
│ 李四 │ NULL     │    ← 李四还在！虽然订单被过滤了
│ 王五 │ 3        │
└──────┴──────────┘

写法 B 的结果（WHERE 中过滤）：
┌──────┬──────────┐
│ name │ order_id │
├──────┼──────────┤
│ 张三 │ 1        │
│ 王五 │ 3        │
└──────┴──────────┘
```

**为什么？**

核心区别在于 **JOIN 的执行顺序**：

ON 条件在 JOIN 操作**执行时**施加。`LEFT JOIN ... ON ...` 的语义是："先把两个表按 ON 条件连接，然后保留左表所有行"。即使李四的订单被 ON 条件滤掉了，LEFT JOIN 仍然保留李四，只是右表列补 NULL。

WHERE 条件在 JOIN 操作**完成后**施加。`WHERE o.status = 'active'` 在 JOIN 已经生成全部结果之后，再过滤掉不符合条件的行。李四没有 active 订单，JOIN 结果中 o.status 为 NULL，`NULL = 'active'` 不成立，整行被过滤掉。

**所以 LEFT JOIN 的 WHERE 过滤相当于把 LEFT JOIN 变成了 INNER JOIN**。

#### 记忆方法

| 过滤条件位置 | LEFT JOIN 的行为 |
|-------------|-----------------|
| ON 中 | 在 JOIN 时过滤右表，左表行始终保留 |
| WHERE 中 | JOIN 完成后过滤结果，左表可能被剔除 |

#### 实践原则

1. **LEFT JOIN 时，右表的过滤条件写在 ON 中**（在 JOIN 阶段过滤右表）
2. **只筛选左表的条件写在 WHERE 中**（JOIN 完成后只筛左表）
3. **INNER JOIN 时，ON 和 WHERE 等效**——因为 INNER JOIN 本身就不会保留无匹配的行

```sql
-- 正确做法：查所有用户及其未取消订单
SELECT u.name, o.id AS order_id
FROM users u
LEFT JOIN orders o ON u.id = o.user_id AND o.status = 'active';

-- 正确做法：查有未取消订单的用户（不需要保留无订单用户）
SELECT u.name, o.id AS order_id
FROM users u
INNER JOIN orders o ON u.id = o.user_id AND o.status = 'active';
-- 或者等价地：
SELECT u.name, o.id AS order_id
FROM users u
INNER JOIN orders o ON u.id = o.user_id
WHERE o.status = 'active';
```

## 事务：要么全做，要么全不做

### 为什么需要事务？

考虑一个经典的转账场景：

```sql
-- A 转账 100 元给 B
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- A 扣钱
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- B 加钱
```

如果第一条 UPDATE 成功执行后，服务器突然宕机、网络中断、或者程序异常退出，会发生什么？

A 账户扣了 100 元，但 B 账户没有收到。**100 元凭空消失了**。这是不可接受的。

事务（Transaction）解决这个问题：将这两条 UPDATE 操作**绑定为一个不可分割的单元**，要么全部成功，要么全部撤销。

### SQL 中的事务操作

```sql
START TRANSACTION;  -- 开启事务（或者用 BEGIN）

UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- A 扣钱
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- B 加钱

COMMIT;             -- 提交：两条修改一起生效
```

如果在两条 UPDATE 之间出现了任何错误：

```sql
START TRANSACTION;

UPDATE accounts SET balance = balance - 100 WHERE id = 1;

-- 服务器在这一刻宕机了 —— 重启后，刚才的 UPDATE 会自动回滚

-- 或者手动回滚：
ROLLBACK;           -- 回滚：两条修改一起撤销
```

**重要概念**：事务期间的操作只在**内存缓冲池**中修改，直到 `COMMIT` 才写入磁盘。如果在 COMMIT 之前崩溃，MySQL 在重启时通过 **undo log** 将事务的修改全部撤销，保证数据回到事务开始前的状态。

### ACID 四大特性

事务必须满足四个特性，统称 **ACID**。这四个特性是一个有机整体，不是孤立的四条规则。

#### A——原子性（Atomicity）

**定义**：事务是一个不可分割的最小工作单元。事务中的操作要么全部完成，要么全部不发生。

**实现机制**：**undo log**（回滚日志）。

当一条 UPDATE 或 DELETE 执行时，InnoDB 会在修改数据之前，先把修改前的数据写入 undo log。如果事务需要回滚，InnoDB 从 undo log 中读取旧值，恢复数据。

```
事务执行流程：
1. START TRANSACTION
2. 执行 UPDATE accounts SET balance = balance - 100 WHERE id = 1
   → 修改前：id=1 的 balance=500
   → InnoDB 将 "id=1, 旧值=500" 写入 undo log
   → 修改后：id=1 的 balance=400
3. 如果 ROLLBACK：
   → InnoDB 从 undo log 读出旧值 500，写回
   → id=1 的 balance 恢复到 500
```

#### C——一致性（Consistency）

**定义**：事务执行前后，数据都处于**合法状态**。也就是说，数据必须满足所有的约束规则（主键唯一、外键约束、CHECK 约束、业务逻辑等）。

**一致性是 ACID 的目标，其他三个特性是手段**：
- 原子性保证"全做或全不做"，防止了转账中 100 元消失
- 隔离性保证并发操作不互相干扰，防止看到中间状态
- 持久性保证提交后数据不丢失

它们共同工作，最终保证了**一致性**。

**一致性不是数据库单方面能保证的**——有些一致性需要应用程序层面来确保。例如，转账中 A 和 B 的总余额不变这个"业务一致性"，数据库并不知道这个规则，只能靠应用程序保证两条 UPDATE 同时成功或失败。

#### I——隔离性（Isolation）

**定义**：并发执行的多个事务之间互相隔离，一个事务的执行不能被其他事务干扰。

隔离性是一个"光谱"——完全隔离（串行执行）最安全但性能最差；不完全隔离性能更好但可能产生数据异常。MySQL 提供了 4 个隔离级别供开发者权衡选择。

**实现机制**：**锁** + **MVCC（多版本并发控制）**。下一节详细讲。

#### D——持久性（Durability）

**定义**：一旦事务提交（COMMIT），它对数据的修改就是永久性的，即使系统崩溃也不会丢失。

**实现机制**：**redo log**（重做日志）。

有些人认为"COMMIT 就是数据已经写入磁盘"，这个理解不完全准确。实际上 COMMIT 时，InnoDB 将事务的修改写入 redo log（而不是直接写数据文件），然后返回"提交成功"。如果这个时候宕机，重启后 InnoDB 通过 redo log 将修改**重放**到数据文件中。

```
COMMIT 的简化流程：
1. 事务执行过程中，修改集中在内存的 buffer pool 中
2. 每次修改先写 redo log（顺序写入磁盘，非常快）
3. COMMIT 时：将 redo log 从内存刷到磁盘
4. 返回 "OK" 给客户端
5. 后台线程在适当时候将 buffer pool 中的脏页写入数据文件

如果步骤 5 完成前宕机 → 重启时读取 redo log 重放修改
如果步骤 3 写入 redo log 前宕机 → 相当于没提交，数据不变
```

> **为什么 redo log 比直接写数据文件快？**
> 1. redo log 是**顺序写**（追加到文件末尾），数据文件是**随机写**（散落在磁盘各处）。顺序写比随机写快几个数量级（机械硬盘上约 100 倍）。
> 2. redo log 只记录"第 X 页第 Y 偏移量写入了什么值"，数据量远小于完整的数据页。

## 事务隔离级别

### 并发事务的三个数据问题

当多个事务并发执行时，如果没有适当的隔离，会产生以下三个问题。需要理解的是，**每个问题本质上都是"一个事务看到了不该看到的数据"**。

#### 问题 1：脏读（Dirty Read）

**定义**：事务 A 读到了事务 B **未提交**的数据。如果事务 B 之后回滚，事务 A 读到的是"不存在"的数据。

```
时间线：
事务 A                         事务 B
│                              │
├─ START TRANSACTION           │
│                              ├─ START TRANSACTION
│                              ├─ UPDATE accounts SET balance=0 WHERE id=1
│                              │  (未提交)
├─ SELECT balance FROM accounts ── 读到 balance=0 （脏读！）
│  WHERE id=1                  │
│                              ├─ ROLLBACK （B 回滚了）
│  (A 现在以为 balance=0，但其实正确的值是 100)
│                              │
```

**为什么这是问题？** 事务 A 读到了一条**从未真正存在过的数据**。如果 A 基于这个错误的值做了进一步操作（如扣款、写报表），数据就出错了。

#### 问题 2：不可重复读（Non-Repeatable Read）

**定义**：事务 A 内两次读取**同一行数据**，得到不同的结果。原因是事务 B 在这期间修改并提交了该行数据。

```
时间线：
事务 A                         事务 B
│                              │
├─ START TRANSACTION           │
├─ SELECT balance FROM accounts ── 读到 100
│  WHERE id=1                  │
│                              ├─ UPDATE accounts SET balance=200
│                              │  WHERE id=1
│                              ├─ COMMIT
├─ SELECT balance FROM accounts ── 读到 200 （不可重复读！）
│  WHERE id=1                  │
│  (明明在一个事务内，两次读同一行，结果不一样)
│                              │
```

**脏读 vs 不可重复读的区别**：
- 脏读：读到的是**未提交**的数据（可能回滚）
- 不可重复读：读到的是**已提交**的数据（但两次读不一致）

#### 问题 3：幻读（Phantom Read）

**定义**：事务 A 内两次执行**同一个范围查询**，第二次查询返回了**不同的行数**（多了或少了）。原因是事务 B 在这期间插入或删除了满足条件的行。

```
时间线：
事务 A                         事务 B
│                              │
├─ START TRANSACTION           │
├─ SELECT * FROM accounts      │
│  WHERE balance > 50          │
│  → 返回 2 行                 │
│                              ├─ INSERT INTO accounts (balance=300)
│                              ├─ COMMIT
├─ SELECT * FROM accounts      │
│  WHERE balance > 50          │
│  → 返回 3 行 （幻读！）      │
│  (多出了一行 "幻影" 数据)     │
│                              │
```

**不可重复读 vs 幻读的区别**：
- 不可重复读：同一行的值变了（**行内容**变化）
- 幻读：满足条件的**行数**变了（**行集合**变化）

### 四种隔离级别

MySQL 提供了 4 个隔离级别，从低到高依次是：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现方式 |
|---------|:---:|:--------:|:---:|---------|
| READ UNCOMMITTED | 可能 | 可能 | 可能 | 不加任何锁（有副作用） |
| READ COMMITTED | 避免 | 可能 | 可能 | MVCC（每次查询生成新的 Read View） |
| **REPEATABLE READ**（MySQL 默认） | 避免 | 避免 | 可能（InnoDB 通过间隙锁避免） | MVCC（事务开始时生成 Read View）+ 间隙锁 |
| SERIALIZABLE | 避免 | 避免 | 避免 | 所有查询都加锁（性能最差） |

#### READ UNCOMMITTED（读未提交）

**行为**：事务可以读到其他事务未提交的数据。

**问题**：脏读、不可重复读、幻读都可能发生。

**使用场景**：几乎不用。极少有场景需要读到"可能不存在"的数据。

```sql
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
```

#### READ COMMITTED（读已提交）

**行为**：事务只能读到其他事务**已提交**的数据。

**实现**：MVCC（Multi-Version Concurrency Control）。每次查询执行时，生成一个新的 **Read View**（读视图），只读取在 Read View 创建时已经提交的版本。

**问题**：不可重复读和幻读仍然可能发生（因为两次查询之间其他事务可能提交了新数据）。

**使用场景**：Oracle、PostgreSQL 的默认隔离级别。MySQL 的 READ COMMITTED 用于对一致性要求不那么严格的场景（如报表统计，要求看到最新已提交数据）。

#### REPEATABLE READ（可重复读）

**MySQL 的默认隔离级别**。

**行为**：在同一个事务内，多次读取同一行数据总是得到相同的值。即使其他事务修改并提交了该行，当前事务也看不到。

**实现**：MVCC 的 Read View 在**事务开始时**生成一次，整个事务期间重用这个 Read View。所以事务一开始看到的"数据快照"在整个事务生命周期内不变。

**InnoDB 如何解决幻读？**

上面说了 REPEATABLE READ 理论上存在幻读问题——因为 MVCC 的快照读（Snapshot Read）可以防止不可重复读，但无法阻止其他事务插入新行。然而 InnoDB 通过 **间隙锁（Gap Lock）**和 **Next-Key Lock** 解决了这个问题。

```sql
-- 在 REPEATABLE READ 下，以下查询不会出现幻读：
SELECT * FROM accounts WHERE balance > 50 FOR UPDATE;
-- FOR UPDATE 触发了当前读（而不是快照读）
-- InnoDB 给查到的行加上行锁，给范围加上间隙锁
-- 阻止了其他事务在这个范围内插入新行
```

**所以 InnoDB 的 REPEATABLE READ 实际上避免了幻读**（严格来说，是在大多数场景下避免了幻读）。

#### SERIALIZABLE（可串行化）

**行为**：最强的隔离级别，所有事务**串行执行**（实际是隐式为所有 SELECT 语句加上 `FOR SHARE` 锁）。

**问题**：性能最差，并发度急剧下降。

**使用场景**：对数据一致性要求极端严格的场景（如金融系统某些核心操作），但一般不会用这么高的隔离级别——通常通过其他手段保证一致性。

### 隔离级别的选择权衡

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;
-- MySQL 8.0: REPEATABLE-READ (默认)

-- 修改当前会话的隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 修改全局隔离级别（需要 SUPER 权限）
SET GLOBAL TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

**实践建议**：

| 场景 | 推荐级别 | 原因 |
|------|---------|------|
| 一般 Web 应用 | READ COMMITTED 或 REPEATABLE READ | 兼顾性能和数据一致性 |
| 报表/统计查询 | READ COMMITTED | 需要看到最新数据，不需要事务内重复读一致 |
| 转账、记账等财务操作 | REPEATABLE READ | 需要事务内多次读取结果一致 |
| 极端一致性要求 | SERIALIZABLE（很少用） | 性能代价大，通常用应用层方案替代 |

## 总结：索引 + 事务知识图谱

此时你已经建立了 MySQL 核心原理的知识体系。以下是这些概念之间的关联：

```
SQL 查询
  ├── 走索引 → 了解 B+Tree 结构
  │     ├── 聚簇索引（主键，存完整行）
  │     ├── 二级索引（普通索引，存主键值）
  │     │     ├── 回表（二级索引查完还要去聚簇索引拿数据）
  │     │     └── 覆盖索引（二级索引已包含所有需要列，不回表）
  │     ├── 联合索引（多列组合）
  │     │     └── 最左前缀匹配（决定联合索引哪些查询能走）
  │     └── 索引失效（10种场景）
  │
  ├── JOIN 多表
  │     ├── INNER / LEFT / RIGHT / CROSS
  │     └── ON vs WHERE（LEFT JOIN 时的陷阱）
  │
  └── 事务
        ├── ACID（原子性 → undo log，持久性 → redo log）
        ├── 隔离性 → 隔离级别
        │     ├── 脏读（RU 会发生）
        │     ├── 不可重复读（RC 会发生）
        │     └── 幻读（RR 通过间隙锁解决）
        └── 隔离级别选择 →
              RU < RC < RR < SERIALIZABLE
              (性能递减，一致性递增)
```
