## 1.1 HTTP 协议：互联网的通用语言

### 1.1.1 HTTP 是什么？它解决了什么问题？

**HTTP（HyperText Transfer Protocol，超文本传输协议）** 是 Web 的基石。当你在浏览器中输入一个网址时，浏览器就是用 HTTP 协议向服务器请求数据的。

简单来说，HTTP 定义了一套**客户端（Client）与服务器（Server）之间通信的规则**：客户端发送一个"请求（Request）"，服务器返回一个"响应（Response）"。

```
客户端（浏览器）                   服务器
    |                               |
    |---- HTTP 请求 (Request) ----->|
    |   "我要访问 /index.html"      |
    |                               |
    |<--- HTTP 响应 (Response) -----|
    |   "这是你需要的页面内容"      |
    |                               |
```

**核心特征**：
- **基于请求-响应模式**：客户端主动发起，服务器被动响应
- **无状态（Stateless）**：服务器默认不记忆之前的请求。每次请求都是独立的
- **应用层协议**：HTTP 工作在 OSI 第七层（应用层），底层依赖 TCP 传输

> **为什么 HTTP 被设计成无状态的？** 这是为了简化服务器设计。如果每个服务器都要记住所有用户的全部历史交互，服务器的内存和设计复杂度会急剧上升。但"无状态"也带来了问题——购物车、登录状态怎么维护？这正是后面要讲的 Cookie、Session、JWT 要解决的。

### 1.1.2 HTTP 请求的组成

一个 HTTP 请求由三部分组成：

```
POST /api/login HTTP/1.1          ← 请求行（Request Line）
Host: www.example.com             ← 请求头（Headers）
Content-Type: application/json
Authorization: Bearer xyz123

{"username": "alice", "password": "pass"}  ← 请求体（Body）
```

**请求行**包含三个要素：
1. **方法（Method）**：告诉服务器你想做什么。常见的有 GET（获取）、POST（提交）、PUT（更新）、DELETE（删除）
2. **路径（Path）**：你要访问的资源位置，如 `/api/login`
3. **协议版本（Version）**：如 `HTTP/1.1` 或 `HTTP/2`

**请求头**：键值对形式的元数据，告诉服务器额外的信息，如：
- `Host`：目标服务器的域名（HTTP/1.1 强制要求，因为一台服务器可以托管多个域名）
- `Content-Type`：请求体的格式（JSON、表单数据等）
- `Authorization`：身份认证信息
- `User-Agent`：客户端的类型和版本
- `Cookie`：之前服务器设置的身份标识

**请求体**（可选）：GET 请求没有请求体，POST/PUT 通常有请求体，包含要提交的数据。

### 1.1.3 HTTP 响应的组成

```
HTTP/1.1 200 OK                      ← 状态行（Status Line）
Date: Mon, 13 Jul 2026 08:00:00 GMT ← 响应头（Headers）
Content-Type: application/json
Content-Length: 45
Set-Cookie: sessionId=abc123

{"message": "登录成功", "token": "xyz"}  ← 响应体（Body）
```

**状态行**包含：
1. **协议版本**
2. **状态码（Status Code）**：三位数字，表示请求的结果
3. **状态描述（Reason Phrase）**：对人友好的简短说明

## 1.2 HTTP 状态码详解

状态码是服务器对请求结果的"标准化回答"。通过状态码，客户端可以立即知道请求是否成功，以及失败的原因。状态码是分**类别**的，由第一位数字决定：

| 类别 | 范围 | 含义 | 通俗理解 |
|------|------|------|----------|
| **1xx** | 100-199 | 信息性 | 服务器还在处理，请继续 |
| **2xx** | 200-299 | 成功 | 你的请求我处理好了 |
| **3xx** | 300-399 | 重定向 | 你要的东西不在这里，去别处找 |
| **4xx** | 400-499 | 客户端错误 | 你的请求有问题，我没法处理 |
| **5xx** | 500-599 | 服务器错误 | 你的请求没问题，但我自己出故障了 |

### 常见状态码逐一解释

**200 OK** — 一切正常，请求成功。最"愉快"的状态码。

**201 Created** — 资源创建成功。通常在 POST 请求后返回，表示新资源已创建。

**301 Moved Permanently** — 资源永久移动。搜索引擎会更新索引指向新地址。
> 场景：网站从 `http://old.com` 迁移到 `https://new.com`，返回 301 告诉客户端和搜索引擎"以后都去新地址"。

**302 Found** — 资源临时移动，但未来还可能回到原地址。搜索引擎会保留原索引。
> 场景：未登录用户访问需要登录的页面，临时跳转到登录页。

**304 Not Modified** — 资源未修改。用于缓存优化：客户端可以缓存资源，下次请求时带上条件，如果资源没变，服务器返回 304 告诉客户端"用你缓存的就行"，省流量省时间。

**400 Bad Request** — 请求格式错误。服务器看不懂你的请求。
> 常见原因：JSON 格式错误、缺少必填字段、参数类型不对。

**401 Unauthorized** — 未认证。客户端没有提供身份凭证，或者凭证无效。
> 场景：访问需要登录的 API 但没有带 token。

**403 Forbidden** — 禁止访问。客户端已经认证（知道你是谁），但你没有权限做这个操作。
> 场景：普通用户尝试删除管理员才能删除的资源。**401 和 403 的区别**：401 是"你是谁？"（没登录），403 是"你不行！"（没权限）。

**404 Not Found** — 资源不存在。服务器找不到你请求的资源。
> 场景：URL 拼写错误，或资源已被删除。

**405 Method Not Allowed** — 请求方法不被允许。
> 场景：API 只支持 POST，但你用了 GET。

**429 Too Many Requests** — 请求太频繁，被限流了。
> 场景：短时间内发了太多请求，触发了服务器的频率限制。

**500 Internal Server Error** — 服务器内部错误。服务器出问题了，但具体原因不明。
> 这是最令人头疼的状态码——你作为客户端只知道"服务器挂了"，但不知道为什么。需要查看服务器日志才能定位。

**502 Bad Gateway** — 网关错误。服务器作为网关或代理，从上游服务器收到了无效响应。
> 场景：Nginx 反向代理后面的应用服务器挂了，Nginx 向上游请求失败，返回 502。

**503 Service Unavailable** — 服务暂时不可用。服务器过载或正在维护。
> 场景：服务器 CPU 100% 了，无法处理新请求。通常还会返回 `Retry-After` 头告诉客户端多久后再试。

**504 Gateway Timeout** — 网关超时。服务器作为代理，在规定时间内没有收到上游服务器的响应。

> **为什么需要这么多状态码？** 状态码让**机器**能够自动判断如何处理响应。浏览器收到 301 会自动跳转，收到 401 会自动弹出登录框——这都是通过状态码自动触发的。状态码是人与机器之间的"协议"。

## 1.3 GET 与 POST：HTTP 中最常用的两个方法

### 1.3.1 GET：请求数据

GET 方法用于**获取资源**，就像你去图书馆借书——你告诉管理员你要什么书，管理员拿给你。

**GET 请求的特点**：
- 参数放在 URL 中：`https://api.example.com/users?id=123&page=2`
- 有长度限制（浏览器和服务器对 URL 长度有限制，通常 2048 字符左右）
- **幂等（Idempotent）**：同一个 GET 请求发 1 次和发 100 次，对服务器的效果是一样的
- **可缓存**：浏览器可以缓存 GET 请求的结果
- **不应该有请求体**（虽然 HTTP 规范没有严格禁止，但实践中大多数服务器会忽略）

### 1.3.2 POST：提交数据

POST 方法用于**提交数据并创建/修改资源**，就像你寄快递——你填好单子交给快递员。

**POST 请求的特点**：
- 参数放在请求体中：用户看不到（但仍然可以被抓包工具捕获）
- **没有长度限制**（理论上可以上传大文件）
- **非幂等**：同一个 POST 请求发 2 次，可能创建 2 条重复记录
- **不可缓存**（一般情况下浏览器不会缓存 POST 响应）
- 常用于：登录、注册、提交表单、上传文件

### 1.3.3 核心区别：语义上的不同

| 维度 | GET | POST |
|------|-----|------|
| **语义** | 查询/获取 | 创建/提交 |
| **幂等性** | 是 | 否 |
| **参数位置** | URL 查询字符串 | 请求体 |
| **缓存** | 可缓存 | 不可缓存（默认） |
| **安全性** | 参数在 URL 中可见 | 参数在请求体中略隐蔽 |
| **书签/历史** | 可收藏书签 | 无法收藏 |
| **数据长度** | 有限制 | 理论上无限制 |

> **一个常见的误解："POST 比 GET 更安全"** 这个说法不完全对。POST 的参数虽然在 URL 中不可见，但 HTTPS 加密后才真正安全。如果使用 HTTP，POST 的请求体同样是明文传输，Wireshark 抓包一样能看到。真正的"安全"指的是敏感数据必须走 HTTPS，而不是用 GET 还是 POST。

### 为什么设计两个方法？

HTTP 设计 GET 和 POST 是出于**语义化**考虑——就像自然语言中有"疑问句"和"祈使句"一样。有了明确的语义区分：

1. **搜索引擎**可以放心地爬取 GET 链接，而不会因为爬取而修改数据
2. **浏览器**知道 GET 可以安全地刷新和回退，而 POST 刷新时会弹出"是否重新提交表单"的提示
3. **缓存代理**可以缓存 GET 响应以提高性能

如果在实际应用中反过来用（比如用 GET 删除数据、用 POST 查询数据）——技术上可以运行，但会破坏这些约定，可能造成安全隐患（搜索引擎的爬虫不经意间删除了你的数据）。

## 1.4 HTTPS 与 TLS 握手：安全通信的建立过程

### 1.4.1 为什么需要 HTTPS？

HTTP 是**明文传输**的，这意味着：
- 用 Wireshark 抓包就能看到你发送的所有数据
- 中间人（比如公共 WiFi 的提供者）可以窃听、篡改数据

HTTPS = HTTP + **TLS（传输层安全协议）**，它在 HTTP 和 TCP 之间加了一层加密。

```
HTTP  （明文）          HTTPS （加密）
┌──────┐               ┌──────┐
│ HTTP │               │ HTTP │
├──────┤               ├──────┤
│ TCP  │               │ TLS  │  ← 新加的加密层
├──────┤               ├──────┤
│ IP   │               │ TCP  │
└──────┘               │ IP   │
                       └──────┘
```

HTTPS 解决了三大安全问题：
1. **机密性**：数据加密，窃听者看不懂
2. **完整性**：数据在传输中不被篡改
3. **身份认证**：确保你连接的是真正的服务器，而不是冒充的"假网站"

### 1.4.2 密码学基础：对称加密 vs 非对称加密

要理解 TLS 握手，先要理解两个核心概念：

**对称加密**：
- 加密和解密使用**同一个密钥**
- 就像你和朋友共享一把钥匙开门
- 优点：速度快、适合大数据量
- 问题：密钥怎么安全地传给对方？如果直接通过网络发送，密钥本身可能被截获

**非对称加密**：
- 使用一对密钥：**公钥（Public Key）** 和 **私钥（Private Key）**
- 公钥加密的数据只能用私钥解密，私钥加密的数据只能用公钥解密
- 就像一把上锁的邮箱——任何人都可以通过投信口往里投信（公钥加密），但只有你有钥匙可以打开（私钥解密）
- 优点：不需要事先交换密钥
- 问题：速度慢，不适合加密大量数据

TLS 手法的巧妙之处在于：**结合了两者的优点**。

### 1.4.3 TLS 1.2 握手过程详解

TLS 握手的核心目标：**让客户端和服务器安全地协商出一个"会话密钥"（Session Key），后续通信全部用这个密钥进行对称加密**。

整个流程需要用到**三个随机数**和一个"预主密钥"（Pre-Master Secret）：

```
客户端（浏览器）                        服务器
    |                                     |
    | 1. Client Hello                     |
    |   - TLS 版本（如 1.2）              |
    |   - 支持的加密套件列表               |
    |   - 随机数 random_C (client random) |
    |────────────────────────────────────>|
    |                                     |
    | 2. Server Hello                     |
    |   - 选定的 TLS 版本                 |
    |   - 选定的加密套件                   |
    |   - 随机数 random_S (server random) |
    |   - 服务器的数字证书                 |
    |<────────────────────────────────────|
    |                                     |
    | 3. 客户端验证证书                   |
    |   - 检查证书是否由可信 CA 签发      |
    |   - 检查域名是否匹配                |
    |   - 检查证书是否过期                |
    |   - 提取证书中的服务器公钥           |
    |                                     |
    | 4. Client Key Exchange              |
    |   - 生成随机数 pre-master secret    |
    |   - 用服务器公钥加密 pre-master     |
    |────────────────────────────────────>|
    |                                     |
    | 5. 双方各自计算会话密钥              |
    |   session_key = PRF(random_C +      |
    |                   random_S +        |
    |                   pre-master)        |
    |                                     |
    | 6. Change Cipher Spec               |
    |   "接下来我要用加密通信了"          |
    |<───────────────────────────────────>|
    |                                     |
    | 7. Finished (加密的消息)            |
    |   "握手完成，验证一下之前通信      |
    |    有没有被篡改"                    |
    |<───────────────────────────────────>|
```

> **为什么需要三个随机数？**
>
> 假设只有 random_C + random_S：这两个随机数是明文传输的，如果黑客记录了所有的握手数据，理论上有可能破解会话密钥。
>
> 引入 pre-master secret（第三个随机数）并用服务器的**公钥加密**传输后，即使黑客抓到了全部网络包，由于没有服务器的私钥，他也无法解密 pre-master secret。因此，**三个随机数共同保证了会话密钥的不可预测性和安全性**。
>
> 可以把这想象成：你和朋友各写一个数字在明信片上寄给对方（两个随机数）——所有人都能看到。然后你把一个数字放进只有对方才能打开的保险箱里寄出（pre-master secret）——只有收件人能打开。你们三人各自用这三个数字通过一个公式计算出最终的密码。

### 1.4.4 数字证书是怎么回事？

握手第 2 步中服务器发送的"数字证书"（Digital Certificate）起到了**身份认证**的作用。

**为什么需要证书？** 假设没有证书：你的浏览器第一次连接某个服务器时，服务器给你一个公钥。但你怎么知道这个公钥真的属于 `www.example.com`，而不是中间人伪造的？数字证书解决的就是这个"信任"问题。

**数字证书的组成**：
- 服务器的域名
- 服务器的公钥
- 证书颁发机构（CA，Certificate Authority）的数字签名
- 有效期

**信任链（Chain of Trust）**：
```
根证书（Root CA，预装在操作系统/浏览器中）
    └── 签发 → 中间证书（Intermediate CA）
            └── 签发 → 你的服务器证书
```

你的操作系统和浏览器预装了一批**根证书**（如 VeriSign、Let's Encrypt 等权威机构的根证书）。当服务器发来证书时，浏览器会沿着证书链向上追溯，直到找到一个受信任的根证书。如果追溯成功，说明这个证书是可信的。

> **可视化类比**：数字证书就像是"护照"。护照上有你的照片和信息（服务器信息和公钥），有签发机关的章（CA 签名）。你出国时，移民局检查护照——他们信任的是签发护照的机构（CA），而不是直接信任你这个人。

### 1.4.5 TLS 1.3 做了什么改进？

TLS 1.3（2018年发布）相比 1.2 有两大改进：

1. **更快的握手**：将握手从 2-RTT（来回两次）减少到 1-RTT（来回一次）
   ```
   TLS 1.2：客户端 → 服务器 → 客户端 → 服务器  （2 次往返）
   TLS 1.3：客户端 → 服务器                        （1 次往返，客户端直接发送密钥协商信息）
   ```
   
2. **更安全的加密套件**：移除了已经不安全的加密算法，只保留少数强力算法。这意味着不会因为配置了"过时但兼容"的算法而引入安全漏洞。

## 1.5 TCP 三次握手：建立可靠连接的基石

### 1.5.1 TCP 是什么？

TCP（Transmission Control Protocol，传输控制协议）是一种**面向连接、可靠的**传输层协议。

想象你要和朋友寄快递：
- **UDP（用户数据报协议）**：就像寄明信片——你把信扔进邮筒就不管了，不保证对方一定能收到，也不保证顺序。快，但不可靠。
- **TCP**：就像寄挂号信——需要双方确认收件地址、签收、有回执。慢，但可靠。

TCP 保证：
- 数据**不丢失**（丢失就重传）
- 数据**不乱序**（到达后按顺序排列）
- 数据**不重复**（重复的会被过滤）
- **流量控制**（接收方处理不过来时，告诉发送方慢点发）
- **拥塞控制**（网络拥堵时自动降低发送速度）

### 1.5.2 三次握手过程

在传输数据之前，TCP 需要先建立一个连接。这个建立过程就是**三次握手（Three-Way Handshake）**。

```
客户端                             服务器
  |                                  |
  | ─── SYN (seq=x) ──────────────> |  第1次：客户端说"我要连接你"
  |                                  |
  | <── SYN+ACK (seq=y, ack=x+1) ── |  第2次：服务器说"收到，我也准备好了"
  |                                  |
  | ─── ACK (seq=x+1, ack=y+1) ──> |  第3次：客户端说"好的，开始通信"
  |                                  |
  | <======== 开始数据传输 ========> |
```

**逐次解释**：

1. **第一次握手（SYN）**：客户端向服务器发送一个 SYN（Synchronize）报文，随机生成一个序列号 `seq=x`。此时客户端进入 `SYN_SENT` 状态。
   - 这意味着什么？客户端在说："你好，我想建立连接，这是我的初始序列号 x。"
   
2. **第二次握手（SYN+ACK）**：服务器收到 SYN 后，分配缓冲区，返回 SYN+ACK 报文。其中 `seq=y` 是服务器自己的初始序列号，`ack=x+1` 表示"我已经收到了你的报文，期待你的下一个序列号是 x+1"。服务器进入 `SYN_RCVD` 状态。
   - 服务器同时在说："我收到了你的请求（确认），这是我的初始序列号 y，我也想建立连接（SYN）。"
   
3. **第三次握手（ACK）**：客户端收到 SYN+ACK 后，分配缓冲区，发送 ACK 确认报文，`seq=x+1`，`ack=y+1`。客户端进入 `ESTABLISHED` 状态。服务器收到后也进入 `ESTABLISHED` 状态。
   - 客户端在说："好的，我收到了你的确认，我们开始通信吧。"

### 1.5.3 为什么是三次，而不是两次？

**这个问题考查的不是表面知识，而是对 TCP 设计原理的理解。**

如果只要两次握手：

```
客户端                             服务器
  |                                  |
  | ─── SYN (seq=x) ──────────────> |
  |                                  |
  | <── SYN+ACK (seq=y, ack=x+1) ── |
  |                                  |  ← 此时服务器认为连接建立了
  |                                  |     开始分配资源等待数据
```

问题是：如果第一次 SYN 报文**在网络中延迟了**，客户端超时后重新发送 SYN，新的连接建立并正常通信、关闭。然后**第一个延迟的 SYN 报文**才到达服务器——服务器以为是新的连接请求，于是发送 SYN+ACK，进入 `SYN_RCVD` 状态并分配资源。

但客户端根本没有发送这个请求，也不会回应 ACK。服务器白白分配了资源，浪费了！

**用状态机来理解**：

TCP 连接需要**双向确认**双方的收发能力都正常：

| 握手步骤 | 客户端知道什么 | 服务器知道什么 |
|---------|---------------|---------------|
| 第1次后 | — | 客户端的发送能力正常 |
| 第2次后 | 服务器的接收和发送能力正常 | 客户端的发送能力正常 |
| 第3次后 | — | 客户端的接收能力正常 |

只有三次握手才能让双方确认：**"你的消息我能收到，我的消息你也能收到"**。

> **一句话总结**：两次握手无法处理"已失效的连接请求"的干扰，也无法保证双方都确认了对方的接收能力。三次握手是最小可行的次数。

### 1.5.4 三次握手的实际代价

- 建立连接需要**1个RTT（往返时间）** 的延迟（实际是 1.5 RTT）
- 如果客户端在上海，服务器在纽约（RTT ≈ 200ms），建立 TCP 连接就要花 100ms
- 这就是为什么 HTTP 的 Keep-Alive（连接复用）很重要——避免频繁建立新连接

## 1.6 TCP 四次挥手：优雅地断开连接

### 1.6.1 为什么断开比建立复杂？

TCP 连接是**全双工（Full-Duplex）** 的，意味着数据可以双向同时传输。因此断开连接也需要双方各自关闭自己的方向——就像两部电话，需要双方都说"我挂了"。

```
主动关闭方                               被动关闭方
   |                                         |
   | ─── FIN (seq=u) ──────────────────> |  第1次：主动方说"我发完了"
   |                                         |
   | <── ACK (seq=v, ack=u+1) ────────── |  第2次：被动方说"知道了，但我还没说完"
   |                                         |
   |  （被动方继续发送剩余数据）              |
   |                                         |
   | <── FIN (seq=w, ack=u+1) ────────── |  第3次：被动方说"我也发完了"
   |                                         |
   | ─── ACK (seq=u+1, ack=w+1) ──────> |  第4次：主动方说"好的，彻底再见"
   |                                         |
```

**逐次解释**：

1. **第一次挥手（FIN）**：主动关闭方（比如客户端）发送 FIN 报文，表示"我的数据已经发完了，我要关闭连接了"。主动方进入 `FIN_WAIT_1` 状态。

2. **第二次挥手（ACK）**：被动关闭方（服务器）收到 FIN 后，发送 ACK 确认，"我收到你的关闭请求了，但我的数据可能还没发完"。被动方进入 `CLOSE_WAIT` 状态。主动方收到 ACK 后进入 `FIN_WAIT_2` 状态。

3. **第三次挥手（FIN）**：被动方把所有数据都发送完毕后，发送 FIN 报文，"现在我的数据也发完了，可以关闭了"。被动方进入 `LAST_ACK` 状态。

4. **第四次挥手（ACK）**：主动方发送 ACK 确认，"好的，我确认收到了"。主动方进入 `TIME_WAIT` 状态（等待 2MSL 后关闭）。被动方收到 ACK 后关闭连接。

### 1.6.2 为什么需要四次挥手，而不是三次？

**根本原因：因为 TCP 是全双工的，两个方向的关闭是独立的。**

当你给对方说"我的数据发完了"（FIN）时，对方可能还有数据要发送给你。所以对方不能立即关闭连接，需要先发 ACK 确认你的 FIN，等自己的数据也发完后，再发 FIN。

如果是一次挥手（合并 FIN+ACK+FIN），意味着：
1. 对方必须立即停止发送数据——但他可能还没发完
2. 无法实现"半关闭"状态——即 A 不再发送但还可以接收

**什么情况下可以变成三次？** 

如果被动关闭方也没有数据要发送了（即收到 FIN 后自己也要立即关闭），它可以把 ACK 和 FIN 合并发送，就变成了三次挥手。但在通用情况下，TCP 协议按四次挥手设计。

> **为什么是"挥手"而不是"握手"？** 建立连接时双方是平等协商的"握手"；断开连接时双方是互道告别的"挥手"。这个比喻很形象。

### 1.6.3 TIME_WAIT 状态和 2MSL

第四次挥手后，主动关闭方会进入 **TIME_WAIT** 状态，持续时间为 **2MSL**（Maximum Segment Lifetime，最大报文段生存时间）。MSL 在 RFC 793 中建议为 2 分钟，实际实现中通常为 30 秒到 2 分钟不等。

**TIME_WAIT 为什么需要 2MSL？**

两个原因：

**原因一：确保被动关闭方收到最后的 ACK**

如果第四次挥手的 ACK 丢失了，被动方会超时重发 FIN。如果主动方没有 TIME_WAIT 状态直接关闭，就无法响应这个重发的 FIN。

```
主动方（已关闭）                      被动方
   |                                         |
   | ─── ACK (丢失了！) ────────────────> |
   |                                         |
   | <── 超时重发 FIN ────────────────── |
   |                                         |
   此时主动方已关闭，无法响应
```

TIME_WAIT 就像给了"最后一次握手"一个保险期。如果 ACK 丢了，主动方在 TIME_WAIT 期间还能重新发送 ACK。

**原因二：防止旧连接的报文干扰新连接**

假设没有 TIME_WAIT，一个连接关闭后立即用相同的 IP 和端口建立新连接。如果旧的网络中还有一个延迟到达的数据包，新连接可能会错误地"接收"这个旧数据包，造成数据混乱。

TIME_WAIT 等待 2MSL 时间，足以确保旧连接的所有报文都在网络中消失。

> **2MSL 为什么是 2 倍？** 主动方的 ACK 需要 MSL 时间到达被动方，被动方重发的 FIN 又需要 MSL 时间回到主动方。所以主动方需要等待 2MSL 确保能收到可能重发的 FIN。

**TIME_WAIT 过多会有什么问题？**

在高并发场景下（比如服务器作为客户端频繁发起短连接），大量连接会积压在 TIME_WAIT 状态，导致端口被占用、系统资源耗尽。

优化方法：
1. 开启 `tcp_tw_reuse`（Linux 参数，允许重用 TIME_WAIT 连接的端口作为客户端）
2. 使用连接池复用连接，减少频繁建立/断开
3. 调整 `tcp_fin_timeout` 缩短 TIME_WAIT 时间（需权衡安全性）

## 1.7 从 Cookie 到 Session 再到 JWT：用户状态管理的演进

### 1.7.1 问题：HTTP 是无状态的

回忆一下 1.1 节——HTTP 是无状态的。这意味着：

1. 你登录了淘宝，服务器验证了你的密码
2. 服务器返回了"登录成功"
3. 你点击"我的购物车"——服务器又不认识你了，需要重新登录

这显然不现实。我们需要一种机制来**维持用户状态**。这就是 Cookie、Session 和 JWT 要解决的核心问题。

### 1.7.2 Cookie：客户端的"便签纸"

Cookie 是**存储在浏览器中的一小段数据**（通常不超过 4KB），由服务器通过 `Set-Cookie` 响应头设置。

**工作原理**：
```
服务器                          浏览器
  |                                |
  | Set-Cookie: sessionId=abc123   |
  | (登录成功后返回)               |
  |──────────────────────────────>|
  |                                | 浏览器存储 Cookie
  |  下一次请求:                   |
  |  Cookie: sessionId=abc123     |
  |<──────────────────────────────|
  |                                |
  | 根据 Cookie 识别用户          |
```

**Cookie 的属性**：
- `Domain`：哪些域名可以访问这个 Cookie
- `Path`：哪些路径下发送 Cookie
- `Expires / Max-Age`：过期时间
- `HttpOnly`：禁止 JavaScript 访问 Cookie（防止 XSS 攻击窃取 Cookie）
- `Secure`：只在 HTTPS 连接中发送
- `SameSite`：限制跨站请求发送 Cookie（防止 CSRF 攻击）

**Cookie 的局限**：
- 大小限制（4KB），无法存大量数据
- 每次请求都会自动带上，浪费带宽
- 数据存储在客户端，不安全（敏感数据不能放 Cookie）
- 浏览器可以禁用 Cookie

### 1.7.3 Session：服务器端的"档案柜"

Session 把数据存储在**服务器端**，客户端只保存一个 Session ID（通常是发到 Cookie 里）。

**工作原理**：
```
服务器                          浏览器
  |                                |
  | 1. 用户登录                    |
  |<──────────────────────────────|
  |                                |
  | 2. 在服务器创建 Session        |
  |   sessionId -> {用户信息}      |
  |                                |
  | 3. Set-Cookie: SESSION_ID=xxx  |
  |──────────────────────────────>|
  |                                |
  | 4. 再次请求，自动带 Cookie    |
  |   Cookie: SESSION_ID=xxx       |
  |<──────────────────────────────|
  |                                |
  | 5. 根据 xxx 找到 Session      |
  |    对象，恢复用户状态         |
```

**Session 的核心优势**：敏感数据存在服务器，客户端只存一个无意义的 ID，安全很多。

**Session 的问题**：

1. **内存占用**：每个用户占用服务器内存。100 万用户 = 大内存
2. **可扩展性（Scalability）**：如果用户第一次请求落到服务器 A，Session 存在 A 上。下一个请求落到服务器 B——B 找不到这个 Session！

```
   用户请求 ─→ 负载均衡 ─→ 服务器 A（有 Session）
              第二次请求 ─→ 服务器 B（没有 Session！）→ 需要重新登录
```

**Session 共享的解决方案**：
- **Session 黏滞（Sticky Session）**：把同一个用户的请求始终转发到同一台服务器——但该服务器挂了，Session 就丢了
- **Session 复制**：在多个服务器之间同步 Session——网络开销大
- **集中式 Session 存储**：把 Session 放到 Redis 等外部存储中——最常用的方案
  ```
  所有服务器从同一个 Redis 读取 Session 数据
  任何服务器都能服务任何用户
  ```

### 1.7.4 JWT：无状态的终极方案

JWT（JSON Web Token）是一种**无状态**的身份认证方案。它不把用户信息存在服务器，而是**加密后存在令牌中发给客户端**。服务器收到令牌后自己解码验证。

```
传统 Session：
  服务器保存：{ sessionId: {userId: 1, role: "admin"} }
  客户端保存：sessionId (在 Cookie 中)

JWT：
  服务器保存：什么都不存
  客户端保存：完整的用户信息（经过签名/加密的令牌）
```

**JWT 的结构**：`xxxxx.yyyyy.zzzzz`（三个部分用点号分隔）

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9        ← Header
.                                              
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkFsaWNlIn0  ← Payload
.
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c    ← Signature
```

#### Header（头部）

Header 通常包含两部分信息：

```json
{
  "alg": "HS256",     // 签名算法（HMAC SHA256）
  "typ": "JWT"        // 令牌类型
}
```

- `alg` 告诉验证方：这个 JWT 是用什么算法签名的
- `typ`：固定为 "JWT"

#### Payload（载荷）

Payload 是真正的数据部分，包含"声明（Claims）"。有三种声明：

**注册声明（Registered Claims）**：预定义的标准化字段

| 字段 | 全称 | 含义 | 示例 |
|------|------|------|------|
| `iss` | Issuer | 签发者 | `"iss": "auth.example.com"` |
| `sub` | Subject | 主题（通常是用户 ID） | `"sub": "123456"` |
| `aud` | Audience | 接收方 | `"aud": "api.example.com"` |
| `exp` | Expiration Time | 过期时间（Unix 时间戳） | `"exp": 1720800000` |
| `nbf` | Not Before | 在此时间之前不可用 | `"nbf": 1720796400` |
| `iat` | Issued At | 签发时间 | `"iat": 1720796400` |
| `jti` | JWT ID | 唯一标识 | `"jti": "unique-id-123"` |

**公共声明（Public Claims）**：自定义的字段，但需要避免冲突（建议在 IANA JSON Web Token Registry 中注册或使用命名空间）。

**私有声明（Private Claims）**：双方约定的自定义字段。

```json
{
  "sub": "123456",
  "name": "Alice",
  "role": "admin",
  "iat": 1720796400,
  "exp": 1720800000
}
```

#### Signature（签名）

签名是 JWT 安全性的核心。它的生成方式：

```
signature = HMACSHA256(
    base64UrlEncode(header) + "." + base64UrlEncode(payload),
    secret_key
)
```

简单来说：把 header 和 payload 用 Base64 编码后用"."连接，然后用一个**只有服务器知道的密钥（secret）** 进行哈希签名。

**验证过程**：
1. 收到 JWT 后，服务器用同样的密钥 + header + payload 重新计算签名
2. 如果计算出的签名和 JWT 中的签名一致 → 数据未被篡改
3. 检查 `exp` 是否过期 → 在有效期内
4. 提取 payload 中的用户信息

因为密钥只在服务器端，客户端无法伪造签名——客户端如果修改了 payload 中的 `"role": "admin"`，重新计算出的签名会和原签名不一致，服务器会拒绝。

> **JWT 的安全性保证**：JWT 用签名保证了**完整性**（Integrity）——数据没有被篡改。但默认的 JWT 并不加密 payload（payload 只是 Base64 编码，不是加密，任何人都可以解码看到内容）。如果需要加密，应使用 JWE（JSON Web Encryption）。

### 1.7.5 Cookie vs Session vs JWT：对比总结

| 维度 | Cookie | Session | JWT |
|------|--------|---------|-----|
| **数据存储位置** | 客户端（浏览器） | 服务器内存/Redis | 客户端（令牌本身） |
| **服务器状态** | 可以存状态（配合 Session） | 有状态 | **无状态** |
| **可扩展性** | 好（不需要服务端存储） | 差（需要 Session 共享方案） | **最好**（任何服务器都能验证） |
| **安全性** | 不安全（可篡改/窃取） | 安全（数据在服务端） | 较安全（签名防止篡改） |
| **跨域支持** | 差（同源策略限制） | 差（依赖 Cookie） | **好**（可以放在 Authorization 头） |
| **大小限制** | 4KB | 取决于服务器内存 | 较大（通常几 KB） |
| **适合场景** | 传统 Web 应用 | 有状态服务 | RESTful API、微服务、移动端 |
| **性能开销** | 小 | 查存储（Redis/DB） | 验证签名（计算开销） |

### 1.7.6 实际场景中的选择

**场景一：传统的服务端渲染 Web 应用**
- 使用 Cookie + Session 是最自然的选择
- 用户登录后，服务器创建 Session，Session ID 写入 Cookie
- 每次请求，服务器从 Session 获取用户状态

**场景二：RESTful API + 移动端/前端分离**
- JWT 更适合
- 移动端不支持 Cookie 怎么办？JWT 放在 Authorization 请求头中
- 微服务架构下，每个服务都能独立验证 JWT，不需要集中式的 Session 存储
- 前端（React/Vue）把 JWT 存在 localStorage，每次请求手动带上

**场景三：单点登录（SSO）**
- JWT 是理想的方案
- 用户在认证中心登录一次，获取 JWT
- 用同一个 JWT 访问所有子系统，每个子系统独立验证 JWT

> **JWT 的"无状态"是一把双刃剑**：优点是不需要服务器存储，缺点是无法主动注销。Session 过期了服务器可以删除 Session 对象，JWT 只要还在有效期内就有效。如果要主动让 JWT 失效，需要额外的黑名单机制（又回到了有状态）。

## 1.8 各协议之间的关系：一张全景图

```
┌─────────────────────────────────────────────────────────┐
│                   应用层（Application）                   │
│  ┌──────┐  ┌──────┐  ┌───────┐  ┌──────┐  ┌───────┐  │
│  │ HTTP │  │ HTTPS│  │ DNS  │  │ FTP  │  │ WebSvc│  │
│  └──┬───┘  └──┬───┘  └───┬───┘  └──┬───┘  └───┬───┘  │
│     │         │           │         │          │       │
├─────┼─────────┼───────────┼─────────┼──────────┼───────┤
│     │    ┌────┘           │         │          │       │
│     │    │   传输层（Transport）      │          │       │
│  ┌──┴────┴──┐                       ┌──┴───────┴──┐   │
│  │   TCP    │                       │    UDP      │   │
│  │ (可靠)   │                       │  (不可靠)   │   │
│  └────┬─────┘                       └──────┬──────┘   │
│       │                                    │           │
├───────┼────────────────────────────────────┼───────────┤
│       │              网络层（Internet）     │           │
│       └───────────────┬────────────────────┘           │
│                      ┌┴┐                              │
│                      │IP│                              │
│                      └─┘                              │
├────────────────────────────────────────────────────────┤
│                   网络接口层（Link）                     │
│            以太网、WiFi、PPP、...                       │
└────────────────────────────────────────────────────────┘
```

**关键关系**：
- HTTP 不关心数据怎么到达对方——它依赖 TCP 做可靠传输
- TCP 不关心数据怎么路由——它依赖 IP 做寻址和路由
- IP 不关心数据怎么在线路上传输——它依赖链路层（以太网等）做物理传输
- HTTPS 在 HTTP 和 TCP 之间加了一层 TLS 加密
- JWT 是应用层的一种数据格式，通过 HTTP 头传输
